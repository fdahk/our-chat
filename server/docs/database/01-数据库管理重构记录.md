# 01 · 数据库管理重构记录

> **时间**:2026-06-06
> **范围**:our-chat/server 全量数据库管理重构
> **作者**:架构演进会议结论 + 实施记录

## 1. 背景

OAuth IdP 模块落地后,出现了一连串数据库管理上的债务,集中暴露成两个问题:

### 1.1 没有迁移工具

历史上 our-chat 用 `mysql2` + `src/database/mySql.ts` 里的 `init()` 函数(裸 `CREATE TABLE IF NOT EXISTS` 字符串)管理 schema。问题:

- 无版本记录(不知道哪些 DDL 已应用)
- 无回滚机制
- 多副本启动可能并发抢 metadata 锁
- DDL 字符串散落,难审计

OAuth 模块加入 3 张新表时,我曾用"内联 DDL + 启动时跑 IF NOT EXISTS"延续这个模式,但维护两份 DDL(`init.ts` 和 `migrations/*.sql`)会漂移,且违反"按最佳实践"的目标。

### 1.2 类型不安全的 raw SQL

`mysql2.execute<RowDataPacket[]>` 把行类型靠开发者手工断言:

```ts
interface UserRow extends RowDataPacket {
  id: number;
  username: string;
  // ...写错字段名编译器不会告诉你
}
const [rows] = await mySql.execute<UserRow[]>('SELECT * FROM users WHERE id = ?', [id]);
```

字段拼写错误、表/列重命名、可空性约束全靠开发者自律。schema 跟代码两条线漂移是迟早的事。

## 2. 决策过程

### 2.1 候选方案对比

| 工具 | 周下载量 | 类型 | 跟现状的距离 |
|---|---|---|---|
| Knex(仅 migration) | ~2M | Query Builder + Migration | 小:只引入 migration runner,代码不动 |
| Sequelize | ~1.6M | 全 ORM | 中:全部 mysql2 调用要改 ORM API |
| TypeORM | ~2.1M | Decorator ORM | 中:同上,且维护活跃度争议 |
| Drizzle ORM | ~700K(增长最快) | TS-first ORM | 中:新工具,跟 agent-server 不统一 |
| **Prisma** | ~3.2M(已是第一) | Schema-first ORM + Migration | 中:同上,**但 agent-server 已用 Prisma** |

### 2.2 选 Prisma 的决定性理由

1. **跨项目统一**:agent-server `apps/node-server/prisma/schema.prisma` 已经在用 Prisma 6.x,团队学习成本为零
2. **schema-as-code**:一份 `schema.prisma` 表达全部 schema + 关系 + 索引,远胜散落的 `CREATE TABLE`
3. **类型安全**:Prisma Client 自动生成 TS 类型,字段拼写错误编译时即报
4. **业界主流**:2024+ npm 周下载量已超 Sequelize/TypeORM 之和
5. **raw SQL 不放弃**:`$queryRaw` + `$executeRaw` 完全支持,复杂查询保留 SQL 直觉

### 2.3 不选 Knex 的理由

最初一度倾向"只加 migration 不改查询",评估发现 raw SQL 类型不安全才是更深的痛——只解决 migration 是治标。Prisma 一次性解决两个问题。

## 3. 重构范围

| 维度 | 改动 |
|---|---|
| 依赖 | 移除 `mysql2`,新增 `@prisma/client` + `prisma` + `dotenv` |
| Schema | 新建 `prisma/schema.prisma` 描述全部 10 个 model(7 个 our-chat 表 + 3 个 OAuth 表) + 8 个 enum |
| Migration | `prisma/migrations/0_initial/migration.sql`(`prisma migrate diff` 生成,187 行) |
| DB 入口 | 新建 `src/database/prisma.ts` 单例,删 `src/database/mySql.ts` |
| BigInt 序列化 | 新建 `src/database/bigint-json.ts` polyfill,在 `app.ts` 顶部 import |
| 业务代码 | 13 个源文件全部从 `mysql2.execute` 改 Prisma Client API |
| 测试 | 8 个 spec 文件的 mock 从 `mySql.execute` 改 `prisma.<model>.<method>` |
| 启动流程 | `server.ts` 启动时 `applyPendingMigrations()`(包装 `prisma migrate deploy`)+ `seedDefaultClient()` |
| 工具脚本 | `package.json` 加 `db:migrate:dev/deploy/status/reset/baseline` + `db:studio` |

## 4. 不向 ORM API 完全迁移的取舍

Prisma 同时提供 model API(`prisma.user.findUnique`)和 raw SQL(`$queryRaw`)。本次重构的策略:

- **简单 CRUD** → model API,类型自动推导
- **复杂 JOIN / 聚合 / 动态字段** → 保留 raw SQL via `$queryRaw`(本次没遇到必需场景,全部用 model API)
- **mass assignment 类调用**(如原 `UPDATE users SET ? WHERE id = ?`)→ **借机收缩成显式白名单**(见 `routes/user.ts`),是个 security 改进

## 5. Schema 漂移修复

反向工程过程中发现两处 init() DDL 与实际代码用法不一致,**借此机会补全 schema**:

| 表.列 | 现象 | 修复 |
|---|---|---|
| `users.gender` | `friend.ts` SELECT 用,init() DDL 无 | schema 加 `gender String? @db.VarChar(16)` |
| `friendships.remark` | `friend.ts` SELECT 用,init() 无 | schema 加 `remark String? @db.VarChar(64)` |
| `friendships.status` ENUM | init() 只有 `pending/accepted/blocked`,代码用了 `sent` | enum 加 `sent` |

production DB 可能已经 ALTER 加过这些字段,新 migration 在已存在的库上需要手动 baseline(见 §7)。

## 6. BigInt 处理

Prisma 把 MySQL `BIGINT` 映射成 JS `BigInt` 类型,而 `JSON.stringify` 不支持 BigInt。

选择:**不改 DB 类型**(BIGINT 保留)+ 全局加 `BigInt.prototype.toJSON` polyfill 转 `Number`。

理由:
- 用户 ID 实际不会超 `Number.MAX_SAFE_INTEGER`(2^53)
- 边界 polyfill 比每处 `Number(user.id)` 维护成本低
- 改 BIGINT→INT 影响 production 数据,风险大

polyfill 位置:`src/database/bigint-json.ts`,在 `app.ts` / `prisma.ts` / `vitest.setup.ts` 三处都 `import` 一次确保任何入口先生效。

## 7. 已存在 DB 的 baseline 流程

dev 环境如果 MySQL 里**已有 our-chat 7 张表**(历史 init() 跑过),第一次启用 Prisma 需要 baseline:

```bash
# 1. 假装 0_initial migration 已应用,不真跑 SQL
pnpm db:baseline

# 2. 之后正常 deploy 后续 migration
pnpm db:migrate:deploy
```

全新空库:

```bash
pnpm db:migrate:dev    # 第一次:应用全部 migration + 跑 client generate
# 或
pnpm db:migrate:deploy # CI / prod:只 deploy,不 generate
```

详见 `docs/database/02-migration-SOP.md`。

## 8. 测试影响

- 8 个 spec 文件的 mock 从 `vi.mock('mySql')` 改 `vi.mock('prisma')`
- mock 接口从 `execute: vi.fn()` 改为 `<model>.<method>: vi.fn()`
- BigInt 字段在 mock 数据用 `1n` 字面量(JS BigInt 语法)
- vitest 加 `setupFiles: ['./vitest.setup.ts']` 让 BigInt polyfill 在测试也加载
- discovery/jwks 这种不查 DB 的端点测试,仍要 `vi.mock` 阻止 PrismaClient 实例化(避免缺 DATABASE_URL)

**结果:122/122 测试全过**(原 114 + 重写后保持等效),OAuth reuse 检测的 3 个核心攻击场景仍然覆盖。

## 11. 经验沉淀

- **Prisma migrate 不接管已有 DB**:必须先 baseline,否则会试图重建已有表
- **`provider = "prisma-client"`(Prisma 7 新 generator)默认要 accelerateUrl**;self-hosted MySQL 仍用 `prisma-client-js`
- **mock 接口从 `execute` 升级到 model 方法**实际让测试更可读——每个 mock 调用都对应业务语义
- **Prisma 的 `$transaction` 用回调形式**比 mysql2 的 `getConnection()` 干净,事务里的 model 调用自动用同一连接
- **BigInt 序列化坑早 polyfill**,等到运行时报 `Do not know how to serialize a BigInt` 再修代价大
