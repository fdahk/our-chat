# 02 · 数据模型

## 1. 表清单

| 表 | 用途 | 大致行数预估 | TTL |
|---|---|---|---|
| `oauth_clients` | 注册的 OAuth client(client_id / 允许的 redirect_uri / scope) | 个位数 | 永久 |
| `oauth_codes` | authorization code(短期一次性) | 高峰 ~ 千行 | 1 min(过期清理) |
| `oauth_refresh_tokens` | refresh token + family,支持 rotation 检测 | 累计百万级 | 30 day + 滚动清理 |

## 2. DDL(完整)

> 见 `src/database/migrations/001_oauth_tables.sql`。本节说明字段语义和索引取舍。

### 2.1 `oauth_clients`

```sql
CREATE TABLE oauth_clients (
  client_id           VARCHAR(64)  PRIMARY KEY,
  client_name         VARCHAR(128) NOT NULL,
  client_type         ENUM('public', 'confidential') NOT NULL,
  client_secret_hash  VARCHAR(255) NULL,                      -- confidential 必填,bcrypt
  redirect_uris       JSON         NOT NULL,                  -- 数组,exact match 校验
  allowed_scopes      JSON         NOT NULL,                  -- 数组
  allowed_grant_types JSON         NOT NULL DEFAULT (JSON_ARRAY('authorization_code', 'refresh_token')),
  token_lifetime_sec  INT          NOT NULL DEFAULT 900,      -- 默认 15 min
  refresh_lifetime_sec INT         NOT NULL DEFAULT 2592000,  -- 默认 30 day
  require_pkce        TINYINT      NOT NULL DEFAULT 1,        -- public 永远为 1
  created_at          DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at          DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  disabled            TINYINT      NOT NULL DEFAULT 0
);
```

**字段说明**:

- `client_id`:对外公开的标识。public client 用人类可读字符串(`our-chat-web`);confidential 可用 UUID
- `client_type`:`public`(SPA / Native,无 secret)/ `confidential`(服务端集成,有 secret)
- `client_secret_hash`:**只存 bcrypt 哈希**,跟密码一样的安全等级。public client 必为 NULL
- `redirect_uris`:**JSON 数组,严格 exact match**(协议 + 主机 + 端口 + 路径完全相同)。绝不允许通配符或前缀匹配——RFC 8252 强制
- `allowed_scopes`:允许的 scope 列表,如 `["openid", "profile", "agent-server"]`,请求超出范围 → `invalid_scope`
- `allowed_grant_types`:本 client 可用的 grant 类型,通常是 `["authorization_code", "refresh_token"]`
- `require_pkce`:public client 永远要 PKCE,confidential client 也强烈推荐要(OAuth 2.1 default)
- `disabled`:软删除 / 紧急停用

**索引**:`client_id` 即 PK,无需额外索引(查询永远经 client_id)

### 2.2 `oauth_codes`

```sql
CREATE TABLE oauth_codes (
  code                  VARCHAR(64)  PRIMARY KEY,         -- 高熵随机
  client_id             VARCHAR(64)  NOT NULL,
  user_id               INT          NOT NULL,
  redirect_uri          VARCHAR(512) NOT NULL,            -- 必须与 /token 请求一致
  code_challenge        VARCHAR(128) NOT NULL,            -- base64url(SHA256(verifier))
  code_challenge_method ENUM('S256') NOT NULL,            -- 不接受 plain
  scope                 VARCHAR(512) NOT NULL,
  nonce                 VARCHAR(255) NULL,                -- OIDC scope 时存
  expires_at            DATETIME     NOT NULL,            -- 创建时 +1 min
  used                  TINYINT      NOT NULL DEFAULT 0,  -- 一次性,使用后立即置 1
  created_at            DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,

  INDEX idx_expires (expires_at),                          -- 清理任务用
  INDEX idx_user (user_id)                                 -- 用户登出时 invalidate 所有 code
);
```

**关键约束**:

- `code` 是 PK,使用时 `UPDATE used = 1`(原子操作),并发请求第二次会拿到 used = 1 → 拒绝
- `redirect_uri` 必须**完整存储**而非仅校验,因为 `/token` 端点要做"redirect_uri must match"二次校验(RFC 6749 §4.1.3)
- `code_challenge_method` 只接 `S256`,DDL 上 ENUM 强制
- `expires_at` 1 min 后,但即使过期未清理,`used = 1` 仍然保留作为审计证据(后台清理任务清的是 used = 1 且 expires_at < NOW() - 1 day 的)

### 2.3 `oauth_refresh_tokens`

```sql
CREATE TABLE oauth_refresh_tokens (
  jti           VARCHAR(64) PRIMARY KEY,              -- RT 自身的唯一标识
  family_id     VARCHAR(64) NOT NULL,                 -- 同一登录会话的 RT 链
  client_id     VARCHAR(64) NOT NULL,
  user_id       INT         NOT NULL,
  scope         VARCHAR(512) NOT NULL,
  issued_at     DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
  expires_at    DATETIME    NOT NULL,
  revoked       TINYINT     NOT NULL DEFAULT 0,
  rotated_to    VARCHAR(64) NULL,                     -- 下一根 RT 的 jti
  rotated_at    DATETIME    NULL,
  revoke_reason VARCHAR(64) NULL,                     -- 'rotation' / 'reuse_detected' / 'logout' / 'admin'

  INDEX idx_family (family_id),                        -- reuse 检测时按 family 全量更新
  INDEX idx_user_active (user_id, revoked, expires_at), -- 用户登出 / 主动撤销
  INDEX idx_expires (expires_at)                        -- 清理过期
);
```

**字段语义**:

- `jti`:RT 本身就用 `jti` 作为标识符,RT 字符串里包含 `jti`(签发时生成 random,塞到 JWT payload)
- `family_id`:同一个用户 + 同一次登录会话开始的 RT 链共享一个 family。**rotation 链是单链表,family_id 是这条链的 ID**
- `rotated_to` + `rotated_at`:rotation 发生后填充,代表本根 RT 已被合法地"用过"
- `revoke_reason`:审计追溯用,值域有限,代表撤销原因

**核心查询模式**:

```sql
-- 1. RT 提交时校验
SELECT * FROM oauth_refresh_tokens
WHERE jti = ? AND client_id = ?
LIMIT 1;
-- 校验:revoked = 0, expires_at > NOW(), rotated_to IS NULL

-- 2. 检测到 reuse,立即 invalidate 整个 family(原子操作)
UPDATE oauth_refresh_tokens
SET revoked = 1, revoke_reason = 'reuse_detected'
WHERE family_id = ? AND revoked = 0;

-- 3. 正常 rotation
UPDATE oauth_refresh_tokens
SET rotated_to = ?, rotated_at = NOW(), revoke_reason = 'rotation'
WHERE jti = ?;
INSERT INTO oauth_refresh_tokens (jti, family_id, ...) VALUES (?, <same family>, ...);

-- 4. 用户登出
UPDATE oauth_refresh_tokens
SET revoked = 1, revoke_reason = 'logout'
WHERE user_id = ? AND revoked = 0;
```

## 3. 索引取舍

| 表 | 索引 | 取舍考虑 |
|---|---|---|
| `oauth_codes` | `idx_expires` | 清理任务用;不上 user_id 索引节省写开销(code 表写多读少) |
| `oauth_refresh_tokens` | `idx_family` | reuse 检测必备,O(family 链长度) ~= 几十级 |
| | `idx_user_active` | 用户主动登出 / 管理员撤销,覆盖 `(user_id, revoked, expires_at)` 三联,避免回表 |
| | `idx_expires` | 清理任务 |

**有意不上的索引**:

- `oauth_codes.client_id`:code 表本来体量小,且查询永远经 PK
- `oauth_refresh_tokens.client_id`:RT 校验时已经经 jti(PK)+ client_id(WHERE),用 PK 单点查询足够

## 4. TTL 清理任务

后台定时任务(每 5 分钟一次):

```sql
-- 清理过期 code(used = 1 留一天作审计)
DELETE FROM oauth_codes
WHERE (used = 0 AND expires_at < NOW())
   OR (used = 1 AND expires_at < DATE_SUB(NOW(), INTERVAL 1 DAY));

-- 清理过期 refresh(revoked 留 7 天作审计)
DELETE FROM oauth_refresh_tokens
WHERE (revoked = 0 AND expires_at < NOW())
   OR (revoked = 1 AND issued_at < DATE_SUB(NOW(), INTERVAL 7 DAY));
```

**为什么不立即清理 revoked 的**:防止重用检测的告警出现后,管理员能查到完整 family 历史(reuse_detected 的时间 / IP / UA 等),用于事后分析攻击模式。

## 5. 跟现有 users 表的关系

- `oauth_codes.user_id`、`oauth_refresh_tokens.user_id` 都指向 `users.id`,但**不上外键约束**(MySQL 外键开销 + 不便迁移)
- 用户删除时(`users.status = 'deleted'`),不联动删除 OAuth 表——sign 时如果用户已 deleted,直接拒签,旧 token 自然过期失效

## 6. 数据迁移策略

- 新建系统 → 直接跑 migration
- 已有 our-chat 系统接入 → 加 migration 不影响现有业务(只新增表,不动 users 表)
- 回滚:`DROP TABLE oauth_clients, oauth_codes, oauth_refresh_tokens;` 即可,无 schema dependency
