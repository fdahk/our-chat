# 01 · 架构设计

## 1. 角色 / 信任边界

```
┌────────────────────────────────────────────────────────────────────┐
│                        信任域 A:our-chat                          │
│                                                                    │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │            our-chat server(Express)                          │  │
│  │  ┌─────────────────────┐    ┌───────────────────────────┐    │  │
│  │  │  现有业务路由         │    │  新增 OAuth/OIDC IdP 模块  │    │  │
│  │  │  - /api/login         │    │  - /oauth/authorize       │    │  │
│  │  │  - /user/...          │    │  - /oauth/token           │    │  │
│  │  │  HS256 + cookie       │    │  - /oauth/revoke          │    │  │
│  │  │                       │    │  - /oauth/introspect      │    │  │
│  │  │  共享 user 表 ←───────┼────┤  - /oauth/userinfo        │    │  │
│  │  │  共享 session 概念     │    │  - /.well-known/jwks.json │    │  │
│  │  │                       │    │  - /.well-known/openid-…  │    │  │
│  │  └─────────────────────┘    │  RS256 + JWKS              │    │  │
│  │                              └───────────────────────────┘    │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                    │
│  RSA 私钥                                                          │
│  oauth_clients / oauth_codes / oauth_refresh_tokens(MySQL)        │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
                            │ JWKS 公钥(HTTP)
                            │ access_token(Bearer)
                            ▼
┌────────────────────────────────────────────────────────────────────┐
│                信任域 B:agent-server(Resource Server)             │
│                                                                    │
│  - 只持公钥(经 JWKS 异步拉取 + 缓存)                              │
│  - 验签 + 校验 iss / aud / exp                                      │
│  - 接受任何符合 audience='agent-server' 的 token                    │
└────────────────────────────────────────────────────────────────────┘
```

**关键设计**:`our-chat` 内部业务鉴权(HS256 cookie)和对外签发的 OAuth token(RS256 Bearer)**共用同一个用户表,但 token 协议完全独立**。这两套永远不互相替换——业务侧不接受 RS256 token,IdP 端点不读 HS256 cookie(除了 `/oauth/authorize` 检测"是否已登录")。

## 2. 模块拆分

```
src/oauth/
├── index.ts            装配 oauthRouter,挂到 /oauth
├── discovery.ts        GET /.well-known/openid-configuration
├── jwks.ts             GET /.well-known/jwks.json
├── authorize.ts        GET /oauth/authorize(PKCE 授权码端点)
├── token.ts            POST /oauth/token(authorization_code + refresh_token grant)
├── revoke.ts           POST /oauth/revoke(RFC 7009)
├── introspect.ts       POST /oauth/introspect(RFC 7662)
├── userinfo.ts         GET /oauth/userinfo(OIDC Core)
├── keys.ts             RSA 私钥/公钥加载,kid 管理
├── clients.ts          oauth_clients CRUD + 校验
├── storage.ts          oauth_codes / oauth_refresh_tokens CRUD
├── pkce.ts             code_verifier / code_challenge S256 校验
├── errors.ts           OAuth 标准错误响应(invalid_request 等)
├── types.ts            共享类型
└── audit.ts            结构化审计日志(签发 / 撤销 / 重用检测)
```

## 3. 数据流(端到端时序)

### 3.1 PKCE Authorization Code Flow

```
Web SPA              our-chat server              MySQL
  │                       │                          │
  │ 1. 生成 verifier/challenge                       │
  │                       │                          │
  │ 2. GET /oauth/authorize?                         │
  │    client_id=our-chat-web                        │
  │    &response_type=code                           │
  │    &redirect_uri=...                             │
  │    &code_challenge=<S256>                        │
  │    &code_challenge_method=S256                   │
  │    &state=<random>                               │
  │    &scope=openid profile agent-server            │
  │    &nonce=<random>                               │
  │ ────────────────────►│                          │
  │                       │ 3. 校验 client / redirect│
  │                       │ ───────────────────────►│ SELECT oauth_clients
  │                       │ ◄───────────────────────│
  │                       │                          │
  │                       │ 4. 校验登录(读 HttpOnly cookie)│
  │                       │    未登录 → 302 /login?next=<原 URL>│
  │                       │    已登录 → 继续          │
  │                       │                          │
  │                       │ 5. 生成 code,存 challenge│
  │                       │ ───────────────────────►│ INSERT oauth_codes
  │                       │                          │
  │ ◄───302 redirect_uri?code=<C>&state=<echo>──────│
  │                       │                          │
  │ 6. 校验 state 一致     │                          │
  │                       │                          │
  │ 7. POST /oauth/token                             │
  │    grant_type=authorization_code                 │
  │    &code=<C>                                     │
  │    &code_verifier=<原 verifier>                  │
  │    &client_id=our-chat-web                       │
  │    &redirect_uri=<同 step 2>                     │
  │ ────────────────────►│                          │
  │                       │ 8. SELECT oauth_codes,   │
  │                       │    校验 used/expires/    │
  │                       │    redirect_uri 一致     │
  │                       │ ◄───────────────────────│
  │                       │ 9. SHA256(verifier) ==  │
  │                       │    stored challenge?     │
  │                       │ 10. UPDATE used=1       │
  │                       │     INSERT oauth_refresh_tokens│
  │                       │ 11. RS256 签 access_token│
  │                       │     + id_token + refresh │
  │ ◄────────────────────│                          │
  │   { access_token,                                │
  │     refresh_token,                               │
  │     id_token,                                    │
  │     token_type: 'Bearer',                        │
  │     expires_in: 900,                             │
  │     scope: '...' }                               │
  │                                                  │
  │ 12. AT 存 sessionStorage                         │
  │                                                  │
  │ 13. 调 agent-server with Authorization: Bearer <AT>│
```

### 3.2 Refresh Token Rotation

```
SPA                    our-chat server          MySQL
  │                          │                     │
  │ POST /oauth/token        │                     │
  │ grant_type=refresh_token │                     │
  │ &refresh_token=<RT1>     │                     │
  │ &client_id=our-chat-web  │                     │
  │ ───────────────────────►│                     │
  │                          │ SELECT by jti       │
  │                          │ ───────────────────►│
  │                          │ ◄───────────────────│
  │                          │ 检查 revoked? expired?│
  │                          │ 检查 rotated_to 已填?│
  │                          │   ★ 已填 = 同一 RT  │
  │                          │      第二次使用 →   │
  │                          │      family 全部 invalidate│
  │                          │                     │
  │                          │ 正常路径:           │
  │                          │ 1. 签新 RT2(同 family_id)│
  │                          │ 2. UPDATE RT1 set   │
  │                          │    rotated_to=jti2  │
  │                          │ 3. INSERT RT2       │
  │                          │ 4. 签新 AT          │
  │ ◄────────────────────────│                     │
  │ { access_token: AT_NEW,                        │
  │   refresh_token: RT2 }                         │
```

### 3.3 Family Invalidation on Reuse

```
攻击者偷到 RT1,用一次:
  attacker ─RT1─► /oauth/token ─► RT1.rotated_to = jti(RT2_a)
                  ◄ { AT_a, RT2_a }

受害者下次正常 refresh:
  victim   ─RT1─► /oauth/token ─► SELECT RT1
                                  ◄ 发现 rotated_to 已填!
                                  ──► UPDATE oauth_refresh_tokens
                                       SET revoked = 1
                                       WHERE family_id = <RT1.family_id>
                                  ──► 攻击者持有的 RT2_a 也立即作废
                                  ──► 返回 invalid_grant
  victim ◄── 强制重新登录 + 审计日志告警
```

## 4. 与 our-chat 现有架构的衔接点

| 衔接点 | 说明 |
|---|---|
| **共享 user 表** | OAuth 签发的 token 里 `sub = users.id`,跟现有业务一致 |
| **共享登录页** | `/oauth/authorize` 未登录时跳 `/login?next=...`,登录成功跳回 |
| **共享 session 检测** | `/oauth/authorize` 通过 HttpOnly `token` cookie 判断是否已登录(只读,不签发新 cookie) |
| **不共用 token** | 业务 HS256 token 跟 OAuth RS256 token 完全独立,JwtSecret 互不相干 |
| **共用错误日志体系** | OAuth audit 走现有 logger,字段加 `module: 'oauth'` |

## 5. 部署形态(目标)

```
                    ┌────────────────────┐
                    │  Reverse Proxy     │
                    │  (nginx / Caddy)   │
                    └────────────────────┘
                              │
            ┌─────────────────┼──────────────────┐
            ▼                                    ▼
   ┌─────────────────┐                ┌─────────────────────┐
   │  our-chat web   │                │  our-chat server    │
   │  (Vite SPA)     │                │  + IdP module       │
   └─────────────────┘                │  RSA 私钥 / MySQL    │
                                      └─────────────────────┘
                                                ▲
                                                │ JWKS pull
                                                │
                                      ┌─────────────────────┐
                                      │  agent-server       │
                                      │  (resource server)  │
                                      └─────────────────────┘
```

- 单机 dev:全部 `localhost`,nginx 可省
- 生产:nginx 同根域路由(`chat.x.com` 给 web/server,`ai.x.com` 给 agent-server),JWKS 经互联网拉取
- 多机扩展:IdP 模块可水平扩展(只读 RSA 私钥 + 共享 MySQL),无 sticky session 需求
