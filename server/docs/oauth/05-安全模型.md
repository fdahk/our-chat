# 05 · 安全模型

## 1. 威胁模型

OAuth IdP 是高价值目标,因为它持有"签发凭证的能力"。系统化列出威胁:

| 威胁 | 资产 | 影响 | 本系统防御 |
|---|---|---|---|
| 私钥泄漏 | RSA 私钥 | 任意伪造 token | 文件 chmod 600 + secret volume + 不进 git + 启动 fail-fast |
| 数据库泄漏 | RT family 数据 | 无直接 token 泄漏(只是 jti) | RT 经签名 + DB 双校验,DB 数据不足以伪造 |
| Authorization code 拦截 | code | 换取 token | PKCE 强制 + code 短 TTL + 一次性 |
| Open redirect | redirect_uri | 钓鱼跳转 | 严格 exact match,拒绝任何通配 |
| CSRF on /authorize | 用户 session | 强制授权给攻击者 | state 强制 + same-site cookie |
| XSS in SPA | AT/RT 内存 / 存储 | 凭证外带 | 不在本 IdP 范围(SPA 侧责任),但 RT rotation 限制损失 |
| RT 偷取 | refresh_token | 永续刷新 | Family + rotation + reuse 检测 |
| Replay attack | code / id_token | 重复使用 | code 一次性,id_token nonce |
| Client secret 泄漏 | confidential client 密钥 | 假冒 client | bcrypt 哈希存储 + 不在日志输出 + 轮换流程 |
| Timing attack | client_secret 比较 | 字典攻击 | bcrypt.compare 自带 timing-safe |
| Token introspection 探测 | token 存在性 | 撞库 | 仅 confidential client 可调,且响应不区分"无效"和"不存在" |
| 暴力 / 高频 | /token 端点 | DOS / 撞 code | 限流(per IP + per client_id) |

## 2. PKCE 的角色与残留风险

### 2.1 PKCE 防什么

- ✅ Native URL scheme 抢注的 code 拦截
- ✅ Log / Referer 泄漏 redirect URL 后的 code 拦截
- ✅ 同设备恶意进程监听 redirect 的 code 拦截

### 2.2 PKCE 不防什么(本系统残留风险)

- ❌ SPA 进程内被攻陷(XSS / 供应链)→ verifier 与 token 全失守
- ❌ 物理设备被 root → Keychain / 内存皆可读
- ❌ 用户被钓鱼到假登录页 → IdP 凭证泄漏,跟 PKCE 无关

参考 [agent-server 跨服务鉴权方案 / 方案 F §11–14](../../../../agent-server/docs/backend/跨服务鉴权方案/方案F-OAuth2授权码PKCE.md#11-安全前提verifier-的可信度依赖)。

## 3. Refresh Token Rotation + Family Invalidation

**这是本系统对抗 RT 偷取的核心机制。** 详细原理见 [agent-server 跨服务鉴权方案 / 方案 G §5.3](../../../../agent-server/docs/backend/跨服务鉴权方案/方案G-BFF会话.md#53-那双-token-还剩什么价值rotation-是灵魂)。

### 3.1 触发逻辑(实现细节)

```
POST /oauth/token (grant_type=refresh_token)
  ↓
SELECT * FROM oauth_refresh_tokens WHERE jti = ?
  ↓
分支判断:
  A. 不存在               → invalid_grant
  B. revoked = 1           → invalid_grant + audit (this RT 已被人 rotation 或登出过)
  C. rotated_to IS NOT NULL → ★ REUSE DETECTED ★
        ↓
     UPDATE oauth_refresh_tokens
       SET revoked = 1, revoke_reason = 'reuse_detected'
       WHERE family_id = ? AND revoked = 0
     ↓
     audit.log({ level: 'WARN', event: 'rt_reuse_detected', family_id, user_id, ip, ua })
     ↓
     return invalid_grant
  D. expires_at < NOW()    → invalid_grant
  E. 都正常 → 进入 rotation 流程
```

### 3.2 正常 rotation(事务保护)

```sql
BEGIN;

INSERT INTO oauth_refresh_tokens (jti, family_id, ...) VALUES (?, ?, ...);  -- new RT

UPDATE oauth_refresh_tokens
  SET rotated_to = ?, rotated_at = NOW(), revoke_reason = 'rotation'
  WHERE jti = ?                                                              -- old RT
    AND rotated_to IS NULL;                                                  -- 防并发

-- 如果 UPDATE 影响行数 = 0,说明被并发抢先了 → rollback + 触发 reuse
IF ROW_COUNT() = 0 THEN
  -- 走 reuse 分支
END IF;

COMMIT;
```

**并发安全**:`rotated_to IS NULL` 在 UPDATE 的 WHERE 里,MySQL InnoDB 行锁保证唯一性。两个并发 refresh 必然只有一个成功 UPDATE。

## 4. 严格的 redirect_uri 校验

```
注册时:["https://chat.example.com/oauth/callback"]

/authorize 时收到:https://chat.example.com/oauth/callback
  → 字符串 exact match → 通过

/authorize 时收到:https://chat.example.com/oauth/callback?foo=bar
  → 不 exact match → 400 拒绝

/authorize 时收到:https://chat.example.com/oauth/callback/  (尾部斜杠不同)
  → 不 exact match → 400 拒绝

/authorize 时收到:https://chat.example.com.evil.com/oauth/callback (子域伪装)
  → 不 exact match → 400 拒绝
```

**不允许通配符 / 前缀 / 正则匹配。** RFC 8252 强制。

**在 /token 端点还要二次校验**:redirect_uri 必须跟 code 创建时存的一致。这防的是"同一 client 注册了多个 redirect_uri,攻击者拿到 code 后用别的 uri 兑换"。

## 5. 限流策略

### 5.1 维度

| 端点 | 限流 key | 阈值 | 行动 |
|---|---|---|---|
| `/oauth/token` | IP + client_id | 60 / min | 429 |
| `/oauth/token` reuse 命中 | user_id | 1 / hour 触发即冻结 | 立即冻结 + 告警 |
| `/oauth/authorize` | IP | 30 / min | 429 |
| `/oauth/introspect` | client_id | 600 / min | 429 |
| `/oauth/userinfo` | user_id | 60 / min | 429 |

### 5.2 实现

经现有 our-chat 限流中间件(若有)或 Redis token bucket。本模块预留接口,默认不开(由部署方按流量决定)。

## 7. 用户登出与全设备登出

### 7.1 单设备登出

```
POST /oauth/revoke
  token = <current refresh_token>
  → revoke 该 jti
```

### 7.2 全设备登出(用户主动 / 改密码触发)

```sql
UPDATE oauth_refresh_tokens
  SET revoked = 1, revoke_reason = 'logout_all'
  WHERE user_id = ? AND revoked = 0;
```

> AT 仍然 stateless,短 TTL 内会"残留",这是有意识的取舍 —— 用 introspection 端点可让 resource server 实时校验(但有 1 跳网络成本)。
