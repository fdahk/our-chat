# 06 · 开发规范

## 1. 代码风格

### 1.1 模块化

- 每个端点一个文件,不堆 `oauth/routes.ts`
- 每个文件只暴露一个 `RequestHandler`(或一组紧密相关的 RequestHandler)
- 共享逻辑放 `keys.ts` / `clients.ts` / `storage.ts` / `pkce.ts`

### 1.2 错误处理

- 所有端点 try/catch,任何未捕获错误 → 500 `server_error`,**不暴露细节**
- 错误响应必须符合 [03-API契约 §0.1](./03-API契约.md#01-错误响应格式rfc-6749-52)
- 业务错误用 `OAuthError` 类(`src/oauth/errors.ts`),自带 HTTP code + error code 映射

```ts
throw new OAuthError('invalid_grant', 'Code already used');
// 中间件转换为:
// 400 { error: 'invalid_grant', error_description: 'Code already used' }
```

### 1.3 注释

- 注释只写**业务逻辑 / 协议要求**,不写"这一段处理 XXX"这种废话
- 涉及 RFC 章节的逻辑必须标注:
  ```ts
  // RFC 6749 §4.1.3: redirect_uri 必须与 /authorize 时一致
  if (storedCode.redirect_uri !== body.redirect_uri) { ... }
  ```
- 不写里程碑 / 章节号(如 "M5 阶段"、"对应 02 文档 §3")

### 1.4 命名

| 概念 | 命名 |
|---|---|
| code 表行 | `OAuthCode` |
| RT 表行 | `OAuthRefreshToken` |
| client 表行 | `OAuthClient` |
| 解码后的 access token payload | `AccessTokenClaims` |
| 解码后的 refresh token payload | `RefreshTokenClaims` |
| id_token payload | `IdTokenClaims` |
| `grant_type=authorization_code` 的处理函数 | `handleAuthorizationCodeGrant` |
| `grant_type=refresh_token` 的处理函数 | `handleRefreshTokenGrant` |

## 2. 测试规范

### 2.1 覆盖范围(最低线)

| 测试类型 | 覆盖范围 | 工具 |
|---|---|---|
| 单测 | `pkce.ts` / `keys.ts` / `storage.ts` / `errors.ts` 等纯逻辑 | vitest + mock |
| 端点单测 | 每个端点的 happy path + 各错误分支 | vitest + supertest + mock mysql |
| 集成测试 | 完整 PKCE flow(authorize → callback → token → refresh → revoke) | vitest + 真实 MySQL container |
| 安全测试 | 重用 RT / 已用 code / PKCE 不匹配 / open redirect 拒绝 | 同上 |
| 验签测试(agent-server) | JWKS pull + token verify | agent-server 集成测试 |

### 2.2 测试结构

```
test/oauth/
├── unit/
│   ├── keys.test.ts                RSA 密钥加载 / kid 索引 / JWKS 生成
│   ├── pkce.test.ts                S256 校验、verifier 长度边界、错 method 拒绝
│   ├── clients.test.ts             client 校验、redirect_uri 严格匹配
│   ├── storage.test.ts             code / RT CRUD,reuse 检测的 SQL 路径
│   └── errors.test.ts              OAuthError → 响应映射
│
├── endpoint/
│   ├── discovery.test.ts           响应内容快照
│   ├── jwks.test.ts                响应内容、缓存头
│   ├── authorize.test.ts           各种参数错 / 未登录跳转 / 成功生成 code
│   ├── token-auth-code.test.ts     authorization_code grant 全分支
│   ├── token-refresh.test.ts       refresh_token grant + rotation
│   ├── revoke.test.ts
│   ├── introspect.test.ts

└── integration/
    ├── flow-happy-path.test.ts      端到端 PKCE 全流程
    ├── flow-reuse-attack.test.ts    模拟攻击者偷 RT 后 family 被烧
    └── flow-key-rotation.test.ts    双 kid 共存期的验签
```

### 2.3 测试覆盖率门槛

- 单测分支覆盖率 ≥ 90%
- 端点测试每个分支至少一条用例
- 集成测试覆盖完整 happy / reuse / 多个错误恢复路径

### 2.4 测试中的密钥

- 用一对**测试专用** RSA 密钥(`test/fixtures/test-rsa-private.pem`),固定不变
- 不要每次测试现生成密钥(slow,而且 JWT 测试结果不可对照)

### 2.5 性能测试

- `/oauth/token` 单端点 throughput baseline:1000+ req/s 单机
- JWKS endpoint:静态字符串,理论上 nginx 都能直接缓存

## 3. 提交规范

- 中文,`type(scope): message`,如 `feat(oauth): authorization code 端点 + PKCE 校验`
- 一个 PR 解决一类事情:不混 "加端点" 和 "重构存储"
- 提交前必须通过:
  ```
  pnpm typecheck && pnpm test && pnpm lint
  ```
- 涉及 schema 改动必须附 migration SQL + 回滚脚本

## 4. 数据库 migration 规范

`src/database/migrations/NNN_<动作>.sql`:

- `NNN` 自增三位
- 文件内容必须**幂等**(可重复执行不报错):`CREATE TABLE IF NOT EXISTS`
- 一个文件做一件事(便于回滚)
- 文件顶部注释写"目的 / 回滚方式"

```sql
-- 001_oauth_tables.sql
-- 目的:新增 OAuth IdP 三张表
-- 回滚:DROP TABLE oauth_refresh_tokens; DROP TABLE oauth_codes; DROP TABLE oauth_clients;

CREATE TABLE IF NOT EXISTS oauth_clients (...);
CREATE TABLE IF NOT EXISTS oauth_codes (...);
CREATE TABLE IF NOT EXISTS oauth_refresh_tokens (...);
```

## 5. 错误码与日志的耦合

| HTTP | error | level | event |
|---|---|---|---|
| 400 | invalid_request | INFO | param_invalid |
| 400 | invalid_grant(常规) | INFO | grant_rejected |
| 400 | invalid_grant(reuse 命中) | **WARN** | rt_reuse_detected |
| 401 | invalid_client | INFO | client_rejected |
| 500 | server_error | **ERROR** | internal_error |

reuse_detected 必须 **WARN** + 触发监控告警通道。

## 6. 配置 / Env

| 变量 | 默认 | 说明 |
|---|---|---|
| `OAUTH_ISSUER_BASE_URL` | `http://localhost:3007` | iss claim + Discovery URL 基地 |
| `OAUTH_PRIVATE_KEY_FILE` | `keys/oauth-private-dev.pem` | 私钥路径 |
| `OAUTH_ACTIVE_KID` | `dev` | 当前签发用的 kid |
| `OAUTH_RETIRED_KIDS` | `` | 仍发公钥但不签新的 kid 列表(逗号分隔) |
| `OAUTH_AT_TTL_SEC` | `900` | access token TTL |
| `OAUTH_RT_TTL_SEC` | `2592000` | refresh token TTL(30 day) |
| `OAUTH_CODE_TTL_SEC` | `60` | authorization code TTL |

`.env.example` 必须**完整列出**所有变量。
