# 01 · 生产部署最佳实践与演进路线图

> **本文目标**:把"生产部署"做完整盘点。回答两个问题:
> 1. 当前 Docker + GitHub Actions 在工业界部署成熟度的哪一级?
> 2. 完整最佳实践长什么样,我们怎么演进?
>
> **诚实回答先放在前面**:当前是 **L2 雏形**(Level 2),离工业界的"最佳实践"(L3-L4)还有显著距离。但 L4 对个人项目过度,**L3 是合理目标**。本文给完整路线图。

---

## 0. 部署成熟度的 5 个等级

| Level | 名字 | 适合规模 | 典型工具栈 | 业界代表 |
|---|---|---|---|---|
| L1 | 单机 Docker Compose | 个人 / 演示 / hobby | `docker-compose up` 在 VPS | 个人博客 / 大量 GitHub side project |
| L2 | Container Registry + 手工部署 | 小团队 / 单产品 | GHCR/Docker Hub + SSH + `docker pull && docker run` 或 PaaS(Fly.io/Render/Railway) | YC 早期初创、独立开发者 |
| L3 | Kubernetes + GitOps + 全套可观测 | **中型团队 / 严肃生产** | K8s + ArgoCD + Prometheus + Grafana + Loki + OTel + Secret manager + IaC | Notion / Linear / Vercel 内部 / 大部分 SaaS |
| L4 | Service Mesh + 渐进发布 + 多 region | 大厂 / 多团队 | L3 + Istio/Linkerd + Argo Rollouts + Multi-region + 灾备 | Stripe / Shopify / Slack |
| L5 | 自研 PaaS + Cell-based | 超大规模 | 自研容器调度 / Service Mesh / 监控 | Google / Meta / 阿里 / 字节 |

### 当前定位

- ✅ L1 已完成(`docker-compose up`)
- ✅ L2 雏形(Dockerfile + GHCR push 经 `.github/workflows/cd.yml`,**但没有真实部署目标**)
- ❌ L3 没开始
- ❌ L4 / L5 不考虑(过度)

**对个人项目 + 校招 portfolio 的合理目标:做到 L3**。简历可写"基于 K8s + ArgoCD + OTel 的完整生产部署栈",而不是"会用 Docker"。

## 2. L3 完整最佳实践 ── 每个组件展开

下面**对每个 L3 组件**讲清楚:**是什么 / 业界谁在用 / 我们怎么集成 / 关键踩坑**。这是个人项目 + 校招的合理深度,L4 简单提一下。

### 2.1 容器编排:Kubernetes

#### 是什么

Kubernetes 是把"声明性 YAML → 真实运行的容器集群"的控制循环系统。核心抽象:

- **Pod**:一组共享网络/存储的容器(应用 + sidecar)
- **Deployment**:一组同构 Pod 的期望状态 + 滚动升级策略
- **Service**:稳定的虚拟 IP / DNS,把流量路由到 Pod
- **Ingress**:把外部 HTTP 流量路由到 Service(配合 Nginx Ingress / Traefik)
- **ConfigMap / Secret**:配置 / 凭据注入
- **Job / CronJob**:一次性 / 定时任务

#### 选型:K3s vs EKS/GKE/AKS

| 方案 | 月成本 | 复杂度 | 适合场景 |
|---|---|---|---|
| **K3s** 单节点 | $5-10(VPS) | 低,一行命令装 | 个人项目 / 学习 |
| **K3s** 多节点 | $20-50 | 中 | 小型 SaaS |
| **AWS EKS** | $73 + 节点 | 中-高 | 严肃生产,云原生 |
| **GCP GKE Autopilot** | 按 Pod 计费 | 低-中 | 中小型生产,运维省心 |
| **Azure AKS** | 节点费 | 中 | Azure 系生态 |
| **DigitalOcean Kubernetes** | $12 + 节点 | 低 | 个人项目 / 初创 |

**对我们这个项目**:推荐 **K3s on VPS**(DigitalOcean / Hetzner / Vultr,$5-10/月)。生产级 K8s API 完整,学习/部署成本极低。

#### 关键概念扫盲

- **声明式 ≠ 命令式**:你提交 YAML 说"我要 3 个 Pod",K8s 自己驱动到这个状态。你不直接说"启动一个、停止一个"
- **控制循环**:每个 Controller 持续对比"期望 spec"与"实际 status",有差异就调和(reconcile)。这是 K8s 全部威力的来源
- **etcd**:K8s 的"数据库",存所有 YAML 状态
- **Operator pattern**:用户定义 CRD(Custom Resource Definition),写自己的 Controller 管自定义资源(如 PostgresOperator 管 PG 集群)

#### 业界对照

- GitHub.com 跑在自研编排(早期),新业务上 K8s
- GitLab.com 自家 K8s
- Notion 内部 K8s
- Discord 早期用 Docker 后切 Twine(自研),后又 K8s 混合
- 国内大厂普遍 K8s(腾讯 TKE / 阿里 ACK / 字节火山)

#### 我们集成的最小 YAML

```yaml
# k8s/server-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: our-chat-server }
spec:
  replicas: 2
  selector: { matchLabels: { app: our-chat-server } }
  template:
    metadata: { labels: { app: our-chat-server } }
    spec:
      initContainers:
        - name: migrate
          image: ghcr.io/fdahk/our-chat-server:latest
          command: ['npx', '--no-install', 'prisma', 'migrate', 'deploy']
          envFrom: [{ secretRef: { name: db-secret } }]
      containers:
        - name: server
          image: ghcr.io/fdahk/our-chat-server:latest
          ports: [{ containerPort: 3007 }]
          envFrom:
            - { secretRef: { name: db-secret } }
            - { secretRef: { name: app-secret } }
          readinessProbe:
            httpGet: { path: /health, port: 3007 }
            periodSeconds: 5
          livenessProbe:
            httpGet: { path: /health, port: 3007 }
            periodSeconds: 10
          resources:
            requests: { cpu: 100m, memory: 256Mi }
            limits: { cpu: 1000m, memory: 1Gi }
---
apiVersion: v1
kind: Service
metadata: { name: our-chat-server }
spec:
  selector: { app: our-chat-server }
  ports: [{ port: 80, targetPort: 3007 }]
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: our-chat-server
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt
spec:
  tls: [{ hosts: [api.our-chat.com], secretName: tls }]
  rules:
    - host: api.our-chat.com
      http:
        paths:
          - { path: /, pathType: Prefix, backend: { service: { name: our-chat-server, port: { number: 80 } } } }
```

#### 关键踩坑

- **资源 requests/limits 必须设**:否则一个 Pod OOM 拖死整个 Node
- **readiness ≠ liveness**:readiness 失败 = 不进 Service 路由(但 Pod 仍跑);liveness 失败 = 重启 Pod。混用会无限重启循环
- **initContainers 跑完才启动主容器**:migration 是经典用法
- **PG / Redis 等有状态服务别上 K8s**:用云托管(RDS / ElastiCache)更省心。K8s 上跑 Stateful workload 是高级议题

### 2.2 GitOps:ArgoCD(部署模式的范式转换)

#### 是什么 ── Pull vs Push 的根本区别

**Push-based**(传统 CI/CD):
```
GitHub Actions → kubectl apply / Helm upgrade → 推到 K8s
```
- CI 持有 K8s 凭据(攻击面)
- 应用状态不可追溯("当前生产是哪个 commit?要看 CI 日志")
- 回滚要重跑 pipeline

**Pull-based**(GitOps):
```
                                ┌─────────┐
GitHub repo (yaml) ──────────►  │ ArgoCD  │ ───watches──► Git
                                │ in K8s  │ ───reconcile──► K8s API
                                └─────────┘
```
- ArgoCD 跑在 K8s 集群内,**主动拉** Git 的最新 YAML
- "生产状态 = Git 的 main 分支" ── 永远可追溯
- 回滚 = `git revert` + ArgoCD 自动应用
- CI 没有 K8s 凭据(只 build & push image)

#### 选型:ArgoCD vs Flux

| 维度 | ArgoCD | Flux |
|---|---|---|
| UI | ✅ 强大 Web UI | ❌ 无内建(可加 Weave GitOps UI) |
| 文档 | ✅ 易学 | ✅ |
| 多集群 | ✅ ApplicationSet | ✅ |
| CNCF 状态 | Graduated | Graduated |
| 业界采用 | Intuit / Red Hat / 大量企业 | Weaveworks / GitLab Auto-DevOps |

**推荐 ArgoCD**:UI 友好,概念清晰,学习曲线低。

#### 文件结构

```
infra/                          ← 单独的 GitOps repo(或同 repo 子目录)
├── apps/
│   ├── our-chat-server/
│   │   ├── deployment.yaml
│   │   ├── service.yaml
│   │   ├── ingress.yaml
│   │   └── kustomization.yaml
│   └── postgres/
│       └── ...
└── argocd-apps.yaml             ← ArgoCD Application 定义
```

ArgoCD Application:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: our-chat-server
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/fdahk/our-chat-infra
    path: apps/our-chat-server
    targetRevision: main
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true       # 删 Git 里删掉的资源
      selfHeal: true    # 自动 reconcile 手动改动
```

#### 业界对照

- GitLab 自家用 Flux
- Intuit(QuickBooks)创立 ArgoCD,现 LF/CNCF
- Red Hat OpenShift 内建 ArgoCD
- 几乎所有现代 K8s 教程都教 GitOps

#### 关键踩坑

- **不要直接 kubectl apply**:破坏 GitOps 不变量。ArgoCD 会 self-heal 把你的改动覆盖回去
- **Secret 在 Git 里要先加密**:用 SOPS 或 Sealed Secrets(详见 §2.3)
- **多环境(dev/staging/prod)用 Kustomize overlay**,不要 fork repo

### 2.3 Secret 管理:External Secrets Operator + 云 KMS

#### 痛点

```yaml
apiVersion: v1
kind: Secret
metadata: { name: db-secret }
data:
  DATABASE_URL: cG9zdGdyZXM6Ly91c2VyOnBhc3M=    # ← base64,跟明文等价
```

这种 Secret 进 Git 仓库等于密码明文公开。

#### 解法

**External Secrets Operator (ESO)** + 云 secret manager:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata: { name: db-secret }
spec:
  refreshInterval: 1h
  secretStoreRef: { name: vault, kind: ClusterSecretStore }
  target: { name: db-secret }    # ← 自动同步成 K8s Secret
  data:
    - secretKey: DATABASE_URL
      remoteRef: { key: prod/database/url }
```

ESO Pod 持续从 Vault / AWS Secrets Manager / GCP Secret Manager 拉真值,同步成 K8s Secret。

**Git 里只有引用,没有明文**。

#### 选型

| 方案 | 适合 |
|---|---|
| **HashiCorp Vault** | 自托管,功能最全,业界事实标准 |
| **AWS Secrets Manager** | AWS 系 |
| **GCP Secret Manager** | GCP 系 |
| **Doppler** | SaaS,小团队简单 |
| **1Password Secrets Automation** | 团队已用 1Password |
| **Sealed Secrets**(替代方案) | 不引入外部依赖,用非对称加密把 Secret 直接进 Git |

**对个人项目**:**Doppler 免费层** 或 **Sealed Secrets** 起步。

#### 业界对照

- Vault 是 HashiCorp 当家产品,Stripe / Shopify / Adobe 大量使用
- AWS Secrets Manager 是默认选项(AWS 上)
- ESO 是 CNCF 项目,2024 已成事实标准

#### 关键踩坑

- **Secret rotation 要演练**:DB 密码改了,K8s Secret 自动更新,但 Pod 不会自动重启读新值。需要 `reloader` 或主动 rollout
- **Vault Token 怎么入手**:鸡生蛋问题。一般方案:K8s ServiceAccount Token → Vault K8s Auth Method 换 Vault Token
- **不同环境完全隔离**:dev/staging/prod 三个 Vault 路径,K8s ClusterSecretStore 区分

### 2.4 可观测性三件套:Metrics / Logs / Traces

#### 概念

OpenTelemetry 把可观测性分三类信号:

| 信号 | 看什么 | 工具 | 我们要做 |
|---|---|---|---|
| **Metrics**(指标) | 数值时序(QPS / P95 延迟 / 内存) | Prometheus + Grafana + AlertManager | `/metrics` 端点 + scrape |
| **Logs**(日志) | 文本事件流 | Loki + Promtail / Vector | stdout JSON → Loki |
| **Traces**(分布式追踪) | 请求穿过多个服务的完整链路 | OpenTelemetry + Tempo / Jaeger | OTel SDK instrument |

#### Metrics:Prometheus + Grafana

##### 应用侧改动

```ts
// src/metrics.ts
import { collectDefaultMetrics, Counter, Histogram, register } from 'prom-client';
collectDefaultMetrics({ prefix: 'our_chat_' });

export const httpRequests = new Counter({
  name: 'our_chat_http_requests_total',
  help: 'HTTP requests',
  labelNames: ['method', 'route', 'status'] as const,
});

export const httpDuration = new Histogram({
  name: 'our_chat_http_request_duration_seconds',
  help: 'HTTP request duration',
  labelNames: ['method', 'route', 'status'] as const,
  buckets: [0.005, 0.01, 0.05, 0.1, 0.5, 1, 5],
});

// app.ts
app.get('/metrics', async (_req, res) => {
  res.setHeader('Content-Type', register.contentType);
  res.send(await register.metrics());
});
```

##### Prometheus scrape 配置

```yaml
scrape_configs:
  - job_name: our-chat-server
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_label_app]
        regex: our-chat-server
        action: keep
```

##### Grafana 面板必备

- HTTP 请求 QPS(by route)
- HTTP P50/P95/P99 延迟
- 错误率(5xx %)
- DB 连接池占用
- Prisma 查询延迟
- Node 进程:CPU / RSS / Heap / Event Loop Lag

##### 关键踩坑

- **指标基数(cardinality)爆炸**:`labelNames` 不要放 user_id / request_id 这种无限维度,否则 Prometheus 内存爆
- **Histogram bucket 要精心设**:跟 SLO 对齐(如果你的 SLO 是 "P99 < 500ms",bucket 一定要在 0.5 附近密集)

#### Logs:Loki + Promtail

##### 为什么 Loki 而非 ELK

- Loki:**索引只 label,不索引 log content**。便宜 10x
- ELK:全文索引,贵但搜得快
- 中型项目 Loki 够用

##### 应用日志要结构化

```ts
// src/logger.ts
import pino from 'pino';
export const logger = pino({
  level: process.env.LOG_LEVEL ?? 'info',
  formatters: {
    level: (label) => ({ level: label }),
  },
  // 生产 JSON,dev pretty
  ...(process.env.NODE_ENV === 'production' ? {} : {
    transport: { target: 'pino-pretty' },
  }),
});
```

每条 log 强制带:`request_id` / `user_id` / `service` / `trace_id`。

##### 关键踩坑

- **千万别 console.log JSON 字符串拼接**:用 logger.info({ key: val }, msg) 让 logger 序列化
- **Promtail 解析 K8s label 当 Loki label**:让"按 namespace 过滤"等可用
- **PII(用户信息)别进日志**:GDPR / 合规

#### Traces:OpenTelemetry + Tempo

##### 为什么需要

请求 `POST /oauth/token` 经过:authorize → consumeCode → loadProfile(PG)→ signAccessToken → insertRefreshToken(PG)。

要知道**慢在哪一步**,只看 metrics 拍脑袋,要看 trace。

##### 应用侧

```ts
// src/tracing.ts
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';

const sdk = new NodeSDK({
  serviceName: 'our-chat-server',
  instrumentations: [getNodeAutoInstrumentations()],
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_ENDPOINT,    // 指向 Tempo / Jaeger
  }),
});
sdk.start();
```

auto-instrumentation 会**自动**给 Express / Prisma / HTTP / Socket.IO 加 span,几乎零业务侵入。

##### 业界对照

- OpenTelemetry 是 CNCF Graduated,2024 已是事实标准
- Datadog / NewRelic 等商业 APM 都接 OTel
- Grafana Tempo / Jaeger 是开源 backend

##### 关键踩坑

- **采样率**:生产环境 1% 起步,否则 trace 量爆炸
- **trace_id 注入 log**:logger 自动从 OTel context 取,日志和 trace 才能关联
- **跨服务传播**:用 W3C Trace Context 标准 header,不用自家 header

### 2.5 基础设施即代码:Terraform / Pulumi

#### 痛点

云资源(VPC / RDS / K8s 集群 / DNS)手工 console 点 → 不可复现 / 不可审计 / 不可灾备。

#### 解法

**Terraform**(HCL DSL):

```hcl
resource "digitalocean_kubernetes_cluster" "main" {
  name    = "our-chat-prod"
  region  = "sgp1"
  version = "1.30.0-do.0"
  node_pool {
    name       = "default"
    size       = "s-2vcpu-4gb"
    node_count = 2
  }
}

resource "digitalocean_database_cluster" "postgres" {
  name       = "our-chat-pg"
  engine     = "pg"
  version    = "16"
  size       = "db-s-1vcpu-1gb"
  region     = "sgp1"
  node_count = 1
}
```

`terraform apply` 一键起整套云资源。改 HCL → 提交 PR → 评审 → 再 apply,所有变更可追溯。

#### Terraform vs Pulumi

| 维度 | Terraform | Pulumi |
|---|---|---|
| 语言 | HCL(声明) | TypeScript / Go / Python(命令式) |
| 学习曲线 | 简单 | 中(但能复用项目语言) |
| 生态 | 巨大,几乎所有云都支持 | 也大,API 跟 Terraform 等价 |
| 业界采用 | 事实标准 | 增长快 |

**推荐 Terraform**(社区资源最多)。前端工程师如果不想学 HCL,Pulumi + TS 是合理替代。

#### 业界对照

- Datadog / Stripe / Shopify 全部 Terraform
- HashiCorp 自家产品(Terraform)是 IaC 第一
- Crossplane 是新派(K8s CRD 描述云资源,GitOps 风格)

#### 关键踩坑

- **state 文件要 remote**:Terraform Cloud / S3 + DynamoDB lock,不能本地
- **`terraform destroy` 是核武器**:生产分支严格 review,加 `prevent_destroy = true` lifecycle 防误删
- **circular dependency**:VPC → 子网 → K8s → ALB,顺序要对

### 2.6 Progressive Delivery:Argo Rollouts

#### 概念

| 策略 | 描述 | 适合 |
|---|---|---|
| **Rolling Update** | 一个个替换 Pod | 默认,大部分场景 |
| **Blue-Green** | 全套新版本起来 → 流量切换 → 旧版本下线 | DB schema 不兼容时 |
| **Canary** | 先 5% 流量给新版本 → 看指标 → 逐步扩大 | 风险高 / 想 A/B test |

K8s 原生只支持 Rolling。Canary / Blue-Green 要 **Argo Rollouts** 或 **Flagger**。

#### 示例:Canary

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata: { name: our-chat-server }
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5        # 5% 流量给新版本
        - pause: { duration: 10m }
        - setWeight: 20
        - pause: { duration: 10m }
        - setWeight: 50
        - pause: { duration: 10m }
        - setWeight: 100
      analysis:
        templates:
          - templateName: error-rate-check    # 自动看 Prometheus,error rate 涨了自动回滚
```

#### 业界对照

- Netflix 自家 Spinnaker(Canary 鼻祖)
- Argo Rollouts 是 GitLab / Intuit 主流
- Flagger 是 Weaveworks 出,跟 Flux 配套

---

### 2.7 Policy as Code:OPA Gatekeeper / Kyverno

#### 为什么需要

没有约束的 K8s 集群,任何人能:
- 跑 root container
- 申请 100GB 内存
- 不设 resource limits
- 用 `latest` tag

久了集群就乱。Policy as Code 让"集群规则"也声明式。

#### 示例(Kyverno)

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata: { name: require-resources }
spec:
  validationFailureAction: enforce
  rules:
    - name: require-cpu-memory
      match:
        resources: { kinds: [Pod] }
      validate:
        message: '必须设置 resources.requests 和 limits'
        pattern:
          spec:
            containers:
              - resources:
                  requests: { cpu: '?*', memory: '?*' }
                  limits:   { cpu: '?*', memory: '?*' }
```

任何不设资源限制的 Pod 会被 admission webhook 拒绝。

#### 业界对照

- OPA(Open Policy Agent)是 CNCF Graduated,通用策略引擎
- Gatekeeper 是 OPA 的 K8s 集成
- Kyverno 是 K8s-native(更易学,YAML 写 policy)

**推荐 Kyverno**(YAML 更直观,无需学 Rego)。

### 2.8 SLO/SLI/Error Budget(Google SRE 标杆)

#### 概念

- **SLI (Indicator)**:可量化的服务质量(error rate / latency P99)
- **SLO (Objective)**:目标(99.9% 请求成功 / P99 < 500ms)
- **Error Budget**:可用的"违反 SLO 额度"(99.9% = 一周 ≤ 10 分钟挂)

#### 用途

Error budget 没用完 → 可以发新版本(承担风险)
Error budget 用完了 → **禁止部署新版本**(必须先修稳定性)

这是 Google SRE 的核心机制,让工程团队和产品团队对"风险 vs 速度"有量化对话。

#### 工具

- **Sloth**:Prometheus + Grafana 上跑 SLO,YAML 定义
- **OpenSLO**:CNCF 提的 SLO 标准

## 3. L4 (overkill 警告)简述

L4 在 L3 基础上加:

- **Service Mesh (Istio / Linkerd)**:服务间 mTLS / 流量切分 / 熔断 / 重试,**控制平面 + 数据平面 (sidecar)**。运维复杂度大幅上升。Linkerd 比 Istio 轻量
- **Multi-region 灾备**:跨 region 异步复制 + 自动 failover。需要 Spanner-like DB 或者跨 region eventual consistency
- **Cell-based architecture**:把用户/数据按 cell 分隔,故障只影响一个 cell。AWS / Slack / Discord 都用
- **混沌工程**:Chaos Mesh / Litmus 主动注入故障,验证容错

**个人项目 / 校招完全不需要**。L4 用于"百万 DAU + 9 个 9 可用性"场景。

## 7. 关键概念扫盲(简历面试常问)

### Q1:Push 部署和 Pull 部署的区别?

Push:CI 拿 K8s 凭据,主动 kubectl apply。问题:凭据外泄面大,生产状态不在 Git。
Pull:ArgoCD 跑在集群内,主动拉 Git。优点:凭据不出集群,Git 是单一真相源。

### Q2:Liveness 和 Readiness 探针的区别?

Liveness 失败 → 重启 Pod(进程僵死时用)。
Readiness 失败 → 暂时不进 Service 路由(启动中、DB 暂时断了时用)。
**混用会无限重启**:DB 抖动 → readiness 触发 liveness → Pod 重启 → 启动需要 DB → 又失败,死循环。

### Q3:为什么应用启动时跑 migration 是反模式?

- 多副本并发抢锁
- migration 失败 = 应用 CrashLoopBackOff = 雪崩
- 回滚困难
- 违反 12-factor build/release/run 分离

标准做法:init container / CI job。详见 `docs/database/05`。

### Q4:为什么 K8s 是声明式而非命令式?

声明式 = 你说"我要什么状态",K8s 自己驱动到那个状态。任何时候 spec ≠ status,Controller 调和。这让:
- 多人协作可预测(YAML diff 直接看)
- 故障自动恢复(节点挂了,Controller 自动重调度)
- GitOps 成为可能(Git 状态 = 集群状态)

### Q5:Sidecar pattern 是什么?

一个 Pod 跑两个容器:**主容器(业务)+ sidecar(辅助)**。共享网络 / 存储,独立生命周期。
经典用例:Istio Envoy(流量代理)、Fluent Bit(日志)、cloud-sql-proxy(DB 连接)。

### Q6:什么是 Operator Pattern?

K8s 内建 Controller 管 Pod / Deployment 等通用资源。但如果想管"PostgreSQL 集群"这种复杂资源,可以:
1. 定义 CRD(`kind: PostgresCluster`)
2. 写自己的 Controller(Operator)
3. 用户提交 CR,Operator 把它转成 StatefulSet + Service + Secret 等
经典 Operator:Postgres Operator / Cert-Manager / Prometheus Operator。

### Q7:Helm vs Kustomize?

- **Helm**:模板引擎 + 包管理器。强大但 Go template 易乱
- **Kustomize**:K8s 原生 overlay,YAML patch YAML。简单但功能少
**业界趋势**:简单场景 Kustomize,复杂场景 Helm + Helmfile 编排
