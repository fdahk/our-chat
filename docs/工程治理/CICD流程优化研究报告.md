# CI/CD 流程优化研究报告 —— 国内弱网环境下的高效可靠部署

> 目标:把"提交即自动部署"做到**弱网也稳、增量也快、零手工**。本报告先做问题诊断与手段全景,
> 再给本项目的决断方案与落地顺序,最后附实战踩坑。读者:维护本部署链路的工程师。

---

## 0. 摘要(TL;DR)

**现状痛点**:GitHub Actions 构建镜像没问题,卡在**国内服务器拉镜像**这一段——弱网 + 大镜像 + 依赖第三方公益镜像源,导致 `docker pull` 反复卡死、`Run Command Timeout`,部署不可靠。

**根因(一句话)**:镜像分发链路不稳。具体是四叠加——① 镜像源不可靠(公益代理)② 镜像偏大(server 778MB 带全量 devDeps)③ 构建缓存易失效(GHA 7 天清,稀疏部署必失效)④ 拉取无容错(一次卡死=整次部署崩)。

**决断方案(按收益/成本排序)**:

| 优先级 | 措施 | 解决什么 | 成本 |
|---|---|---|---|
| P0 | **拉取重试 + 断点续传** | 弱网卡顿不再一击毙命 | 改部署脚本几行,立即可做 |
| P1 | **换同云 registry(腾讯 TCR)** | 根治拉取不稳(广州↔广州内网) | 开通+改 CI,需账号凭据 |
| P2 | **镜像瘦身(prod-only 依赖)** | 减少每次传输量 | 改 Dockerfile,需 build 验证 |
| P3 | **registry 构建缓存** | 依赖层跨多次部署长期稳定、不再无谓重下 | 改 CI 缓存配置 |

**组合效果**:小镜像 + 同城源 + 只传增量 + 失败续传 = 代码改动时只传几 MB、弱网也能稳定 1–3 分钟完成。

## 1. 背景:部署架构与它的演进史

### 1.1 当前架构
```
push → GitHub Actions(海外 runner)
        ├─ build:构建镜像(多阶段)
        └─ push:推到镜像仓库
                         │
                         ▼
            SSH 到国内服务器(腾讯云·广州)
            生成 .env → docker compose 拉镜像 → up 起容器
```
两个仓库(our-chat IM 主站 + agent-server AI 后端)同机部署,经 `oc-shared` 网络互通。

### 1.2 演进史(每一步都是被国内网络逼出来的)
- **GHCR**:registry 的 API/manifest 国内能访问,但**镜像层(blob)走 GitHub 的 CDN,国内被墙** → 拉取 0 字节进度。验证:`docker pull` 105s 零进度,而 `https://ghcr.io/v2/` 0.09s 就响应。
- **Docker Hub + daocloud 公益镜像源**:能拉了(复用 `registry-mirrors`),但**公益源对大层(milvus 1.5G、server node_modules 上百 MB)频繁抖动/卡死**。
- **ICP 备案通过**:解决了**对外**访问(80/443 不再被云厂商边缘封),但**不解决出站拉镜像**——备案管的是入站 Web,镜像拉取是出站,两码事。
- **结论**:**出站拉镜像的可靠性,是本 CICD 的命门**。本报告围绕它展开。

## 2. 把"一次部署"拆开:阶段与瓶颈(MECE)

部署 = **构建 → 推送 → 拉取 → 启动**。逐段定位成本与风险:

| 阶段 | 在哪执行 | 主要成本 | 风险等级 |
|---|---|---|---|
| 构建 build | GitHub runner(海外,带宽好) | CPU + 构建缓存命中率 | 中(缓存失效→重装依赖,变慢但不致命) |
| 推送 push | runner → registry(海外→仓库,快) | 带宽 | 低 |
| **拉取 pull** | **国内服务器 ← registry** | **带宽(弱网)** | **★★★ 卡死/超时,部署命门** |
| 启动 up | 服务器本地 | 健康检查/依赖等待 | 低(本地操作) |

**瓶颈毫无悬念:拉取阶段。** 它跨越"国内弱网 ← 不可靠 registry"这条最差的链路。所以**所有优化的第一性原理是:让"拉取"这段要么不发生(命中缓存)、要么很小(瘦身)、要么很快(同城源)、要么失败能续(容错)。**

---

## 3. 什么叫"高效的 CICD":评价维度

不先定义指标,优化就是拍脑袋。本项目关心五个维度,**本次核心 KPI 是「弱网成功率」**:

1. **可靠性 / 弱网成功率**(本次核心):弱网下一把过的概率。
2. **端到端时长**:push 到容器健康的 wall-clock。
3. **增量效率**:理想是"改代码只传几 MB,依赖层命中缓存"。
4. **自动化程度**:零手工、提交即部署、续期/网络等无需人盯。
5. **可复现 / 可回滚**:镜像不可变 + 精确 tag,能一键回退。

## 4. 优化手段全景(MECE,按链路分层)+ 业界对比

把手段按"作用在链路哪一层"分类,互斥不重叠:

### 4.1 镜像分发层(registry 选型)—— 收益最大
**问题**:从哪个仓库拉、链路通不通快不快。

| 选项 | 国内可靠性 | 成本 | 评价 |
|---|---|---|---|
| GHCR | ❌ blob 被墙 | 免费 | 国内出局 |
| Docker Hub + 公益镜像源 | ⚠️ 大层抖动 | 免费 | 现状,不稳 |
| **腾讯 TCR 个人版** | ✅✅ 同云内网 | 免费 | **本项目首选**(服务器同在广州) |
| 阿里云 ACR 个人版 | ✅ 国内快 | 免费 | 跨云备选 |
| 自建 registry(`registry:2`) | ✅ 可控 | 要运维+TLS+鉴权 | 重 |
| 本地 build 不用 registry | ✅✅ 不拉应用镜像 | 服务器吃资源 | 见 4.6 |

**业界**:大厂用私有 registry + **P2P 镜像分发**(Dragonfly / Uber Kraken)+ **镜像预热**,解决"千台机同时拉一个大镜像"的洪峰;中小团队直接用**云厂商托管 registry 的同区域实例**,靠"同 VPC/同区域内网"拿到稳定低延迟。本项目属后者:**服务器在腾讯云广州,就用腾讯 TCR(同城),这是最朴素也最有效的解。**

### 4.2 镜像大小层
**问题**:每次要传多少字节。手段:
- **多阶段构建**(已有):编译期工具不进运行镜像。
- **prod-only 依赖**:运行镜像只装 `dependencies`,去掉 `devDependencies`。本项目 server 的 runner 现在**把全量 node_modules(含 devDeps)整个拷进去**了(778MB),是主要肥肉。
  - ⚠️ 坑:`prisma` CLI 启动迁移要用,若它在 devDeps,prod-only 会删掉它导致迁移失败(agent-server 早期就栽过)。瘦身必须保证迁移仍可跑 → **需 build 验证**。
- **更小的基础镜像**:alpine(已用)/ distroless。
- **合并 RUN、清理缓存**:减少层数和层内垃圾(apt/apk 缓存)。
- **`.dockerignore`**:别把 `.git`、`node_modules`、测试数据塞进构建上下文。

**业界**:Google distroless、"运行镜像 < 100MB"是常态目标。一个 node 服务运行镜像合理值在 100–200MB,778MB 明显有优化空间。

### 4.3 缓存层 —— 让"依赖只装一次"
**这是用户最直觉的诉求**:"依赖能不能装一次后面跳过"。答案:**Docker 分层缓存本来就干这事**。

- **原理(扫盲)**:镜像是一层层叠的,`pull` 时**按层 digest 比对**,本地已有的层直接跳过。所以只要"依赖层"的 digest 不变,服务器就**不会重下依赖**,只下变化的代码层(几 MB)。
- **前提**:Dockerfile 要把依赖与代码分层。本项目 server 的 `deps` stage 是 `COPY package.json+lock → pnpm install` 单独一层,**写法是对的**——依赖层只在 `package.json/lock` 变化时才失效。
- **那为什么还会重下?** 两种:
  1. 依赖**真变了**(合并代码时动了 lock)。
  2. **构建缓存过期**:CI 用 `cache-to: type=gha`,GHA 缓存 **7 天不用就被清**;本项目部署稀疏(隔好几天一次),缓存被清 → `pnpm install` 重跑 → 产物 metadata 不同 → 层 digest 变 → **即使依赖没变也重下一次**。
- **解**:构建缓存从 GHA 改 **registry 缓存**(`cache-to: type=registry,ref=...:buildcache`),缓存存进镜像仓库本身,不受 GHA 7 天清理影响 → 依赖层跨多次部署长期稳定命中。

### 4.4 传输层
- **增量拉取**(已做):`up --pull missing` 只拉本地缺失的镜像,不对已缓存的基础镜像做无谓的 manifest 复检(复检也走 flaky 源,会卡)。
- **断点续传**:`docker pull` 失败后重试会**续传已下载的层**(不是从 0)。所以"重试循环"在弱网下极有效——每次重试都在推进。
- **并发/限速**:`max-concurrent-downloads` 等,弱网下有时调小并发反而更稳。

### 4.5 容错层 —— 弱网保命(P0,本次先做)
- **拉取重试 + 退避**:把应用镜像拉取从 `up` 里拆出来,套 `until ... ; do sleep; done` 重试循环。弱网抖动 → 续传重试 → 最终完成,而不是一次卡死整次崩。
- **超时调优**:`command_timeout: 30m`(已做),给慢拉留足时间。
- **拉取与启动分离**:先把镜像拉稳,再 `up`,便于对"拉取"单独重试、单独定位。
- **失败回滚**:镜像不可变 + tag 精确,失败可用旧 tag 一键回退。

### 4.6 备选范式:服务器本地 build(不依赖 registry)
你的另一台 paaawow 服务器就这么干:**Gitee(国内 git)拉源 + 服务器 `docker compose build`**,完全不拉应用镜像,稳跑 7 个月。
- 优:彻底甩开 registry 与镜像拉取。
- 劣:每次部署在服务器重 build(吃 CPU/RAM,本机还跑着 milvus,资源紧),且放弃"构建一次、到处运行"的不可变镜像优势,不如"CI 构建 + 同城 registry 拉取"高效。
- **结论**:作为兜底范式了解即可;本项目优先 4.1 的 TCR 路线。

## 5. 本项目落地方案(决断 + 顺序)

按"收益/成本"排序,分四步推进:

1. **P0 拉取重试 + 续传(本次立即做)**
   两个仓库的部署脚本:把应用镜像拉取拆成带重试的循环(失败续传,最多 8 次),再 `up`。零外部依赖,直接把"弱网一击毙命"变成"弱网慢点但能成"。
2. **P1 换腾讯 TCR(待凭据)**
   服务器同在广州,内网拉取又快又稳又免费。改动同"GHCR→Docker Hub"那次:CI 登录+push 到 TCR、compose 镜像路径改 TCR、部署脚本拉前登录。落地前先在服务器实测一把推/拉确认快稳。
3. **P2 镜像瘦身(需 build 验证)**
   server runner 改 prod-only 依赖(注意保留 `prisma` CLI 以跑迁移),778MB → 预期百 MB 级。配 web/gateway 一并检查。
4. **P3 registry 构建缓存**
   `cache-to/from` 从 GHA 改 registry(配合 TCR),依赖层长期稳定,跨多次稀疏部署都命中,不再无谓重下。

## 6. 弱网部署可靠性的"组合拳"

单点都不够,四件套叠加才稳:

```
小镜像(瘦身)        → 要传的字节本来就少
   +
同城源(TCR)         → 链路稳、延迟低、带宽足
   +
只传增量(--pull missing + 稳定分层 + registry 缓存)  → 大多数部署只传几 MB 代码层
   +
失败续传重试(容错)  → 万一抖动,续传重试兜底,不一击毙命
   = 弱网也能稳定成功
```

## 8. 实战踩坑记录(本次链路调试的沉淀)

1. **GHCR blob 被墙**:registry API 通 ≠ 能拉镜像;层数据走 CDN,国内被墙。判别:API 秒响应但 `docker pull` 零进度。
2. **公益镜像源只盖 Docker Hub**:`registry-mirrors` 只代理 `docker.io`,不代理 quay/gcr/ghcr(etcd 在 quay,得单独配 `quay.m.daocloud.io`)。
3. **备案 ≠ 能拉镜像**:备案解决入站 80/443 被封;拉镜像是出站,与备案无关。
4. **GHA 构建缓存 7 天过期**:稀疏部署导致依赖层缓存失效、被迫重下,误以为"每次都下依赖"。
5. **`set -e` + `[ -n "$X" ] && echo` 陷阱**:`.env` 生成里末尾的条件 echo,当变量空时 `[ -n ]` 退出 1,放在 `{ } > .env` 组里恰好不致命(因失败在 `&&` 豁免位),但写法脆弱,值得知道。
6. **`docker compose pull` 会复检所有镜像**:含已缓存的基础镜像,复检也走 flaky 源会卡 → 改 `up --pull missing` 只拉缺失。
7. **`--ignore-pull-failures` 治不了"卡住"**:它只忽略"失败",不解决"hang"(无错也无进度)。真正的解是续传重试 + 同城源。
