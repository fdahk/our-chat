# 音视频通话 — 断连恢复与同步设计方案

> 范围:our-chat web 端 WebRTC 1:1 语音/视频通话的「会话保持、断连恢复、状态同步」。
> 结论先行:**真正的无感恢复终点是 SFU(媒体服务器中转),但可以分阶段把会话层最佳实践先落到现有 P2P 架构上**。本文先讲透原理与术语,再做方案对比,最后给分阶段落地设计。

---

## 0. 术语表(先读)

> 后文大量用到这些词,先在这里一次性讲清楚,读者不必另开搜索。

| 术语 | 全称 / 含义 | 通俗解释 |
|---|---|---|
| **WebRTC** | Web Real-Time Communication | 浏览器内置的实时音视频/数据传输框架,免插件直接在两端之间传媒体流。 |
| **信令(Signaling)** | — | 通话双方"互换建立连接所需信息"的带外通道。WebRTC **不规定**信令怎么走,本项目用 Socket.io 传 `call:*` 事件。信令只负责"牵线",媒体本身不走信令。 |
| **`RTCPeerConnection`(PC)** | — | 浏览器里代表"一条点对点连接"的对象。它背后是原生媒体引擎(libwebrtc):持有 UDP 套接字、编解码器线程、加密密钥、ICE 状态。**它是本方案一切问题的核心**——刷新会把它连同原生资源一起销毁。 |
| **SDP** | Session Description Protocol | 一段文本,描述"我这端的媒体能力":编解码器、ICE 参数、加密指纹、轨道信息。双方交换 SDP 才知道怎么对接。 |
| **Offer / Answer(JSEP)** | JavaScript Session Establishment Protocol | 协商范式:发起方生成 **Offer**(SDP)发给对端,对端生成 **Answer**(SDP)回来,双方各自 `setLocalDescription`/`setRemoteDescription` 后媒体参数达成一致。 |
| **重协商(Renegotiation)** | — | 在已有 PC 上重新走一轮 Offer/Answer(如加轨道)。本方案 Phase 0 的"恢复"是**新建 PC 后从零做一轮 Offer/Answer**,不是在旧 PC 上重协商。 |
| **ICE** | Interactive Connectivity Establishment | 在两端之间"找出一条能通的网络路径"的算法。它收集候选地址、互相试探、选出可用的一对。 |
| **ICE candidate(候选)** | — | 一个"可能可达"的地址:`host`(本机局域网 IP)、`srflx`(经 STUN 发现的公网映射地址)、`relay`(经 TURN 中转地址)。双方把候选互发,逐对做连通性测试。 |
| **STUN** | Session Traversal Utilities for NAT | 一个轻量服务器,告诉你"你在公网上长什么样(IP:端口)",用于穿透 NAT。只帮发现地址,不转发媒体。 |
| **TURN** | Traversal Using Relays around NAT | 当两端无法直连时(对称 NAT/防火墙),**由 TURN 服务器中转媒体**。保底但耗服务器带宽。 |
| **NAT 穿透** | Network Address Translation 穿透 | 家用/企业网络里设备是内网 IP,要让两台内网设备直连,需借 STUN/TURN 把内外地址对应起来。 |
| **ICE restart** | — | 在 PC 还活着时,重新跑一遍 ICE(换新的 ufrag/pwd 凭据、重新收集候选、重测连通)。用于**网络切换/抖动**(WiFi↔4G)。注意:**刷新后 PC 已不存在,ICE restart 无从谈起**。 |
| **DTLS-SRTP** | Datagram TLS / Secure RTP | WebRTC 媒体强制加密:先在选定的 ICE 路径上做 DTLS 握手协商密钥,再用 SRTP 加密 RTP 媒体包。**密钥绑定在具体 PC 上,新 PC 必然重做握手**。 |
| **RTP / SRTP** | Real-time Transport Protocol / Secure RTP | 实际承载音视频数据包的协议(SRTP=加密版)。 |
| **SSRC** | Synchronization Source | RTP 流的唯一标识号,用于区分/同步不同媒体流。 |
| **MediaStream / Track** | — | `getUserMedia()` 拿到的本地音视频流;一条流含若干 track(音频轨/视频轨)。 |
| **P2P / Mesh** | Peer-to-Peer / 网状 | 媒体在参与者之间**直连**。1:1 就一条直连;N 人则每人对其余 N−1 人各建一条(网状)。 |
| **SFU** | Selective Forwarding Unit(选择性转发单元) | 一台媒体服务器:每个客户端只**上行一份**流给它,它**不解码、原样选择性转发**给其他订阅者。每端只需 1 条连接。 |
| **MCU** | Multipoint Control Unit(多点控制单元) | 也是媒体服务器,但它**解码所有流、混合成一路、再编码**下发。客户端只收 1 路,但服务器 CPU 极重。 |
| **simulcast / SVC** | 联播 / 可伸缩视频编码 | 同一摄像头同时编出多档清晰度,SFU 按订阅方网络选合适档位下发。SFU 的高级能力。 |
| **心跳(Heartbeat)** | — | 定时小包探活,用于及时发现"连接已死"(僵尸连接)。 |
| **seq_ack(序列号确认)** | — | Discord 信令里每条消息带序列号,客户端回报"我收到的最后序号",断连重连后服务器据此**重放漏掉的消息**。 |
| **thundering herd(惊群)** | — | 大量客户端在同一时刻一起重连,把服务器压垮。 |
| **jittered backoff(抖动退避)** | — | 重试间隔指数增长 + 加随机抖动,把重连请求在时间上摊开,缓解惊群。 |
| **redux-persist** | — | 把 Redux 状态序列化进存储引擎(默认 `localStorage`)的库。本项目 `web/src/store/rootStore.ts` 用它持久化 `user`/`friendReq`。 |
| **sessionStorage / localStorage** | Web Storage | 浏览器本地 KV 存储。**sessionStorage** 按标签页隔离、关页即清、刷新保留;**localStorage** 同源跨标签页共享、长期保留。 |

## 1. 问题定义

### 1.1 现象
通话过程中**刷新页面 → 通话直接断**,且对端不是干净挂断,而是卡几秒后报"网络连接失败"。

### 1.2 根因(已核对现状代码)
1. **PeerConnection 随 JS 运行时销毁**(核心)。详见 §2.2 的引擎层原理。
2. **通话状态没持久化**:`web/src/store/rootStore.ts` 的 redux-persist `whitelist` 只有 `['user','friendReq']`,`call` slice 刷新即清零 → 重载后完全不知道刚才在通话。
3. **服务端是纯无状态转发**:`server/src/utils/socket.ts` 的 `call:*` 仅按 `callId`(格式 `call_<caller>_<callee>_<ts>`)解析双方 userId 做 relay,**不记录任何通话会话状态**。重连方回来后,没有权威源能告诉它"这通还活着"。
4. **刷新方不发 `call:end`**:`disconnect` 只摘 presence(在线状态);对端只能等 ICE 超时 → `failed` → `handleConnectionFailed()` 立即 `cleanup()`,**无重连宽限**。

### 1.3 必须区分的三类中断(决定恢复策略)
| 类型 | JS 运行时 | PeerConnection | 正确恢复手段 |
|---|---|---|---|
| **A. 网络抖动**(切 WiFi/4G、短丢包) | 存活 | 存活 | **ICE restart**(近无感,见 §2.3) |
| **B. 页面刷新 / 误关重开** | 销毁重建 | 没了 | **重新入会 + 新建 PC 走 Offer/Answer**(有几秒中断) |
| **C. 进程/设备崩溃** | 销毁 | 没了 | 同 B,只是离线更久 |

> 我们报告的痛点是 **B**。**关键认知:B 在 P2P 下本质是"重建"而非"续接",必然有几秒中断**;要近无感需 SFU(§4.1、§6 Phase 3)。

## 2. 关键技术原理(讲透机制)

### 2.1 一通 WebRTC 通话是怎么建立的(理解这个才懂"恢复"在重建什么)
一次成功通话要顺序打通**四层**,缺一不可:

1. **信令交换**:发起方通过 Socket.io 把 Offer 发给对端,对端回 Answer(本项目即 `call:start`/`call:accept`)。
2. **SDP 协商(Offer/Answer)**:双方交换媒体能力描述,确定用什么编解码器、谁是发送方等。代码路径:`createOffer → setLocalDescription →(信令发送)→ 对端 setRemoteDescription → createAnswer → setLocalDescription →(信令回送)→ 发起方 setRemoteDescription`。
3. **ICE 连通性建立**:`setLocalDescription` 后浏览器开始收集 ICE 候选,通过信令 `call:ice` 互发(trickle ICE,边收集边发);双方对候选两两做 STUN 连通性测试,选出一对可用路径。
4. **DTLS-SRTP 加密握手**:在选定路径上做 DTLS 握手派生密钥,之后所有 RTP 媒体包用 SRTP 加密传输。

**"恢复一通话"= 把上面四层重新打通一遍。** 在 P2P 下,刷新方什么都没了,只能从第 1 层重来(新建 PC、重取媒体、重发 Offer)——这就是为什么 P2P 刷新恢复 = 完整重建。

### 2.2 为什么刷新必然销毁连接(引擎层原理)
`RTCPeerConnection` 只是 JS 里的一个"句柄",它背后挂着**原生媒体引擎(libwebrtc)**持有的一堆操作系统级资源:
- 已绑定的 **UDP 套接字**(ICE 选定的本地端口);
- **编码器/解码器**线程(常含硬件加速句柄);
- **DTLS 会话密钥**(本次连接独有,不可导出复用);
- **ICE agent** 的候选与连通状态机。

页面刷新 = 浏览器触发导航卸载,**销毁整个 JS realm,并连带回收这些原生对象**。WebRTC 规范**没有提供**任何"序列化 PC / 跨页面交接连接"的 API。因此刷新后这条连接**物理上不存在**,无法"接回",只能新建。原生 App(如微信)进程不随某个页面销毁,所以没有这个问题——这是 web 与原生的本质差异。

### 2.3 ICE restart 的原理与适用边界
当**网络变了但 PC 还活着**(A 类),不需要整通重建,用 ICE restart:
- 机制:发起方 `createOffer({ iceRestart: true })`(或 `pc.restartIce()`),生成**新的 ICE ufrag/pwd 凭据**,重新收集候选、重做连通性测试,在两端建立**新的 ICE 路径**。
- 关键特性:**媒体在旧路径上继续流动,直到新路径就绪才切换**——所以用户几乎无感(可能短暂卡顿)。
- **边界**:它要求 PC 仍然存在。**刷新后 PC 已销毁,ICE restart 完全不适用**——这是很多人误以为"ICE restart 能救刷新"的认知误区。

### 2.4 SFU 为什么让恢复变简单(转发原理 + 服务端状态)
- **拓扑**:每个客户端只与 SFU 建**一条**连接,上行一份流;SFU **不解码**,按订阅关系把流原样选择性转发给其他人。
- **恢复优势**:某客户端断了/换网,**只有它↔SFU 这一条腿需要重建**,其余参与者↔SFU 的连接**纹丝不动**——对端无感。
- **服务端可重建状态**:SFU 持有"谁在发什么流、谁订阅了谁"(producer/consumer)的权威状态。客户端重连时,服务器能**按客户端预期重建自己这侧的状态**,表现得像一次 ICE restart,实现亚秒恢复(LiveKit 的 live migration 即此)。
- **代价**:要部署、运维、扛带宽的媒体服务器集群,且 1:1 也得绕一跳服务器(略增延迟)。

## 3. 业界最佳实践(调研)

### Discord(语音)
- 媒体**不走 P2P**,走 Discord 的 voice server(SFU 式中转)。
- 信令是独立的 **voice gateway(WebSocket)**,有完整 **RESUME 协议**:`Opcode 7 Resume` 带 `session_id`;自 v8 起所有消息带序列号,客户端心跳/Resume 里回 `seq_ack`,网关把断连期间漏掉的消息**缓冲重放**;缓冲过期 → 关闭会话走全新 `Identify`。
- 心跳(opcode 3/6)探活,僵尸连接立即断开重连。
- 启示:**服务端权威会话 + 带序列号的 resume + 缓冲重放**,媒体由服务器锚定。

### LiveKit(开源 SFU,生产级参考)
- **双模自动重连**:
  - **Quick reconnect(resume)**:信令重连 + **ICE restart**,保留 room/track 订阅发布状态,通常近无感。
  - **Full reconnect(回退)**:resume 失败则从头重建,**对端会看到 `ParticipantDisconnected`**,触发 `Reconnecting` 事件供 UI 提示。
- **连接质量 `Lost` 早警**:在 `Reconnecting`/`Disconnected` 之前就感知到客户端↔SFU 这条腿掉了。
- **服务端 grace + live migration**:服务端短暂保留参与者会话;故障/迁移时新服务器**按客户端预期重建服务端状态** → 亚秒恢复。
- 启示:**分级恢复(先 resume 后 full)+ 服务端状态可重建 + 早警**。

### 通用 WebRTC「刷新」恢复(社区共识)
- 把刷新当**重新入会(session re-initiation)**,不复活旧连接。
- **持久化 room/user id** 到 storage/URL,重载自动 rejoin。
- **清干净 zombie**:旧连接/旧 socket 要真销毁,否则脏状态泄漏。
- **唯一身份 + 给服务端清理留时间**,避免与残留旧会话在服务端撞车。
- **ICE restart 只救网络抖动(A 类)**,救不了刷新(B 类)。

### 提炼:五条最佳实践支柱
1. **媒体经服务器(SFU)锚定** —— 恢复只重建受影响那条腿,对端无感。
2. **服务端权威会话** —— session/room id 为锚 + grace 宽限保留。
3. **分级恢复** —— 抖动用 ICE restart(快);刷新/重建用 full rejoin。
4. **Resume 协议** —— 信令重连带 session_id + 序列号,漏的事件重放;过期则全新 identify。
5. **韧性细节** —— 心跳/连接质量早警、抖动退避防惊群、清 zombie、恢复后状态重新同步。

## 4. 方案对比(tradeoff)

### 4.1 媒体拓扑:P2P / Mesh vs SFU vs MCU
| 维度 | P2P / Mesh | SFU | MCU |
|---|---|---|---|
| 客户端连接数(N 人) | N−1(失败点多) | 1 | 1 |
| 客户端上行带宽 | 高(给每个人发一份) | 低(只发一份) | 低 |
| 服务器成本 | 无(无媒体服务器) | 中(转发,不解码) | 高(解码+混流+编码) |
| 端到端延迟 | 最低(直连) | 低(绕一跳) | 较高(转码) |
| 1:1 适配度 | **天然契合** | 可用但多一跳 | 浪费 |
| 断连恢复 | 每条腿各自重建,无中心权威 | **单腿重建,服务端可重建状态** | 同 SFU |
| 多人扩展 | 差(N² 连接) | **好** | 好(客户端轻) |
| 端到端加密(E2EE) | 易(直连) | 较难(服务器是中间人) | 不可能(要解码) |
| 部署/运维复杂度 | 最低 | 中高 | 高 |

> **结论**:我们当前是 1:1,P2P 在延迟/成本上最优,但恢复能力天然弱。SFU 是"无感恢复 + 多人"的正解,但属基建级投入。**短期不切 SFU,先在 P2P 上把会话层做扎实;SFU 留作终态(§6 Phase 3)。**

### 4.2 恢复机制:ICE restart vs Full rejoin
| 维度 | ICE restart | Full rejoin(新建 PC) |
|---|---|---|
| 适用中断 | A 类(网络抖动,PC 存活) | B/C 类(刷新/崩溃,PC 已亡) |
| 是否复用 PC | 是 | 否(新建) |
| 是否重取媒体 | 否 | 是(`getUserMedia`) |
| 用户感知 | 近无感(媒体边切边传) | 几秒中断 |
| 信令开销 | 一轮 Offer/Answer(iceRestart) | 完整一轮 + 重新入会 |
| 本方案归属 | Phase 1 | Phase 0 |

### 4.3 客户端持久化:存哪儿
| 选项 | 生命周期 | 标签页作用域 | 适配通话恢复 |
|---|---|---|---|
| **sessionStorage** | 关标签页即清,刷新保留 | 单标签页隔离 | **最贴合**:刷新能恢复,关页不复活,天然单标签页语义 |
| localStorage | 长期(手动清) | 跨标签页共享 | 不合适:关页后还在,易"复活"早结束的通话 |
| redux-persist(→localStorage) | 同 localStorage | 跨标签页 | 不合适:把易变的 call 状态长期落盘,生命周期不对 |
| URL 参数 | 随页面 | — | 不合适:通话不是可分享的路由状态 |

> 选 **sessionStorage**:刷新保留(能恢复)、关页即清(不复活)、按标签页隔离(天然规避多标签页串话)。再加 `savedAt` + TTL(~2min)兜底防"过几分钟打开旧标签页复活老通话"。

### 4.4 会话权威:客户端驱动 rejoin vs 服务端权威会话
| 维度 | 客户端驱动(Phase 0) | 服务端权威会话(Phase 2) |
|---|---|---|
| 服务端是否记状态 | 否(沿用无状态 relay) | 是(Redis 通话注册表) |
| 实现成本 | 低(复用现有 relay + Offer/Answer) | 中高 |
| "这通还在吗"判定 | 靠对端宽限窗超时 | 服务端权威判定 |
| 多端/多标签裁决 | 做不到 | 能做 |
| 可观测/TTL 清理 | 弱 | 强 |
| 与现有架构契合 | **高(信令本就无状态)** | 引入新状态 |

> 现有信令是**有意做成无状态 relay**,Phase 0 用客户端驱动即可真正交付刷新恢复;服务端权威会话作为 Phase 2 升级。

## 6. 目标方案(分阶段落地)
> 原则:**先用现有 P2P + 现有 relay 把"会话层最佳实践"落地,交付真正的刷新恢复;媒体中转(SFU)作为终态 RFC 单列**。每阶段独立可验收。

### Phase 0 —— P2P 之上的刷新恢复 + 状态同步【本期实施】
解决 B 类中断。对应支柱 2(简化)/3(rejoin 半)/5。
- **持久化**:进 `connected` 时把精简上下文写 sessionStorage(schema 见 §7.1)。
- **重载恢复**:`useCall` 挂载时读取,若有未过期活跃通话 → 进 `reconnecting` 态 → `reset()` 清旧 PC → `getUserMedia` 重取流 → `createOffer` → 通过 **`call:rejoin`**(走现有 relay)发起 → 对端按一轮 Offer/Answer 重建 → 恢复 `connected` 并同步 mute/时长(以原 startTime 锚定)/callType。
- **对端宽限窗**:ICE `disconnected`/`failed` 不立即 cleanup,进 `reconnecting` 宽限窗(~12s);窗内收到 `call:rejoin` → 重建恢复;**超时**才 `endCall`。
- **关页 vs 刷新区分**:`pagehide`/`beforeunload` 时,若非"即将恢复的刷新",发 `call:end` 干净挂断;是刷新则不发,靠对端宽限窗等 rejoin。
- **幂等/清 zombie**:复用同一 `callId`;重建前 `reset()` 清旧 PC;`call:rejoin` 重复到达按 callId + 状态去重(复用现有 `processedEvents`)。

### Phase 1 —— 网络抖动 ICE restart【本期可选,紧随】
解决 A 类。`pc.restartIce()` + 一轮 Offer/Answer,近无感;`getStats()` 扩展现有 2s 轮询做连接质量早警。

### Phase 2 —— 服务端权威会话【后续,贴近 Discord/LiveKit 信令层】
Redis 通话注册表(`callId → {participants, status, startTime, TTL}`),`call:start/accept/end` 维护;`call:rejoin` 先问服务端"还在吗";引入序列号 + 漏事件重放;grace 由服务端裁决。价值:多端裁决、可观测、TTL 兜底。

### Phase 3 —— 终态:引入 SFU【独立大工程,RFC 单列】
媒体改走 mediasoup / LiveKit:对端无感、亚秒恢复 + 突破 1:1 多人。成本:部署运维、带宽、媒体服务器高可用。**不在本期**,需独立立项。

## 7. Phase 0 详细设计(本期实施细节)

### 7.1 持久化数据(sessionStorage)
```ts
// key: "oc.call.active"
interface PersistedCall {
  callId: string;
  peer: { id: number; username: string; nickname: string; avatar: string };
  callType: 'voice' | 'video';
  role: 'caller' | 'callee';   // 决定重建时谁先发 Offer:由重连方发
  startTime: number;            // 用于恢复后续算通话时长
  isMuted: boolean;
  savedAt: number;              // TTL 基准,超 ~2min 视为过期不恢复
}
```

### 7.2 信令:新增 `call:rejoin`
- **契约**:既已统一 proto,**`call:rejoin` 加进 `proto/ourchat/call/v1/call.proto`** 生成类型,不裸写 socket 事件。
- 载荷:`{ callId, from, to, offer }`(offer 为重连方新建 PC 的 SDP)。
- 服务端:按现有 `call:*` relay 模式,从 `callId` 解析对端 userId 转发(无状态)。

### 7.3 恢复时序(文字时序图)
```
刷新方                         信令(relay)            对端(通话中)
  │ 重载, 读 sessionStorage                              │
  │ 命中活跃通话 → status=reconnecting                   │
  │ reset()+getUserMedia()+createOffer()                 │
  │ ──────── call:rejoin{callId,offer} ──────────────▶  │ (宽限窗内)
  │                                                      │ setRemoteDescription(offer)
  │                                                      │ createAnswer()+setLocalDescription
  │ ◀──────── call:accept{answer} ───────────────────── │
  │ setRemoteDescription(answer)                         │
  │ ◀──────── call:ice ⇄ call:ice ───────────────────▶  │ (ICE/DTLS 重新打通)
  │ connectionState=connected → status=connected         │ status=connected
  │ 同步 isMuted / 以 startTime 续算时长 / callType        │
```

### 7.4 状态机变更
`callStore.status` 增加 `'reconnecting'`(介于 `connected` 与 `ended` 之间):UI 显示"重连中…",不渲染"已结束"。

### 7.5 关键边界(必须覆盖)
- **宽限窗 timer**:对端进入 reconnecting 时启动 ~12s 定时器,恢复则清除,超时则 `endCall`+提示。
- **pagehide 区分**:用 sessionStorage 标记区分"刷新(将恢复)"与"真关页(发 call:end)"。
- **TTL**:`savedAt` 超 2min 不恢复并清除,防复活僵尸通话。
- **重复 rejoin 幂等**:同 callId 重复事件去重,避免重复重建。
- **双方同时刷新**:尽力而为,任一方宽限窗超时则干净结束。

### 7.6 验收标准(可测试)
- 通话中刷新**任一端**,≤ ~5s 内自动恢复音/视频,且 **mute 状态、通话时长、语音/视频类型保持**。
- 真正**关闭标签页**:对端 ~1s 内收到挂断,不卡死。
- 宽限窗内(~12s)对端**不误挂**;超时干净结束并提示。

### 7.7 邀请期恢复(主叫呼叫中 / 被叫来电中刷新)
不止「通话中」能恢复,**邀请阶段刷新也能恢复**。关键认知:邀请期刷新后,刷新前创建的 offer 与已通过 trickle 发出的 ICE 候选都已失效(对端持有的旧候选指向已销毁的连接),**不能复用旧 offer 直接接听**,必须重新协商。

- **持久化阶段**:`call` slice 在 `calling`(主叫)/`ringing`(被叫)阶段即写 sessionStorage(`status` 字段区分),不再只在 connected 后才存。
- **主叫呼叫中刷新**:重载 → 复原「正在呼叫」界面 → `rejoinCall` 重发**新 offer**;被叫仍在振铃 → 收到 `call:rejoin` 仅**替换 pendingOffer**(不自动协商),用户照常接听即用新 offer。
- **被叫来电中刷新**:重载 → 复原「来电」界面(`pendingOffer` 置空)→ **等用户接听**;接听时因无有效 offer,改由**被叫发新 offer**(`call:rejoin`),主叫(仍在呼叫中)走重建分支回 answer,双方重新收集候选。
- **服务端**:会话记 `resumeStatus`,邀请期属主(主叫)刷新经 grace 后 `rejoin` 恢复回 `ringing`(而非 connected),保证被叫后续接听仍能正确绑定属主设备;grace 超时则广播 `call:end`,被叫来电消失。

## 7bis. 已实现:多标签页/多设备并发裁决(服务端权威会话)
> 本期把原 Phase 2 的「服务端权威会话」核心一并落地,以支持多标签页/多设备的并发裁决。媒体仍 P2P(SFU 仍属 Phase 3)。

### 设计原理
- **统一的「设备」抽象**:每标签页一个 **per-tab deviceId**(`web/src/utils/device.ts`,sessionStorage UUID),握手 `auth` 携带给服务端。同标签页刷新后稳定(重连可被识别为同一设备),跨标签页/窗口/浏览器各异 → 多标签页与多设备用同一套「设备」逻辑裁决,无需区分。
- **服务端权威会话(Redis,`server/src/services/callSession.ts`)**:
  - `call:session:<callId>` → `{callerId, calleeId, callType, status, startTime, callerDevice, calleeDevice, graceEpoch}`(JSON,带兜底 TTL)。
  - `call:user:<userId>` → `callId`(**忙线索引**,O(1) 判断用户是否在通话中)。
  - 每条 socket 连接加入 `device:<deviceId>` 房间,供「属主路由」精确投递。

### 裁决规则
| 场景 | 规则 | 实现 |
|---|---|---|
| **忙线拒接** | 被叫已在另一通通话 → 回主叫 `call:busy`,不振铃 | `call:start` 前 `tryCreateSession` 查忙线索引 |
| **他处接听** | 某设备接听 → 通知被叫其余设备/标签页停止振铃 | `call:accept` 后向 `room(被叫)` 除接听 socket 外发 `call:handled{accepted}` |
| **他处拒接** | 某设备拒接 → 同上停响 + 通知主叫 | `call:reject` 发 `call:handled{rejected}` |
| **属主路由** | `accept`/`rejoin` 的信令只投给「拥有这通话的那台设备」,不广播全部标签页 | `io.to(deviceRoom(属主))` |
| **grace 宽限** | 属主设备掉线 → 会话转 `reconnecting` + 通知对端 `call:peer-reconnecting`;`GRACE_MS`(12s)内等 `call:rejoin`,超时广播 `call:end` | `disconnect` → `markReconnecting` + `setTimeout`;`graceEpoch` 防竞态、跨副本生效(别处副本 rejoin 自增 epoch 使在途定时器失效) |

### 新增信令(已入 `call.proto` 生成类型)
`CallRejoin`(重新入会带新 offer)、`CallBusy`(忙线)、`CallHandled`(他处已处理)、`CallPeerReconnecting`(对端重连中)。

### 关键路径时序(文字)
- **多设备接听**:主叫 `call:start` → 被叫 A/B 两端同时振铃 → A 接听 `call:accept` → 服务端绑定 A 为属主、给 B 发 `call:handled` → B 静默收起。
- **忙线**:C 呼叫正在通话中的 A → 服务端查忙线索引命中 → 回 C `call:busy`,A 无感。
- **掉线 grace**:A 刷新 → A 的 socket `disconnect` → 服务端 `markReconnecting` + 通知 B `call:peer-reconnecting`(B 显示「重连中…」)→ A 重载后 `call:rejoin` → 服务端按 deviceId 投给 B 的属主设备 → 双方重协商 → connected;A 若 12s 未回 → 服务端广播 `call:end`。

## 8. 不做(YAGNI / 本期边界)
- SFU 媒体中转、多人通话(Phase 3)。
- Discord 式信令序列号 + 漏事件重放(Phase 2 剩余项;本期会话权威已够裁决)。
- 网络抖动 ICE restart(Phase 1)。
- 跨设备通话迁移(进行中的通话从手机转到电脑接听等)。

## 9. 风险与权衡(诚实说明)
- **P2P 刷新恢复有几秒中断,非无感**——架构决定,实现取巧消不掉,无感需 SFU(Phase 3)。
- **双方同时刷新 / 对端也在宽限窗内刷新** → 尽力而为,任一方超时则干净结束。
- **sessionStorage TTL** 防止"几分钟后打开旧标签页复活早已结束的通话"。
