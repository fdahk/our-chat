# 工程组织与部署编排

本文描述 our-chat 作为多语言、多服务单仓(monorepo)的工程组织方式:仓库布局、部署
基础设施的收口、环境变量分层、边缘入口、容器编排拓扑与镜像构建策略。

## 1. 仓库布局

```
our-chat/
├── server/        Node + Express 业务/信令后端(含 OAuth IdP)
├── gateway/       Go 长连接网关(WebSocket 接入层)
├── web/           React + Vite Web 端
├── mobile-swift/  iOS 客户端
├── docker/        部署基础设施(compose + nginx + 部署 env)
└── docs/          技术方案与设计记录
```

组织原则:**每个服务自治**。各服务目录自带依赖清单、`Dockerfile`、测试、分层
`CLAUDE.md`,可独立构建、独立测试、独立运行。服务间不共享源码,只通过网络协议
(HTTP / WebSocket / Redis)与共享密钥契约耦合。

跨服务的"部署面"与"业务面"分离:业务代码留在各服务目录,**所有部署相关的东西
(编排、反向代理、部署环境变量)收口到顶层 `docker/`**。

## 2. 部署基础设施收口:`docker/`

```
docker/
├── docker-compose.yml          生产全栈编排
├── docker-compose.dev.yml      本地开发(仅中间件)
├── .env.example                部署环境变量模板(复制为 docker/.env)
└── nginx/
    ├── nginx.conf              http 全局(worker / 上传体积 / 通用)
    └── conf.d/default.conf     server 块(静态托管 + 同源反代)
```

**设计目的**:部署策略(路由、端口、限额、凭证)与业务镜像解耦。改一条代理规则、
调一个上传上限、换一组凭证,都不需要重新构建任何服务镜像——只改 `docker/` 下的文件。

**运行约定**:在 `docker/` 目录内执行 compose,使其自动加载同目录 `.env`。

```bash
cd docker && cp .env.example .env     # 首次:填好 JWT_SECRET 等
docker compose up -d --build          # 生产全栈
docker compose -f docker-compose.dev.yml up -d   # 本地仅中间件
```

> compose 默认从「执行所在目录」加载 `.env`,因此约定 `cd docker` 后再执行,
> `${VAR}` 替换与 `env_file` 都指向 `docker/.env` 这一份。

## 3. 环境变量分层

项目有三处 env,**不是重复,而是三个不同的消费者、三种不同的加载机制、三种不同的
信任边界**。

| 文件 | 消费者 | 加载机制 | 生效时机 | 主机名/值特征 |
|---|---|---|---|---|
| `docker/.env` | docker compose | `env_file` 整体注入 + `${VAR}` 替换 | 容器编排 | 容器内主机名(`@postgres`、`redis://redis`) |
| `server/.env` | Node `dotenv/config` | 进程启动读 CWD 下 `.env` | 宿主机直跑 | `localhost` 主机名 |
| `web/.env(.local)` | Vite | 构建期注入,仅 `VITE_` 前缀 | 前端构建 | 只能放**公开值**(进浏览器 bundle) |

### 3.1 为什么必须分开

1. **加载机制不同**:compose 是文件注入 + 文本替换;Node 是运行时 `process.env`;
   Vite 是把变量**编译进客户端 bundle**。三者无法共用一个文件。
2. **信任边界不同**:`web/.env` 的值最终下发到浏览器,任何人可见,**只能放公开配置**
   (如 `VITE_AGENT_API_BASE`),严禁放密钥;`docker/.env` / `server/.env` 才放
   `JWT_SECRET`、DB 密码、对象存储凭证等机密。
3. **运行模式不同**:容器化部署走 `docker/.env`;宿主机直跑(`npm run dev` /
   `go run`)各服务读自己目录的 `.env`,因为 dotenv / Vite 只在各自 CWD 查找。

### 3.2 单一部署源(容器模式)

`docker/.env` 是容器模式下的**单一配置源**。compose 用 YAML 锚点把它整体注入所有
服务容器,避免同一变量(如 `JWT_SECRET`、`GATEWAY_INTERNAL_TOKEN`)在多个服务块
里各写一份导致漂移:

```yaml
x-env-file: &env_file
  - .env
services:
  server:
    env_file: *env_file       # 整份 .env 注入,进程直接 process.env 读
    environment:              # 只留容器结构性覆盖
      NODE_ENV: production
      PORT: 3007
  gateway:
    env_file: *env_file
    environment:
      GATEWAY_ADDR: ':8090'
      UPSTREAM_BASE_URL: http://server:3007
```

容器结构性常量(`NODE_ENV`、监听地址、上游地址)留在 `environment:`;配置与机密
(连接串、密钥、OAuth 参数、对象存储)全部走 `docker/.env`。

### 3.3 跨运行模式的同名变量

`JWT_SECRET` 在 server 与 gateway 之间必须一致(gateway 用它验签 server 签发的登录
token),`GATEWAY_INTERNAL_TOKEN` 在 server 与 gateway 之间必须一致(保护内部上行
端点)。

- **容器模式**:两个容器都 `env_file` 自同一份 `docker/.env` → 天然一致,无漂移。
- **宿主机模式**:server 读 `server/.env`,gateway 读自身 env;一致性靠开发者维护。

## 4. 边缘入口:nginx 同源反向代理

生产环境只有一个对外服务——`web` 容器里的 nginx。它同时是**静态资源服务器**和
**同源反向代理(单一 ingress)**;server(3007)、gateway(8090)只在内网,不对外
暴露端口。

```
                浏览器 / iOS
                     │ :8080(唯一对外)
                     ▼
            ┌──────────────────┐
            │  nginx(web 容器) │
            └───┬──────┬─────┬─┘
   /api /user      │ /socket.io │ /ws
   /oauth          ▼            ▼
   /api/upload  server:3007  gateway:8090
        │           │
        └───────────┘
   /(其余)→ 本地静态 SPA(try_files → index.html)
```

| 路径 | 转发 | 说明 |
|---|---|---|
| `/api/` `/oauth/` `/user/` | server:3007 | 业务 API(带 HttpOnly cookie) |
| `/socket.io/` | server:3007 | Socket.io 实时(WS 升级) |
| `/ws` | gateway:8090 | Go 网关原生 WebSocket |
| `/`(其余) | 本地静态 | SPA,回退 `index.html` |

**关键决策**:

- **同源消除跨域 cookie 问题**:前端与 API 同挂一个 origin,浏览器视为同源,
  HttpOnly cookie 自动携带,无 CORS 预检与第三方 cookie 限制。
- **单一入口收敛攻击面**:后端服务藏于 docker 网络,nginx 是唯一 ingress。
- **配置外置**:nginx 配置不烤进镜像,由 compose 从 `docker/nginx/` 以 `:ro` 挂载,
  改路由 / 开 SSL 无需重建前端镜像。
- **上传体积**:http 层 `client_max_body_size 100m`,与 server 端 multer 上限对齐。
  nginx 默认 1MB 会把头像 / 单文件 / 分片(均 >1MB)直接 413。
- **WebSocket**:`/socket.io/` 与 `/ws` 均设 `Upgrade`/`Connection` 头与
  `proxy_read_timeout 3600s`,支撑长连接。

## 5. 容器编排拓扑

两套 compose,对应两种使用场景。

### 5.1 开发:`docker-compose.dev.yml`(只起中间件)

只起 **postgres + redis + minio**(+ 一次性 `minio-setup` 建桶设公读),业务
(server / gateway / web)跑在宿主机热重载。中间件端口直接映射到宿主机,便于本地
调试与连库。

```
postgres :5432   redis :6379   minio :9000(API)/ :9001(控制台)
宿主机:server :3007 ┄ gateway :8090 ┄ web(Vite):5173
```

### 5.2 生产:`docker-compose.yml`(全栈)

中间件 + server + gateway + web(nginx),同处一个 bridge 网络。**只有 web 暴露
端口**(`${WEB_PORT:-8080}:80`),其余服务仅内网可达。

- `depends_on` + `condition: service_healthy`:中间件 healthy 后再起业务,避免启动
  竞态。
- 每个服务自带 healthcheck:server `/health`、gateway `/healthz`、postgres
  `pg_isready`、redis `redis-cli ping`。
- 镜像可覆盖:`${SERVER_IMAGE:-our-chat-server:local}` 等,便于 CI 推送的镜像与本地
  构建切换。

## 6. 镜像构建策略

三个服务的 `Dockerfile` 共性:**多阶段构建 + 依赖层缓存(先拷依赖清单再拷源码)+
运行阶段非 root + 自带 healthcheck**。

### server(Node,多阶段)

`deps`(装依赖)→ `dev`(tsx watch,供 dev override)→ `build`(编译 TS + 生成
Prisma Client)→ `runner`(生产运行时)。`runner` 只带 `dist` + `node_modules` +
`prisma` + 生成的 client,非 root 用户运行,启动时先 apply migration 再 listen。

### gateway(Go,静态二进制)

`CGO_ENABLED=0` 静态编译 + `-ldflags="-s -w"` 裁剪 → 拷进 `alpine:3.20` 精简镜像,
非 root 运行。无运行时依赖,镜像极小。

### web(构建 + nginx)

`node` builder 构建出 `dist/` → 拷进 `nginx:alpine` 托管。**nginx 配置不烤进镜像**,
由 compose 挂载(见 §2、§4),镜像只含静态产物。

## 7. server 启动编排

server 进程内部也有明确的启动序,保证「对外可用」前置依赖已就绪:

```
applyPendingMigrations()   先把 DB 迁移补齐
  → seedDefaultClient()    seed 默认 OAuth client
  → loadKeyStore()         加载 RSA 私钥(OAuth 签发)
  → mountOAuth()           在 listen 前挂 /.well-known/* 与 /oauth/*
  → app.listen()           开始接收 HTTP
  → initSocket(server)     挂载 Socket.io 实时层
```

任一前置步骤失败立即 fail-fast 退出,避免出现「进程起来了但 IdP 未就绪」的窗口。

## 8. 小结:配置与代码解耦

工程组织的一条贯穿主线是**让"改部署"不触碰"改代码/改镜像"**:

- nginx 路由策略 → 挂载文件,不进镜像;
- 部署配置与机密 → `docker/.env` 单一源,`env_file` 注入;
- 镜像来源 → compose 变量可覆盖;
- 运行模式 → 容器(`docker/.env`)与宿主机(各服务 `.env`)各走各的加载机制。

结果:部署面集中在 `docker/` 一处,业务面留在各服务目录,二者独立演进。
