# 文件存储改造:本地磁盘 → S3 兼容对象存储

## 1. 背景与问题

server 当前的文件上传用 multer `diskStorage` 把文件直接写到进程所在机器的本地目录(`uploads/`),并在业务代码里直接操作文件系统(`fs.writeFile` / `fs.readdirSync` / `fs.createWriteStream`)。这套实现有一系列生产硬伤:

- **不可水平扩展**:多副本部署时,各副本的本地盘不共享。A 副本上传的文件,B 副本读不到。
- **单点与易失**:容器/机器宕机或重建,本地盘文件丢失;Docker 部署还踩了 `/uploads` 路径解析(cwd 依赖)和非 root 写权限的坑。
- **秒传是 O(n) 扫盘**:`checkFileExists` 遍历整个 uploads 目录逐个算 MD5,目录一大就拖垮。
- **分片合并占用大量本地 IO**:逐分片读入内存再顺序写出,合并大文件时内存/磁盘压力大。

这些不是"配置问题",而是**架构问题**:文件存储不应该耦合在业务进程的本地文件系统上。

## 2. 目标与设计原则

**目标**:把存储后端从"本地磁盘"换成"S3 兼容对象存储",dev 用 MinIO(本地容器)、prod 用腾讯云 COS,业务代码一份、只换 endpoint。

**原则**:
1. **保留上传协议层的自研逻辑**(分片切分、秒传查重、断点续传、并发控制)——这是有价值、有深度的部分,不砍。
2. **替换存储层**:业务代码不再碰文件系统,改为调用统一的 `StorageService`(S3 API 封装)。
3. **dev / prod 同一套代码**:用 S3 兼容协议,MinIO 与 COS 都用同一份 SDK 调用,仅通过 env 切换 endpoint / 凭证。
4. **前端契约尽量不变**:上传接口的请求/响应结构保持,前端 `FileUploader` 组件零改动或极小改动。

## 3. 业界对比:为什么这是大厂做法

大厂(字节/腾讯/阿里)业务后端的文件存储**几乎都走对象存储**:字节 TOS、腾讯 COS、阿里 OSS;不上公有云的用自建对象存储(Ceph / MinIO / 自研分布式存储)。关键分层:

| 层 | 谁负责 | 内容 |
|---|---|---|
| 上传协议/编排 | **业务后端** | 分片、秒传、断点续传、限流、鉴权 —— 业务确实自己写 |
| 对象存储 | **基础设施/存储团队** | 分布式存储、副本、纠删码 —— 业务只调 SDK,不自己实现 |

也就是说:**业务层自研的是"协议编排",不是"把字节写进本地磁盘"**。本改造正是向这个分层靠拢——保留协议编排,存储交给对象存储。

## 4. 总体架构

```
前端 FileUploader
   │  (HTTP,接口契约不变)
   ▼
server 上传接口(协议编排:分片/秒传/断点续传)
   │  (统一调用,不碰 fs)
   ▼
StorageService(S3 API 封装)
   │  (S3 协议,@aws-sdk/client-s3)
   ├─ dev  → MinIO(docker 容器,localhost:9000)
   └─ prod → 腾讯云 COS(S3 兼容 endpoint)

秒传/分片会话元数据 → Postgres(Prisma)
```

存储抽象层 `StorageService` 是核心:它把所有对象存储操作收敛到一个模块,业务接口只依赖它的接口,不关心底层是 MinIO 还是 COS。

## 5. 技术选型:`@aws-sdk/client-s3`

**选 AWS SDK v3(`@aws-sdk/client-s3` + `@aws-sdk/lib-storage`),不选腾讯云专用 `cos-nodejs-sdk-v5`。**

| | @aws-sdk/client-s3(选) | cos-nodejs-sdk-v5(不选) |
|---|---|---|
| 协议 | 标准 S3 | 腾讯云专用 |
| MinIO 支持 | ✅ 原生 S3 | ❌ 不支持 |
| COS 支持 | ✅(COS 提供 S3 兼容 endpoint) | ✅ |
| dev/prod 一致 | ✅ 一份代码 | ❌ dev 没法用 MinIO |
| 可移植 | S3/MinIO/COS/OSS/R2 全通 | 锁死腾讯云 |

腾讯云 COS 官方提供 **S3 兼容 API**(`https://cos.<region>.myqcloud.com`,用 AWS SDK 直连),所以一套 AWS SDK 代码 dev 连 MinIO、prod 连 COS,完全可行。

**踩坑提示**:MinIO 必须用 **path-style**(`forcePathStyle: true`,即 `endpoint/bucket/key`);COS 既支持 virtual-host 也支持 path-style。统一开 path-style 最省心,用 env 控制。

## 6. 配置设计(env)

```
# S3 兼容对象存储(dev=MinIO / prod=COS,代码一份)
S3_ENDPOINT=http://localhost:9000          # dev MinIO;prod 改 COS endpoint
S3_REGION=us-east-1                         # MinIO 任意;COS 填实际 region 如 ap-guangzhou
S3_BUCKET=our-chat
S3_ACCESS_KEY=minioadmin                    # prod 换 COS SecretId
S3_SECRET_KEY=minioadmin123                 # prod 换 COS SecretKey
S3_FORCE_PATH_STYLE=true                    # MinIO 必须 true
S3_PUBLIC_BASE_URL=http://localhost:9000/our-chat   # 拼接文件访问 URL 的基址
```

- dev:`docker-compose.dev.yml` 默认值即可,开箱即用。
- prod:`.env` 覆盖为 COS 的 endpoint / 凭证 / 公网域名。

## 7. 数据模型(Prisma)

对象存储没有"目录树",秒传和断点续传需要的元数据落 Postgres:

```prisma
// 秒传:md5 → 已存在的对象
model UploadedFile {
  id        BigInt   @id @default(autoincrement())
  md5       String   @unique          // 文件内容 MD5
  objectKey String                    // S3 对象键
  size      Int
  mimeType  String?
  createdAt DateTime @default(now())
}

// 分片会话:fileId → S3 multipart 的 uploadId
model UploadSession {
  id          BigInt   @id @default(autoincrement())
  fileId      String   @unique        // 前端生成的文件标识
  uploadId    String                  // S3 createMultipartUpload 返回
  objectKey   String                  // 最终对象键
  fileName    String
  mimeType    String?
  fileMd5     String?
  totalChunks Int
  createdAt   DateTime @default(now())
}
```

> 分片各 part 的 `ETag` 不入库,改用 S3 `listParts(uploadId)` 实时查询(少维护一张表,且 S3 是 part 状态的权威来源)。

## 8. 存储抽象层接口

`src/storage/storage.service.ts`(示意):

```ts
putObject(key, body: Buffer, contentType?): Promise<void>
getObjectStream(key): Promise<Readable>
deleteObject(key): Promise<void>
headObject(key): Promise<{ size, contentType } | null>
publicUrl(key): string                    // 拼 S3_PUBLIC_BASE_URL

// multipart(分片)
createMultipartUpload(key, contentType?): Promise<string>   // 返回 uploadId
uploadPart(key, uploadId, partNumber, body): Promise<string> // 返回 ETag
listUploadedParts(key, uploadId): Promise<{ partNumber, etag }[]>
completeMultipartUpload(key, uploadId, parts): Promise<void>
abortMultipartUpload(key, uploadId): Promise<void>
```

底层用单例 `S3Client`(读 env)。所有上传接口只 import 这个 service。

## 9. 接口改造清单

挂载不变:`/user/uploads/uploadImg`(简单)、`/api/upload/*`(高级)。

| 接口 | 改造前(本地盘) | 改造后(S3) |
|---|---|---|
| `POST /user/uploads/uploadImg` | multer 存盘 | memoryStorage → `putObject`,返回 `publicUrl` |
| `POST /api/upload/single` | 存盘 + 读盘算 MD5 | memory → MD5(内存 buffer) → `putObject` + 写 `UploadedFile` |
| `POST /api/upload/multiple` | 同上批量 | 同上,`Promise.all` 并发 `putObject` |
| `POST /api/upload/check` | 遍历目录算 MD5 | 查 `UploadedFile` where md5,命中即秒传 |
| `POST /api/upload/chunk` | 分片写本地目录 | `uploadPart(uploadId, partNumber, buffer)`;首片前 `createMultipartUpload` 并写 `UploadSession` |
| `GET /api/upload/resume/:fileId` | 列目录数已传分片 | 查 `UploadSession` 取 uploadId → `listUploadedParts` |
| `POST /api/upload/merge` | 读分片顺序写+删分片 | `completeMultipartUpload(uploadId, parts)`,清 `UploadSession` |
| `POST /api/upload/compress` | sharp 读写本地文件 | memory → `sharp(buffer).jpeg().toBuffer()` → `putObject` |
| `POST /api/upload/stream` | 流式写本地 | `Upload`(lib-storage)流式直传;前端未用,低优先级 |

## 10. 关键机制详解

### 10.1 简单上传
multer 用 `memoryStorage`(文件进内存 buffer,不落盘)→ 算 MD5 → `putObject(key, buffer)` → 写 `UploadedFile` → 返回 `publicUrl(key)`。key 规则:`uploads/{yyyymm}/{md5或uuid}{ext}`。

### 10.2 图片压缩(无需临时文件)
这是上传流程,文件本就在 server 内存里,**sharp 直接 buffer→buffer**:
```ts
const out = await sharp(req.file.buffer).jpeg({ quality }).toBuffer();
await storage.putObject(key, out, 'image/jpeg');
```
不需要"从 COS 下载再压缩"——那是文件已在云上的场景,不适用上传流程。

### 10.3 秒传
前端先 `POST /check { md5 }` → 后端 `UploadedFile.findUnique({ md5 })`,命中直接返回已有 URL(秒传成功),不再传输文件。上传成功后写入该表,后续同内容文件即可秒传。

### 10.4 分片 / 断点续传 / 合并(S3 multipart)
- **初始化**:首个分片到达时(或单独 init),`createMultipartUpload(key)` 得 `uploadId`,存 `UploadSession(fileId→uploadId,key)`。
- **上传分片**:`uploadPart(key, uploadId, chunkIndex+1, buffer)`(S3 partNumber 从 1 起),S3 返回 ETag。
- **断点续传**:`resume` 查 session 的 uploadId → `listUploadedParts` 返回已传分片号给前端,前端跳过已传。
- **合并**:`merge` 调 `completeMultipartUpload(key, uploadId, listParts())`,S3 服务端合并,无需后端读写字节;清理 session。
- **约束**:S3 multipart 除最后一片外,**每片最小 5MB**。前端当前切片 5MB/片,正好满足;需在文档/代码注释里固化这个约束。

## 11. 文件访问

bucket 设为**公有读**,返回 `{S3_PUBLIC_BASE_URL}/{key}` 完整 URL,前端直接访问(dev: `http://localhost:9000/our-chat/...`;prod: COS 公网域名)。

- 删除 `app.ts` 里的 `express.static('/user/uploads')`(文件不再在本地)。
- 如未来需要私有文件,再引入预签名 URL(`getSignedUrl`),本期先用公读,简单且够用。

```yaml
  minio:

      MINIO_ROOT_USER: minioadmin
```

## 13. 分阶段实施

0. **基础设施**:装 SDK、`StorageService`、S3 配置、MinIO 入 dev compose + bucket 初始化。
1. **简单上传 + 压缩**:uploadImg/single/multiple/compress 改 S3(覆盖前端最常用路径)。
2. **秒传**:迁移 `UploadedFile` 表 + check 改造。
3. **分片**:chunk/resume/merge 改 S3 multipart + `UploadSession`。
4. **清理 + 端到端测试**:删本地盘残留(静态服务、目录创建、`UPLOAD_DIR`、冗余 `multer.ts`),MinIO 全链路验证。

每阶段独立可测,前端契约不变,可灰度。

## 14. 踩坑清单

- **MinIO 必须 path-style**(`forcePathStyle: true`),否则 `bucket.localhost` 解析失败。
- **multipart 最小 part 5MB**(除最后一片);前端切片需 ≥5MB。
- **partNumber 从 1 开始**,前端 chunkIndex 从 0,映射时 +1。
- **ETag**:`completeMultipartUpload` 需要每片的 PartNumber+ETag,从 `listParts` 取,顺序按 PartNumber 升序。
- **memoryStorage 内存上限**:大文件分片每片在内存,需配 multer `limits.fileSize` 与分片大小一致,避免单请求吃满内存。
- **CORS**:若未来改前端直传,bucket 需配 CORS;本期中转模式不涉及。
- **公读安全**:公有读意味着 URL 可被任意访问,敏感文件不可用公读(本期 IM 上传的图片/文件按公读处理,后续敏感场景改签名 URL)。

## 15. 兼容与回滚

- 改造保持接口契约,前端不动;`StorageService` 是唯一新增依赖点。
- 历史存量(本地 `uploads/` 旧文件)如需保留访问,可一次性 `mc cp` 刷入 bucket(本期 dev 数据可忽略)。
- 回滚:保留旧实现于 git 历史;`StorageService` 失败时接口返回 5xx,不静默吞。
