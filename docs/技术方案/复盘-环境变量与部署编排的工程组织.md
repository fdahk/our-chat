# 技术复盘:环境变量与部署编排的工程组织

记录一次"对齐工程化最佳实践"的改造:把散落的部署配置收口、把环境变量按运行模式集中。
重点不在结论,而在中途**走过的弯路与纠正过程**——它说明"对标大厂"必须先核实事实。

## 1. 起点

多语言多服务单仓(server=Node、gateway=Go、web=Vite、mobile)。改造前:

- 环境变量散落 4 处:根 `.env.example`、`server/.env.example`、`gateway/.env.example`、`web/.env.example`,且 server 与根之间变量大量重复。
- `docker-compose*.yml` 在仓库根;`nginx.conf` 烤进 web 镜像;本地盘上传残留。

目标:理清环境变量分层,把部署基础设施收口,达到可维护的工程组织。

## 2. 决策一:部署基础设施收口到 `docker/`

把 `docker-compose.yml` / `docker-compose.dev.yml` / 部署 `.env` / `nginx/` 全部收进顶层
`docker/`;容器侧用 `env_file` 单一注入;nginx 配置改为挂载而非烤进镜像。细节见
`工程组织与部署编排.md`,此处不展开。

## 3. 决策二:环境变量分层 —— 一段被事实纠正的判断

这是本次最值得复盘的部分。

### 3.1 初始的错误假设

我最初下了两个结论,**两个都错**:

1. "coze(对标的大厂开源项目)是全局一份 env。"
2. "要达到一份,应该把本地 dev 也容器化(coze 就是这么做的)。"

### 3.2 被仓库事实推翻

实际核实 coze 仓库后:

- `make debug`(coze 官方本地开发)= `middleware`(docker)+ `server`/`fe`(**宿主机直跑**)。
  `scripts/setup/server.sh` 里 `source docker/.env.debug` 后 `go build && ./opencoze` —— **业务在宿主机跑**,不是容器。
- `.env.debug.example` 全是 **localhost 地址**(`MYSQL_HOST=127.0.0.1`、`REDIS_ADDR=127.0.0.1:6379`)。
- coze **并非一份 env**,而是 **两份**:`docker/.env`(prod/容器名)+ `docker/.env.debug`(dev/localhost),都集中在 `docker/`、按运行模式拆分,用 `--env-file` 选择。

结论修正:**coze 本地开发同样是"业务跑宿主机 + 中间件跑 docker",与本项目一致;它的 env 是"集中 + 按运行模式拆",不是"一份"。**

### 3.3 根因:为什么不可能"全局一份"

一份文件无法同时满足两种运行拓扑的主机名:

| 变量 | 容器运行 | 宿主机直跑 |
|---|---|---|
| DATABASE_URL | `@postgres:5432` | `@localhost:5432` |
| REDIS_URL | `redis://redis:6379` | `redis://localhost:6379` |
| S3_ENDPOINT | `http://minio:9000` | `http://localhost:9000` |

容器内靠 docker 网络主机名互连;宿主机直跑靠中间件暴露到 localhost 的端口。
**"看似冗余"的根因是运行拓扑差异,不是配置重复。**

### 3.4 修正后的目标模型

> env 集中在 `docker/`,按运行模式拆:`.env`(prod/容器名)+ `.env.debug`(dev/localhost);
> 宿主机直跑的业务进程读那份集中的 `.env.debug`。`web/.env` 单列(见下)。

## 4. 落地实现

### 4.1 集中的 dev env
新增 `docker/.env.debug.example`(localhost 地址),合并原 `server/.env.example` +
`gateway/.env.example`。共享变量(`JWT_SECRET`、`GATEWAY_INTERNAL_TOKEN`、`REDIS_URL`)
从此**单一源**——顺带修了一个隐患:server 在 `JWT_SECRET` 为空时会自动随机生成,而
gateway 为空会 fail-fast,二者必须共享同一个值,集中后给了 dev 默认值。

### 4.2 宿主机进程如何读集中 env(工具约束决定方案)

server 和 gateway 读 env 的机制不同,因为它们的工具链加载 env 的方式根本不同:

- **Node(server)+ Prisma CLI**:靠 **dotenv** 自动读一个**名叫 `.env` 的文件**(从进程
  工作目录查找)。本质是"喂一个文件"。
- **Go(gateway)**:**没有**内建的 `.env` 自动加载,只认**进程环境变量**(`os.Getenv`)。
  本质是"喂一组环境变量"。

一个喂文件、一个喂环境变量,所以接法不同。

#### server / Prisma:用软链(symlink)

**什么是软链(符号链接 / symbolic link)?** 软链是一种特殊文件,本身不存内容,只是一个
指向另一个路径的"快捷方式/别名";任何对它的读取都被系统**透明转发到目标文件**。即:
`server/` 下有个叫 `.env` 的入口,但它其实通到 `docker/.env.debug` 那一份真实文件。

```bash
# 创建(目标路径相对软链自身所在目录 server/ 解析):
ln -s ../docker/.env.debug server/.env

# 查看(箭头表示这是软链,不是真实文件):
$ ls -l server/.env
server/.env -> ../docker/.env.debug
```

**为什么用软链,而不是复制一份?** Prisma CLI(`migrate` / `generate` / `studio`)和 Node
运行时都**默认从 `server/.env` 读**,这个路径是工具写死的、不便改。若在 `server/` 放一份
真实 `.env`,它就和 `docker/.env.debug` 成了两份会**各自漂移**的副本。软链让 `server/.env`
**就是** `docker/.env.debug`(同一份真实文件、两个访问入口),改一处两处都变。

- 真实文件只有一份:`docker/.env.debug`;
- 软链 `server/.env` 被 `.gitignore` 忽略,不进仓库,由 `make env` / setup 脚本在本地生成;
- 零新依赖、零脚本注入,Prisma 与运行时"无感"复用集中配置;
- 代价:软链是 Unix 特性(macOS/Linux 原生;Windows 需开发者模式)。

#### gateway:用 `source`(把 env 注入进程环境)

**`source` 是什么?** `source <文件>`(等价 `. <文件>`)让**当前 shell 直接执行文件里的每一行**,
就像你亲手把这些行敲进终端。对一个全是 `KEY=value` 的 env 文件,执行结果就是:每行变成
当前 shell 里的一个变量赋值。

**`set -a` 是什么?** 默认 `KEY=value` 只是个**普通 shell 变量**,子进程(`go run` 起来的网关)
**看不到**。`set -a`(allexport)打开后,之后的赋值都**自动导出成环境变量**(子进程可见),
`set +a` 关闭:

```bash
set -a                                 # 开启自动导出
. docker/.env.debug                    # 把文件里每个 KEY=value 读进当前 shell 并导出
set +a                                 # 关闭自动导出
cd gateway && go run ./cmd/gateway     # 网关用 os.Getenv 即可读到这些变量
```

**为什么 gateway 要这样、server 不用?** 因为 Go 没有 dotenv 那种"自动读 .env"的能力,
只认进程环境变量,所以必须在启动前把集中文件**注入进程环境**——`source` 就是 shell 里
干这件事的标准手段(coze 的 `scripts/setup/server.sh` 同样是 `source docker/.env.debug`
后再跑 Go,思路一致)。

#### 小结:两种机制,殊途同归读同一份

| 进程 | 工具如何拿 env | 我们的接法 | 最终读的文件 |
|---|---|---|---|
| server / Prisma | dotenv 自动读 `server/.env` 文件 | 软链 `server/.env → ../docker/.env.debug` | `docker/.env.debug` |
| gateway | 只认进程环境变量(`os.Getenv`) | `set -a; . docker/.env.debug; set +a` | `docker/.env.debug` |

### 4.3 统一 dev 编排:根 Makefile(一键 `make dev`)

**Make / Makefile 是什么?** Make 是任务运行器:`make <目标>` 执行该"目标(target)"下的
命令;目标间可声明依赖(`dev: env deps middleware` 表示跑 `dev` 前先按序跑这三个)。我们用它
做**一键 dev 入口**,也因为它是唯一能干净地为 gateway 做 `source` 注入、并发管理多个前台进程
的地方(npm 脚本不便为 Go 进程 source env)。

核心是 **`make dev` 一条命令拉起全部**(中间件 + server + gateway + web):

```makefile
dev: env deps middleware                 # 先把 集中env / 依赖 / 中间件 准备好
	@printf '⏳ 等待 PostgreSQL'; \        # server 启动要跑 migration,先等 PG 就绪
	until docker compose -f $(DEV_COMPOSE) exec -T postgres pg_isready -U postgres >/dev/null 2>&1; \
	  do printf '.'; sleep 1; done; echo ' ✓'
	@trap 'kill 0' INT TERM EXIT; \        # Ctrl-C/退出时杀掉整个进程组,三个业务一起停
	( cd server && pnpm dev ) & \          # 各自子 shell 后台 & 并发跑
	( set -a; . $(ENV_DEBUG); set +a; cd gateway && go run ./cmd/gateway ) & \
	( cd web && pnpm start ) & \           # 注意 web 是 vite,命令为 pnpm start(非 dev)
	wait                                   # 阻塞,直到 Ctrl-C 触发 trap
```

并发与退出的关键三件套:

- `( ... ) &`:每个业务放进**子 shell 后台运行**,三者并发;
- `wait`:阻塞当前 shell,直到子进程结束;
- `trap 'kill 0' INT TERM EXIT`:捕获 Ctrl-C(INT)等信号后,`kill 0` 向**当前进程组**发信号,
  把三个业务(含 `go run` 编出来的子进程)一起干掉——避免"关了终端进程还残留"。

辅助目标(幂等,`test -f X || cmd` = X 不存在才执行):

```makefile
env:                # 生成集中 dev env(docker/.env.debug)+ 软链 server/.env
deps:               # 首次 pnpm install + 生成 Prisma Client(已就绪则跳过)
middleware / down:  # 起 / 停中间件
```

> 设计取舍:**不拆成 `make server/gateway/web` 多条命令**,而是 `make dev` 一把起全部
> (输出合并、Ctrl-C 一起停),补上了项目原先缺的"一键 dev 入口"。

> `test -f X || cmd` = "X 不存在才执行 cmd",保证 `make env` 可重复执行不报错(幂等)。
> 这也补上了项目原先缺的"根任务入口"——之前没有统一的 dev 启动命令。

### 4.4 连带修复(移动/删除文件后,必须回扫旧引用)

把 compose 移进 `docker/`、删掉散落 env 后,有几处旧引用会断,同步修掉:

- **`server/package.json` 的 `docker:*` 脚本**:原来指向 `../docker-compose*.yml`(已移走)
  → 改成 `../docker/docker-compose*.yml`;否则 `pnpm docker:mw` / `docker:up` 等会报"找不到
  compose 文件"。
- **`server/scripts/setup-dev.sh`**(即 `pnpm setup` 的一键初始化脚本),改了三处:
  1. **准备 env**:原来 `cp .env.example .env`(server 本地,已删)→ 改为生成
     `docker/.env.debug` 并软链 `server/.env`;
  2. **compose 调用**:原来是裸 `docker compose ...`,依赖**已删除的 `server/docker-compose.yml`**
     → 全部改成 `docker compose -f ../docker/docker-compose.dev.yml ...`(脚本里用变量 `$DC` 统一);
  3. **结尾提示**:删掉过时的 pgAdmin `--profile tools`(新 dev compose 无此 profile),
     看日志改走 `pnpm docker:logs:pg`。
- **删** `server/.env.example`、`gateway/.env.example`(内容并入 `docker/.env.debug.example`)。

### 4.5 最终形态
```
docker/.env.example        prod / 容器(env_file 注入)
docker/.env.debug.example  dev / 宿主机(localhost;server 经软链、gateway 经 source 读)
web/.env.example           Vite 构建期(VITE_ 前缀,公开值;跨仓库绝对地址)
```
`web/.env` 单列是因为 Vite 构建期注入 + agent-server 是跨仓库绝对地址;coze 因前端同源无此项,我们有。

## 5. 验证
- `make help` / `make env` 正常;`docker/.env.debug` 可被 `shell source`(gateway 路径)与软链读取(server/Prisma 路径)。
- `docker compose config` 对 prod / dev 均通过。

## 6. 经验教训

1. **对标大厂前先读仓库,别凭印象。** 本次两个错误都源于没核实 coze 的 dev 流程就下结论。
2. **"最佳实践"要看适用前提。** coze 能"少一份 env",前提是同源前端(无构建期 URL);它并没有"一份"。机械照搬别人的形,会得出错误结论。
3. **不可共享的根因常是运行拓扑(主机名),不是配置冗余。** 先分清"为什么不同"再谈"要不要合并"。
4. **工具约束影响方案选择。** Prisma CLI 的 `.env` 自动加载,直接决定了用软链而非环境变量注入。
5. **改目录/移动文件后必须回扫引用**(package.json 脚本、setup 脚本、文档),否则留下隐性断链。
