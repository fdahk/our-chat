# 06 · 路由组件改懒加载，拆分首屏巨型包

> 类别：性能（首屏加载 / 网络）
> 涉及文件：`src/router/index.tsx`

## 一、问题概述

`router/index.tsx` 把所有页面级组件**静态 import** 到路由表顶部：

```tsx
import Layout from "@/views/layout/index.tsx";
import LoginView from "@/views/loginView/index.tsx";
import RegisterView from "@/views/registerView/index.tsx";
import DirectoryView from "@/views/directoryView/index.tsx";
import ChatView from "@/views/chatView/index.tsx";
```

静态 import 意味着这五个视图、以及它们各自牵连的依赖（chatView 拖着 socket、WebRTC、文件上传；settingView 拖着 cropper 图片裁剪库……）**全部被打进同一个入口 chunk**。结果是：用户访问登录页 `/login`，浏览器却要先下载、解析、执行整个聊天系统的代码——而这些代码在登录这一刻一行都用不到。

构建产物印证了这点：拆分前 `dist/assets/` 只有一个约 **1327 kB** 的 `index-*.js`，入口 HTML 直接引它，首屏必须等它全部到位才能渲染。

## 二、为什么这是性能问题：JS 是"阻塞型"资源

图片慢，顶多占位符晚点出现；但 JS 不一样，它要走「下载 → 解析 parse → 编译 compile → 执行 execute」四步，且 SPA 的首屏**完全依赖 JS 执行完才出现**。包越大：

1. **下载更久**：弱网 / 移动端尤其敏感，1.3MB 和 570KB 是数量级的体感差异；
2. **解析编译更久**：这一步吃主线程，低端机 CPU 上 parse 大 JS 能到数百毫秒，期间页面卡白；
3. **执行更久**：模块初始化、建立 React 树都在主线程同步跑。

衡量首屏的核心指标 FCP / LCP / TTI 全都被入口包体积直接拖累。而登录页本该是最轻的页面，却背着整个 App 的包，这是纯粹的浪费。

## 三、修复：路由级 `lazy`，按需切包

React Router v7 的数据路由（`createBrowserRouter`）原生支持路由级 `lazy` 属性——值是一个返回路由模块的异步函数。把静态 import 换成动态 `import()`：

```tsx
// 把动态 import 的默认导出包装成 React Router 的 lazy 路由模块
const lazyComponent =
    (loader: () => Promise<{ default: ComponentType }>) =>
    async () => ({ Component: (await loader()).default });

const router = createBrowserRouter([
    {
        path: "",
        lazy: async () => {
            const { default: Layout } = await import("@/views/layout/index.tsx");
            return { Component: () => <RequireAuth><Layout /></RequireAuth> };
        },
        children: [
            { index: true, element: <Navigate to="/chat" replace /> },
            { path: "chat", lazy: lazyComponent(() => import("@/views/chatView/index.tsx")) },
            { path: "directory", lazy: lazyComponent(() => import("@/views/directoryView/index.tsx")) },
        ],
    },
    { path: "login", lazy: lazyComponent(() => import("@/views/loginView/index.tsx")) },
    { path: "register", lazy: lazyComponent(() => import("@/views/registerView/index.tsx")) },
    ...devOnlyRoutes,
]);
```

`import()` 是**代码分割的触发点**：Vite/Rollup 见到动态 import，就把目标模块及其私有依赖切成一个独立 chunk，只有真正导航到该路由时才发请求拉取。于是登录页只下载登录所需的代码，聊天系统的包要等你进 `/chat` 才加载。

## 三·补 · 逐行拆解：`lazyComponent` 与路由 `lazy` 到底做了什么

上面那段 `lazyComponent` 三行代码信息密度很高，下面把它和路由 `lazy` 属性拆开，从最底层讲清楚。

### 1. `import("...")` 返回的是什么

`import("@/views/chatView/index.tsx")` 是**动态导入**，它不返回组件，而是返回一个 **Promise**。这个 Promise 兑现（resolve）后得到的值叫 **模块命名空间对象（module namespace object）**——就是这个模块所有导出的集合：

```ts
// chatView/index.tsx 里写的是 `export default ChatView`
const mod = await import("@/views/chatView/index.tsx");
// mod === { default: ChatView, /* 若还有具名导出也会在这里 */ }
mod.default; // 这才是 ChatView 组件本身
```

关键点：**默认导出挂在 `.default` 键上**。所以静态写法 `import ChatView from "..."` 里的 `ChatView`，等价于动态写法里的 `mod.default`。`(await loader()).default` 这一截，干的就是"等 Promise 兑现，取出默认导出"。

### 2. 路由 `lazy` 属性要的是什么——不是组件，是"路由模块"

这是最容易误解的一点。React Router v7 数据路由里，`lazy` 的值是**一个 async 函数**，它返回的不是组件，而是一个**部分路由对象（partial route object）**——即"这条路由里需要延迟加载的那些字段"：

```ts
lazy: async () => {
    // 返回值是路由字段的集合，常见就一个 Component，
    // 还可以带 loader / action / errorElement / ErrorBoundary 等
    return { Component: SomeComponent };
}
```

React Router 的工作流程是：当导航**第一次命中**这条路由时，它调用你的 `lazy` 函数 → `await` 它 → 把返回对象里的字段**合并进这条路由定义**（相当于"这条路由现在才知道自己的 `Component` 是谁"）→ 然后才渲染。解析过一次后会缓存，再次进入不会重复加载。

所以：`import()` 给的是 `{ default: 组件 }`，而 `lazy` 要的是 `{ Component: 组件 }`。**键名对不上**——`default` ≠ `Component`。中间必须有一层转换。`lazyComponent` 就是这层转换器。

### 3. `lazyComponent` 逐层展开

```ts
const lazyComponent =
    (loader: () => Promise<{ default: ComponentType }>) =>   // 第一层：收下"导入动作"
    async () => ({ Component: (await loader()).default });    // 第二层：返回 React Router 要的 lazy 函数
```

它是一个**返回函数的函数**（柯里化）。逐步看 `lazyComponent(() => import("@/views/chatView/index.tsx"))` 这次调用：

1. **传入的 `loader` 是 `() => import(...)`，一个"还没执行"的导入动作**。注意是 thunk（包了一层箭头函数），不是 `import(...)` 本身——这点见下条，至关重要。
2. `lazyComponent` 立刻返回里层那个 `async () => (...)` 函数。**此刻还没有任何网络请求**，只是把"将来要怎么加载"这件事打包好，交给路由表的 `lazy` 字段存着。
3. 将来用户导航到 `/chat`，React Router 调用这个 `lazy` 函数。函数体此时才执行：
   - `loader()` 被调用 → 真正触发 `import(...)` → 浏览器去拉 chatView 的 chunk，得到一个 Promise；
   - `await` 等它兑现 → 拿到模块命名空间对象 `{ default: ChatView }`；
   - `.default` 取出 `ChatView`；
   - 包成 `{ Component: ChatView }` 返回给 React Router。
4. React Router 拿到 `{ Component }`，合并进路由，渲染 `<ChatView />`。

一句话：**`lazyComponent` 把"动态 import 的 `.default`"适配成"React Router 要的 `Component`"，并且把这个适配逻辑复用到每条路由，省得每条都手写一遍 `async () => ({ Component: (await import(...)).default })`。**

对比根路由那条**没用** `lazyComponent` 的写法，正好印证：

```ts
lazy: async () => {
    const { default: Layout } = await import("@/views/layout/index.tsx");
    return { Component: () => <RequireAuth><Layout /></RequireAuth> };
}
```

这里不能直接套 `lazyComponent`，因为 `Component` 不是裸的 `Layout`，而是"被 `RequireAuth` 包过的 Layout"。所以手写展开了——但你能看到它和 `lazyComponent` 内部做的是同一件事：await 拿默认导出、组装成 `{ Component }`。

### 4. 为什么必须是 `() => import(...)` 而不是 `import(...)`

这是新手最常踩的坑。如果写成：

```ts
lazy: lazyComponent(import("@/views/chatView/index.tsx")) // ❌ 错
```

那么 `import(...)` 会在 **router/index.tsx 这个模块被求值的瞬间**就执行——也就是 App 一启动、路由表刚被创建时，五个视图的 chunk 就**全部立刻开始下载**了。懒加载彻底失效，等于换了种写法的"全量加载"。

包一层 `() => import(...)` 是**惰性求值（lazy evaluation）**：把"执行导入"这个动作推迟成一个可调用的函数，只有 React Router 在导航时真正 `loader()` 调用它，`import()` 才发生。`React.lazy` 也是同理，它的参数也必须是 `() => import(...)`。**记住：动态 import 写在箭头函数体里，才能延迟到需要时才触发。**

## 四、两个关键设计决策

### 1. `RequireAuth` 守卫保持同步导入

```tsx
import RequireAuth from "@/utils/requireAuth"; // 不参与懒加载
```

鉴权守卫是进入受保护路由前的**同步关卡**，体积极小（就是读 token、判断有效性、决定放行或跳登录）。如果连它也懒加载，等于每次进受保护页都要先多一次网络往返才能开始鉴权——得不偿失。所以 Layout 懒加载，但用同步的 `RequireAuth` 包裹它。

### 2. 为什么路由级 `lazy` 而非 `React.lazy` + `<Suspense>`

两者都用 `() => import(...)` 切包，**切出来的 chunk 完全一样**。区别在"什么时候触发加载、加载时拿什么兜底"——而这背后是两套完全不同的机制：

| 方案 | 触发时机 | 底层机制 | 加载态由谁负责 |
|------|---------|---------|--------------|
| 路由级 `lazy`（采用） | **导航时**（用户点链接、地址变化） | React Router 把它当作导航生命周期的一环去 `await`，解析完才提交路由状态 | 路由自身：解析期间用 `navigation.state==="loading"` 暴露给你做 pending UI，无需 Suspense |
| `React.lazy` + `<Suspense>` | **渲染时**（React 真要画这个组件） | 组件未就绪时**抛出 Promise**，由最近的 `<Suspense>` 边界接住、渲染 fallback、订阅 Promise，兑现后重渲染 | 你自己：必须布 `<Suspense fallback>`，否则 React 直接报错 |

把两者的底层原理讲透：

- **`React.lazy` 走的是 Suspense 协议，是"渲染时"机制。** `React.lazy(() => import(...))` 返回一个特殊的组件占位。当 React 渲染到它、发现模块还没加载好时，这个占位组件会**直接 `throw` 那个 Promise**（React 约定：渲染中抛出一个 thenable，就表示"我还没准备好，请挂起"）。这个抛出被往上冒泡，被最近的 `<Suspense>` 捕获——于是 Suspense 渲染 `fallback`（比如转圈），同时订阅这个 Promise；Promise 兑现后 React 重新渲染该子树，真正的组件出现。所以：**没有 `<Suspense>` 包着，抛出的 Promise 没人接，React 就报错"A component suspended ... no Suspense boundary"**。加载发生在"组件即将上屏"的渲染阶段。

- **路由级 `lazy` 走的是路由导航生命周期，是"导航时"机制，归 React Router 管，跟 React 的 Suspense 无关。** 当你导航到一条路由，React Router 进入一次**过渡（transition）**：在把新路由真正渲染到屏幕之前，它先检查这条路由有没有未解析的 `lazy`，有就 `await`（和执行 loader 数据加载是同一阶段、同一套等待机制）。只有 `lazy` 解析完、`Component` 就位了，路由器才更新自己的状态去渲染。**所以等 React 开始渲染时，组件早已是现成的，根本不会发生"渲染到一半缺组件"的情况，自然不需要 Suspense 来兜底。**

- **两者的"加载态"体感也因此不同。** `React.lazy`：旧内容会被 Suspense 的 fallback 顶替（容易出现"先白屏/转圈，再出内容"）。路由级 `lazy`：加载发生在导航过程中，你可以让**上一个页面继续停留**直到新路由就绪（读 `useNavigation()` 的 `state` 显示顶部进度条之类），过渡更平滑——这正是"数据路由"把数据加载和代码加载统一进导航生命周期带来的好处。

结论：路由级 `lazy` 与"按路由切包"语义天然吻合——加载发生在"导航"这个用户本来就有等待预期的时刻，由路由统一调度，省去满屏布 Suspense 边界。注意调试子路由（见报告 01）的 `DebugLayout` 内部仍保留了 `<Suspense>`，那是因为它的子路由也用同一套 lazy，外层兜一道更稳妥，属于防御性写法。

## 五、取舍

- **代价**：首次进入某路由时多一次 chunk 请求，弱网下会有极短暂的导航延迟。但换来的是首屏入口包砍半，且后续路由的包可被浏览器独立缓存——总体净赚。
- **进一步优化空间**：可对高频路由做 `<link rel="prefetch">` 或在空闲时预取下一个可能访问的 chunk（router 的 lazy 也支持手动预热）。当前未做，属可选增强。
- **未拆的部分**：`Navigate` 重定向、`RequireAuth` 仍同步——它们都是 KB 级的轻量逻辑，拆了反而增加请求数，不划算。

## 六、业界做法

- **按路由切包是 SPA 的默认最佳实践**，React/Vue 官方文档都把它列为首要性能优化项。
- 大型应用还会叠加：`manualChunks` 把 React、antd 等稳定第三方库单独成 vendor chunk（长效缓存，业务迭代不失效）；对核心路由做预取；用 `build.chunkSizeWarningLimit` 配合监控包体积回归。
- 本次先解决"全部塞进一个入口包"这个最大头的问题；vendor 分包可作为后续独立优化项。

## 七、验证

- `pnpm build`：入口 `index-*.js` 从约 **1327 kB 降到约 570 kB**，其余视图各自产出独立 chunk，入口 HTML 仅引用入口包。
- `pnpm lint`：0 error。
- 开发与生产下登录、注册、聊天、通讯录各路由功能正常，切换路由时按需加载对应 chunk。
