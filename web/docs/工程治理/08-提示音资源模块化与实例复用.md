# 08 · 消息提示音：硬编码路径改模块导入 + Audio 实例复用

> 类别：正确性（资源 404）/ 性能（重复创建对象）
> 涉及文件：`src/hooks/useGlobalMessageListener.ts`

## 一、问题概述

全局消息监听 hook 里这一行同时踩了两个坑：

```ts
export default function useGlobalMessageListener() {
    // ...
    const audio = new Audio('src/assets/audios/message.wav'); // ①路径硬编码 ②每次 render 重建
    // ...
    audio.play(); // 收到消息时播放
}
```

1. **硬编码相对路径 `'src/assets/audios/message.wav'`**——和报告 07 的图片是同一类问题：`src/` 是源码目录，不是 Web 可访问路径，子路由/生产环境下浏览器会拼出取不到的 URL 而 404，提示音根本不响。
2. **`new Audio(...)` 写在组件（hook）函数体里**——每次组件 render 都会**重新构造一个 Audio 对象**。这个 hook 挂在 App 顶层，任何上层状态变化都会触发 render，于是一路狂建 Audio 实例，纯属浪费。

## 二、坑一：资源路径——同 07，改 ES 模块导入

道理与报告 07 完全一致，不再展开。修复是把音频当模块 `import`，交给 Vite 纳入打包、产出带 hash 的真实 URL：

```ts
import messageSound from '@/assets/audios/message.wav';
// ...
audioRef.current = new Audio(messageSound); // messageSound 是构建后真实可访问的 URL
```

> 类型支持：`vite-env.d.ts` 的 `/// <reference types="vite/client" />` 已为 `*.wav/*.mp3` 等音频后缀提供模块声明，`import x from './x.wav'` 才有类型。
>
> 这里选**就地 import**（不像图片那样建集中 `index.ts`）：提示音只此一处用到，单点引用就地 import 最直接；报告 07 也说过"各文件就地 import 也对"，集中导出主要是为多处复用的资源收口。

构建验证：`message.wav` 约 346KB，远超 `assetsInlineLimit`（4KB），故产出为独立的 `dist/assets/message-<hash>.wav`，而非内联。

## 三、坑二：为什么不能在组件体里 `new Audio`——React 渲染模型

React 函数组件/hook 的函数体，**每次 render 都会完整重跑一遍**。任何写在函数体顶层的表达式都会重复执行。`const audio = new Audio(...)` 因此每次 render 都新建一个 `HTMLAudioElement`：

- **浪费对象**：旧的 Audio 还没用上就被丢弃，等 GC 回收；
- **状态丢失**：Audio 对象自身带播放状态（当前进度、是否正在播放、音量等）。每次 render 换一个新实例，意味着这些状态永远留不住。对"播放提示音"这种瞬时操作影响还小，但这是个会咬人的坏习惯——换成需要 `pause()`/续播的场景就直接出 bug。

正解是把"跨 render 需要保持同一个的可变对象"放进 **`useRef`**。ref 的 `.current` 在组件整个生命周期里是**同一个稳定容器**，render 之间不重建：

```ts
const audioRef = useRef<HTMLAudioElement | null>(null);
// 惰性初始化：只有第一次（current 还是 null）才 new，之后所有 render 都复用同一个实例
if (audioRef.current === null) {
    audioRef.current = new Audio(messageSound);
}
// ...
audioRef.current?.play();
```

### 为什么用 `useRef` 而不是 `useState` / `useMemo`

| 方案 | 是否合适 | 原因 |
|------|---------|------|
| `useRef` + 惰性初始化（采用） | ✅ | Audio 是"可变实例、不参与渲染输出"的典型场景。改它不需要触发重渲染，ref 正是为此而生 |
| `useState(() => new Audio(...))` | ⚠️ 可行但不当 | state 用于"变化要驱动 UI 重渲染"的数据；Audio 实例变了不该重渲染，用 state 是语义错配 |
| `useMemo(() => new Audio(...), [])` | ⚠️ 不可靠 | React 官方明确：useMemo **不保证**缓存不被丢弃（内存压力下可能重算）。用它持有"必须唯一"的实例不安全 |
| 模块顶层 `const audio = new Audio(...)` | ❌ | 模块求值时就构造，且全模块共享单例——SSR/多实例场景会有副作用，也无法随组件卸载清理 |

### 为什么写 `if (current === null)` 而不是 `useRef(new Audio(...))`

`useRef(new Audio(messageSound))` 看着更短，但 **`new Audio(...)` 仍会在每次 render 时求值**——只是求值结果除第一次外都被 useRef 丢弃。等于"每次 render 都白建一个 Audio 再扔掉"，没真正省下构造开销。`if (current === null)` 的惰性初始化才能保证 `new Audio` **全程只执行一次**。这是 React 官方文档推荐的"昂贵对象只初始化一次"写法。

## 四、顺带：依赖数组也更干净了

原来 `audio` 是组件体里的局部变量，被下面的 `useEffect`（空依赖）引用，ESLint 的 `exhaustive-deps` 会把它当成"缺失依赖"来警告。改用 `audioRef` 后，**ref 本身是稳定的，无需进依赖数组**，这条警告随之消失——这也是 ref 相比普通变量在 effect 里的额外好处。

## 五、业界做法

- **资源即模块**是 Vite/webpack 标准范式，音频/视频/字体与图片一视同仁（见报告 07）。
- **"可变实例放 ref、惰性初始化"** 是 React 处理命令式对象（Audio、`IntersectionObserver`、第三方图表/地图实例、WebSocket 等）的通用模式——这些对象都不该参与渲染、且必须跨 render 唯一。
- 进一步可做的（本次未做，属增强）：把提示音封装成一个 `usePlaySound` 自定义 hook，内部处理"浏览器自动播放策略"（首次用户交互前 `play()` 会被拒绝，可捕获 rejection 静默忽略）。

## 六、验证

- `grep -rn "'src/assets" src`：路径硬编码全部清零。
- `pnpm build`：产出 `dist/assets/message-<hash>.wav`（约 346KB，未内联，符合超阈值预期）。
- `pnpm lint`：0 error，且原先针对 `audio` 缺失依赖的警告消失。
- 各路由下收到消息/好友请求时提示音正常播放，不再 404。
