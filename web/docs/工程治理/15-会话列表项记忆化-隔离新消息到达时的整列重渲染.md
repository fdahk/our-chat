# 15 · 会话列表项记忆化——隔离"新消息到达"时的整列重渲染

> 类别：性能（渲染 / 重渲染范围控制）
> 涉及文件：`web/src/globalComponents/displayItem/index.tsx`、`web/src/views/chatView/index.tsx`
> 关联：本项是报告 14 末尾明确列出的"正交独立优化项"。报告 14 解决"打字时整列重渲染"，本报告解决"收到新消息时整列重渲染"——两者根因不同、互不替代。

## 一、问题本体

`ChatView` 左侧会话列表这样渲染：

```tsx
{Object.values(globalConversations).map((item) => (
  <DisplayItem
    key={item.id}
    avatar={...buildServerUrl(...)}
    title={globalFriendInfoList[...]?.username}
    content={lastMessages[item.id]?.content || ''}
    isActive={activeConversation === item.id}
    handleClick={handleClickConversation}
  />
))}
```

每当**任意会话来一条新消息**，全局监听器会 `dispatch(addGlobalMessage)` + `dispatch(addLastMessage)`，使 `ChatView` 订阅的 `messages`、`lastMessages` 选择器变化 → `ChatView` 重新执行 → `.map()` 重跑 → **所有 `DisplayItem` 全部重渲染**，哪怕其中只有一行的 `content`（最后一条消息预览）真的变了。

`DisplayItem` 此前**无任何记忆化**，挡不住这次重渲染。结果：**会话越多，每来一条消息，左侧整列都白跑一遍**——重渲染成本随会话数量线性增长，且与"消息属于哪个会话"无关（来 A 的消息，B/C/D… 全部陪跑）。

这是与报告 14 同源但触发点不同的问题：报告 14 是"自身高频 state（打字）波及列表"，本报告是"父组件因**他人**状态变化重渲染，连累所有子项"。

## 二、概念扫盲

### 2.1 `React.memo` 是什么
`memo(Component)` 返回一个记忆化组件：父组件重渲染时，React 对它做 **props 浅比较（shallow compare）**——逐个 prop 用 `Object.is` 比较。**全部相等就跳过这次重渲染**（复用上次结果），有一个不等就照常重渲染。

注意它拦截的是"**父重渲染带来的连带重渲染**"，不拦截自身 state/context 变化引起的重渲染（那是应该发生的）。

### 2.2 为什么浅比较对本场景成立
`DisplayItem` 的 props 全是**基本类型**：`avatar`/`title`/`content` 是字符串，`isActive` 是布尔。浅比较按值比即可：

- 来了 A 会话的新消息 → 只有 A 那行的 `content`（`lastMessages['A'].content`）变了 → 仅 A 重渲染；
- B/C/D 的 `content` 等所有 prop 取值不变 → `Object.is` 全等 → **被 memo 跳过**。

重渲染范围从"整列 N 项"收敛到"真正变化的 1 项"。

### 2.3 关键前提：引用稳定性（这步不做，memo 等于白加）
浅比较对**函数/对象 prop 按引用比**。`handleClick={handleClickConversation}` 若每次 `ChatView` 渲染都是**新函数**，浅比较必然判不等，memo **每次都失效**。

原代码 `const handleClickConversation = async (id) => {...}` 正是"每次渲染新建一个函数"。所以必须用 `useCallback` 把它的引用固定下来：

```tsx
const handleClickConversation = useCallback(async (conversationId: string) => {
  dispatch(initActiveConversation(conversationId));
  const res = await getConversationMessages(conversationId);
  dispatch(initGlobalMessages(res.data ?? []));
}, [dispatch]); // dispatch 引用稳定，故该回调全程同一引用
```

`memo + useCallback` 是**配套出现**的：memo 负责"props 没变就跳过"，useCallback 负责"让函数 prop 真的不变"。缺一不可——这也是为什么报告 14 说"memo 心智负担大、易因一个不稳定 prop 失效"。

## 三、实现

### 3.1 `DisplayItem` 加 memo
```tsx
import { memo } from 'react';
function DisplayItem(props: DisplayItemProps) { /* 原样不动 */ }
export default memo(DisplayItem);
```
组件内部逻辑零改动，只在导出处包一层。所有调用点（会话列表、好友列表等）透明无感，无需改动。

### 3.2 `ChatView` 用 `useCallback` 固定 handler
把 `handleClickConversation` 包进 `useCallback(fn, [dispatch])`。`dispatch`、`getConversationMessages`、`initActiveConversation`、`initGlobalMessages` 均为稳定引用（前者来自 react-redux，后三者是模块级导入），故依赖只需 `[dispatch]`，回调全程同一引用。

其余 prop（`avatar`/`title`/`content`/`isActive`）都是基本类型，天然按值比较，无需额外处理。

## 四、各方案取舍

| 方案 | 思路 | 取舍 |
|------|------|------|
| **memo + useCallback（本次）** | 子项门口设卡 + 稳定函数 prop | ✅ 精准隔离"他人消息"引起的连带重渲染；props 恰好全是基本类型，浅比较开箱即用；首选 |
| 只加 memo 不加 useCallback | 仅包 memo | ❌ 假修复：handleClick 每次新引用，memo 必失效，等于没加还白付比较成本 |
| 把会话列表抽成子组件 + 状态下沉 | 仿报告 14 把 map 移到独立组件 | 对"打字"有效，但本问题的触发源（`lastMessages` 变化）在 redux、列表必须订阅它，下沉无处可沉——故本场景 memo 才是对的工具 |
| 列表虚拟化 | 只渲染视口内会话 | 正交：解决"会话**成百上千**时单次渲染贵"，与本问题（**频繁**触发整列重渲染）不同向；会话量极大时才上 |
| `useMemo` 缓存整段 `map` 结果 | 缓存 VDOM 数组 | 依赖数组要囊括 `globalConversations/lastMessages/...`，任一变就整体失效，粒度太粗、收益不如逐项 memo |

**为什么这次该用 memo，而报告 14 不该？** 区别在触发源：报告 14 的高频 state（草稿）是 `ChatView` **自己的**，可以"下沉"到子组件让父组件压根不重渲染——根因修复优于 memo。而本次触发源是 `lastMessages`（redux 全局态），列表天然要订阅它、父组件该重渲染，没法"下沉"消除；要避免的只是"连累不相关的兄弟项"，这正是 `memo` 的设计用途。**工具要对症，不是 memo 一律不用，而是先判断根因能否靠结构消除，不能再用 memo。**

## 五、踩坑记录

1. **memo 不配 useCallback = 白加**。函数 prop 不稳定时浅比较永远判不等，memo 不仅不生效，还平添每次比较的开销。本次同步用 `useCallback` 固定 `handleClick`。
2. **依赖数组要诚实**。`useCallback` 的依赖必须含回调内引用的所有外部可变值。本例只用到 `dispatch`（稳定），故 `[dispatch]`；若日后回调里读了某个 state，必须加进去，否则会拿到旧闭包值。
3. **别给 memo 传内联对象/数组 prop**。如 `style={{...}}` 或 `data={[...]}` 写在 JSX 里会每次新建引用，使 memo 失效。本组件 `style` 由调用方决定，会话列表未传，安全；但这是 memo 最常见的失效来源，需警惕。
4. **memo 的浅比较本身有成本**。props 极多或组件极轻时，比较开销可能超过重渲染省下的，反而更慢。本组件 props 少（5 个基本类型）、渲染含 DOM 结构与图片，净收益为正。**memo 不是越多越好，要确认"重渲染成本 > 比较成本"再加。**
5. **key 与 memo 是两回事**。会话列表 `key={item.id}` 解决的是"列表项身份识别/复用"，memo 解决的是"identity 不变时要不要重渲染"。本次只动 memo；消息列表用下标 key 的隐患是独立问题，另行评估。

## 六、业界对比与 Web vs Native

- **业界**：`memo + useCallback/useMemo` 是 React 处理"长列表/频繁父更新"的标准组合拳。React 官方性能指南的次序是 **先 colocate（状态下沉）、再 memo**——本项目两份报告正好示范了这个次序：能下沉的（报告 14）下沉，不能下沉的（本报告）才上 memo。社区里 `react-redux` 的 `useSelector` + 组件 memo 也是"只让数据真正变化的组件重渲染"的经典模式；更激进的方案会用 selector 记忆化（reselect）从源头减少订阅触发。
- **React 19 与 React Compiler**：React 19 起的 **React Compiler（编译期自动记忆化）** 目标就是让开发者**不必手写 memo/useCallback**——编译器自动插入记忆化。本项目用 React 19 但（暂）未启用 Compiler，故仍手写。可在报告里点明："若启用 React Compiler，本次手写的 memo/useCallback 可由编译器自动完成"，体现对前沿的认知。
- **Web vs Native（重渲染裁剪的共通法则）**：
  - **React Native**：同一套 reconciler，`React.memo`/`useCallback` 语义完全一致；RN 长列表更进一步用 `FlatList` 的 `getItemLayout` + 自带回收，思路与 memo 互补（一个减重渲染、一个减挂载数量）。
  - **SwiftUI**：靠 `Equatable` 视图 + `@State`/`@Binding` 精确依赖追踪，框架自动跳过 body 未变化的视图——相当于"编译器/运行时自动 memo"，开发者很少手动干预。
  - **Jetpack Compose**：`@Stable`/`@Immutable` 标注 + 智能重组（recomposition）跳过参数未变的 composable，与 `React.memo` 浅比较高度神似；Compose 还有"重组计数器"工具直接看哪个 composable 重组过多。
  - 共通法则：**所有声明式 UI 都在做同一件事——identity/输入不变就跳过重算**。差别只在"手动标注（React 手写 memo、Compose 标 `@Stable`）"还是"框架自动推断（SwiftUI、React Compiler）"。本次手写 memo 正是这条法则在 React 19（未开 Compiler）下的落地。
