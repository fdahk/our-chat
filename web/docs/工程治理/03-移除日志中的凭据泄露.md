# 03 · 移除前端日志中的凭据 / 密码 / 用户数据泄露

> 类别：安全（敏感信息泄露 · sensitive data exposure）
> 涉及文件：`src/views/loginView/index.tsx`、`src/views/registerView/index.tsx`、`src/utils/requireAuth.tsx`

## 一、问题概述

登录 / 注册链路里有多处把敏感数据打进了 `console`，其中几处**直接泄露明文密码和 JWT**：

| 位置 | 代码 | 泄露内容 |
|------|------|---------|
| `loginView` 成功分支 | `console.log('login result', loginData)` | `loginData` 含 **JWT token** |
| `loginView` catch | `console.log(error)` | axios error 的 `error.config.data` 是**登录请求体 → 含明文密码** |
| `loginView` onFinishFailed | `console.log('登录失败:', errorInfo)` | antd `errorInfo.values` 含表单字段 → **明文密码** |
| `registerView` catch | `console.error('注册错误:', error)` | 同上，注册请求体含**明文密码** |
| `registerView` onFinishFailed | `console.log('注册失败:', errorInfo)` | 含**明文密码** |
| `requireAuth` | `console.log('token有效', getTokenRemainingTime(...))` | token 有效期等鉴权信息 |

## 二、为什么这是真问题，而不是"无所谓的调试输出"

直觉上"`console.log` 又不会被别的网页读到，能有什么事？"。但凭据进 console 的危害是**通过旁路**实现的，而且都很现实：

1. **第三方监控 / 会话回放 SaaS 会抓 console**。Sentry（breadcrumbs）、LogRocket、FullStory、Datadog RUM 等普遍默认捕获 `console.*` 输出并上传到它们的服务器。一旦接入，**明文密码和 token 就被持久化到第三方日志里**，这是最严重的泄露面——团队往往意识不到。
2. **浏览器扩展**。装了权限的扩展能读取页面 console。用户机器上的恶意/低质扩展可借此收集凭据。
3. **共享屏幕 / 截图 / 录屏**。演示、远程协助、录教程时控制台一开，token/密码直接出镜。
4. **token 即身份**。JWT 是 bearer token——**谁拿到谁就是你**，无需密码。把它打到 console 等于把"通行证"摊在桌面上。

## 三、概念扫盲：两个最容易忽视的泄露源

### 1）antd `onFinishFailed(errorInfo)` 里藏着密码

很多人以为校验失败回调只有"哪个字段错了"。实际上 antd 的 `errorInfo` 形如：

```ts
{ values: { username, password, ... }, errorFields: [...], outOfDate }
```

`values` 是**整张表单的当前值**，包含 `password`。所以 `console.log('登录失败:', errorInfo)` 等于打印明文密码。

### 2）axios `error` 对象里也藏着密码

`catch (error) { console.log(error) }` 看似只打了个错误，但 axios 的 error 携带 `error.config`，而 `config.data` 是**本次请求的请求体**。登录/注册的请求体就是 `{ username, password }`——明文密码再次入镜。`error.config.headers` 还可能含 `Authorization: Bearer <token>`。

> 教训：**不要整体打印 framework 的 error / errorInfo 对象**，它们的"上下文"字段经常顺带把敏感输入也带出来。要打就只打你确认安全的字段（如 `error.message`）。

## 四、修复

逐处删除或降敏：

- 删掉打印 `loginData`、`errorInfo` 的日志；
- `catch (error) { console.log(error) }` → `catch { ... }`（用 ES2019 的 **optional catch binding**，连绑定都不要，既消除泄露又消除 `no-unused-vars`）；
- 错误本就由响应拦截器统一 `message.error` 给了用户提示，这些 console 没有产品价值，直接删。

修复后这些链路不再向 console 写入任何密码 / token / 表单值。

## 五、取舍：源头删除 vs 构建期统一 drop

报告 05 会在生产构建用 esbuild `drop: ['console']` 把**所有** console 抹掉。既然如此，这里为何还要逐处删？因为两者层次不同，是**纵深防御（defense in depth）**：

| 手段 | 作用范围 | 局限 |
|------|---------|------|
| 构建期 `drop console`（报告 05） | 只在**生产产物**生效 | 开发 / 预发环境仍会打印；若第三方监控在 staging 也接了，照样泄露；依赖"别忘了配 drop" |
| 源头删除敏感日志（本报告） | 所有环境 | 治本——**敏感数据压根不写进日志语句** |

显式地把密码/token 写进日志，本身就是代码异味，不该靠一个构建开关来"兜底擦屁股"。所以：**敏感日志源头删，普通调试日志靠构建 drop**——两层都要。

## 六、业界做法

- **日志脱敏（log redaction）**：服务端常用 pino/winston 的 redact 配置自动把 `password`、`authorization`、`token` 等字段打码。前端虽无统一日志层，但同样原则——绝不原样输出凭据。
- **lint 拦截**：`no-console` 规则（或只允许 `console.warn/error`）从 CI 层面阻止裸 `console.log` 流入。
- **OWASP**：归类于 *A09:2021 – Security Logging and Monitoring Failures* / *Sensitive Data Exposure*，明确要求日志不得记录凭据。

## 七、附带修复

`registerView` 原有两处 `catch (error)` 中 `error` 未使用，触发 `@typescript-eslint/no-unused-vars`（main 上 `pnpm lint` 本就是红的）。本次改成 optional catch binding `catch {}` 一并修复，使 lint 恢复绿色。

## 八、验证

- `npx eslint` 对三个改动文件无报错；`pnpm build` 通过。
- 手动走登录失败 / 网络错误路径，控制台不再出现密码、token、表单值。
