# 01 · 把生产暴露的测试页改造为「开发专属调试系统」

> 类别：安全（攻击面收敛）/ 工程卫生
> 涉及文件：`src/router/index.tsx`、`src/views/debug/`（新增）、`src/views/test/`（移除）

## 一、问题概述

路由表里挂着一个开发期调试用的文件上传演示页：

```tsx
import TestView from "@/views/test/index.tsx";
// ...
{ path: "test", element: <TestView/> }
```

`TestView`（`views/test/index.tsx`）是个 `UploadDemo`，里面放了三套 `FileUploader`（图片压缩、**1GB 大文件分片**、通用上传），只在开发期手测上传组件用。但它被原样编译进**生产包**，并通过 `/test` 对所有登录用户暴露。

## 二、为什么这是问题

1. **攻击面（attack surface）扩大**。这个页面允许上传最大 1GB 的任意类型文件且 `multiple`。即便后端有校验，前端也凭空多了一个探测上传接口、构造恶意分片、打满存储的入口。它不在产品功能里，却 100% 可达。
2. **死代码进包**。`UploadDemo` 连同 `FileUploader`、`antd` 的 `Card/Row/Col` 一起进了主 bundle（当前首屏是 1.34MB 单包，见报告 06），纯负担。
3. **信息泄露**。调试页常带 `console.log`（本页就有 `console.log('上传成功的文件:', files)`），暴露内部数据结构、字段命名、接口契约，方便攻击者做信息收集（reconnaissance）。

## 三、关键纠偏：不要删，要让它「生产不可达」

最初的处理是直接删掉。但调试工具有长期价值，删了等于丢工具——**正确目标是「生产环境不可达」，而不是「代码消失」**。于是改造成一个开发专属的调试子系统。

那"只有开发人员可达"该怎么实现？这里有两条根本不同的路线，必须分清：

| 机制 | 原理 | 能否被绕过 |
|------|------|-----------|
| **运行时鉴权** | 在浏览器里 `if (user.isAdmin) renderDebug()` | ❌ 能。前端代码 100% 在攻击者手里，改内存/断点/直接调接口即可绕过。运行时判断只能改变"显示"，改变不了"代码已下发" |
| **编译期排除（采用）** | `import.meta.env.DEV` 门控，生产构建里该分支被静态消除 | ✅ 不能。代码**根本没被打进生产产物**，无从绕过——没有就是没有 |

**对于纯前端，"编译期排除"才是真正的"开发人员专属"**。运行时鉴权适用于"功能要上线、但按角色显隐"的场景，不适用于"这段代码就不该出现在生产"的场景。

## 四、概念扫盲：`import.meta.env.DEV` 为什么能做到编译期排除

`import.meta` 是 ES Module 的元信息对象。Vite 在其上注入了一组环境变量，其中：

- `import.meta.env.DEV`：开发服务器（`vite`）下为 `true`，生产构建（`vite build`）下为 `false`。
- 它不是普通运行时变量，而是 Vite 在**转译阶段**用 `define` 做的**字面量替换**：源码里的 `import.meta.env.DEV` 在生产构建中被直接替换成文本 `false`。

于是这段：

```tsx
const devOnlyRoutes = import.meta.env.DEV ? [ { path: "debug", lazy: () => import("@/views/debug"), ... } ] : [];
```

在生产构建里变成：

```tsx
const devOnlyRoutes = false ? [ /* ... import("@/views/debug") ... */ ] : [];
```

**Rollup 的 tree-shaking** 看到这是恒为 `false` 的三元分支，会把整个 `true` 分支连同里面的 `import("@/views/debug")` **动态导入一起删除**。动态 import 被删 → 它指向的 `views/debug/*` 模块图没有任何入口引用 → 整个调试子系统不会被 emit 成任何 chunk。

> 验证：`pnpm build` 后 `grep -rl "开发调试中心\|UploadDemo\|debugTools" dist/assets` 无任何命中，也没有多出独立的 debug chunk。而 `vite dev` 下 `/debug` 正常可达。

这就是"编译期排除"——比注释掉、比加 if 判断都彻底。

## 五、新调试系统的结构

```
src/views/debug/
  index.tsx       # DebugLayout：调试中心外壳（顶部 DEV 标识条 + <Suspense><Outlet/></Suspense>）
  home.tsx        # DebugHome：工具列表首页，从 tools.ts 渲染
  tools.ts        # 工具登记表（纯元数据，不引用组件）
  uploadDemo.tsx  # 由 views/test 迁移而来的上传演示页
```

路由（`router/index.tsx`）：

```tsx
const lazyComponent = (loader) => async () => ({ Component: (await loader()).default });

const devOnlyRoutes = import.meta.env.DEV
  ? [{
      path: "debug",
      lazy: lazyComponent(() => import("@/views/debug")),
      children: [
        { index: true, lazy: lazyComponent(() => import("@/views/debug/home")) },
        { path: "upload", lazy: lazyComponent(() => import("@/views/debug/uploadDemo")) },
      ],
    }]
  : [];
// createBrowserRouter([ ...业务路由, ...devOnlyRoutes ])
```

几个设计点：

- **用 React Router v7 的路由级 `lazy` 属性**（不是 `React.lazy`）。`lazy` 是数据路由（data router）原生的代码分割入口：返回 `{ Component }`，由路由器接管加载，无需在外层手搓 `Suspense` 来兜整页。`Suspense` 只在 `DebugLayout` 内包住 `<Outlet/>`，用于切换子工具时的过渡。
- **`/debug` 放在顶层、不在 `RequireAuth` 下**：调试工具应能脱离登录态使用。反正生产里压根没有它，不存在"未登录也能进"的生产风险。
- **`tools.ts` 只存元数据、不 `import` 组件**：首页列表渲染不需要把各工具组件拉进同一个图，保持加载边界清晰。
- **可扩展**：加新调试工具 = `tools.ts` 加一条 + 路由加一条 lazy 子路由 + 建页面文件，三步成系统。

## 六、业界做法对照

- **条件编译**：`import.meta.env.DEV` / `process.env.NODE_ENV !== 'production'` 包裹调试入口做 DCE，是 Vite/webpack 生态的通用手法。
- **独立 playground / Storybook / Ladle**：把组件演示彻底移出主应用，连开发主包都不进。本项目规模小，内建 `/debug` 子系统已足够，无需为一个 demo 引入额外工具链（避免过度工程）。
- **后端 feature flag + 鉴权**：针对"要上线但灰度/限角色"的功能，那是运行时问题；与"调试页不该进生产"是两回事，别混用。

## 七、踩坑提示

- **`React.lazy` vs 路由 `lazy`**：用 `React.lazy(() => import(...))` 也能分割，但需要自己布 `Suspense`，且和数据路由的 `loader`/错误边界整合不如路由级 `lazy` 顺。v7 优先用路由 `lazy`。
- **类型导入**：`router/index.tsx` 没引入 `React`，给 `lazyComponent` 标注时要 `import type { ComponentType } from "react"`，不要写 `React.ComponentType`（全局 `React` 命名空间在新 JSX 转换下未必存在）。
- **别用运行时变量冒充编译期常量**：只有被 Vite `define` 成字面量的 `import.meta.env.DEV` 才能触发 DCE。若改成读 `localStorage`、读普通变量做判断，分支不会被消除，调试代码照样进生产包。
- **迁移后同步清依赖**：删 `views/test` 前 `grep` 确认无其它引用；`uploadDemo.tsx` 的相对路径统一换成 `@` 别名。

## 八、验证

- `vite build` 通过；`grep -rl "开发调试中心\|UploadDemo\|debugTools" dist/assets` 无命中，无独立 debug chunk → 生产产物不含调试系统。
- `vite dev` 下访问 `/debug` 可见工具列表，`/debug/upload` 可达上传演示页。
