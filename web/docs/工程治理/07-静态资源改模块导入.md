# 07 · 硬编码 `src/assets/*` 路径改为 ES 模块导入（修复生产 404）

> 类别：正确性 / 性能（资源加载）
> 涉及文件：`src/assets/images/index.ts`（新增）、`layout`、`chatView`、`settingView`、`directoryView`

## 一、问题概述

多处把图片当成**字符串字面量路径**直接塞进 `src`：

```tsx
<img src={user.avatar ? buildServerUrl(user.avatar) : 'src/assets/images/defaultAvatar.jpg'} />
// directoryView 里还有 'src/assets/images/searchUser.png'、'src/assets/images/newFriend.png'
```

`'src/assets/images/defaultAvatar.jpg'` 是一个**相对路径字符串**，浏览器会基于"当前页面 URL"去解析它。这条路径在生产环境根本取不到——而且在开发环境的子路由下同样取不到。

## 二、为什么它会 404：浏览器怎么解析这个 src

`src` 的值没有前导 `/`，是**相对路径**。浏览器拿当前文档 URL 作 base 去拼：

| 当前页面 | `src='src/assets/images/defaultAvatar.jpg'` 实际请求 | 结果 |
|----------|------|------|
| `http://host/chat` | `http://host/chat/src/assets/...`（`/chat` 当成"目录"） | 404 |
| `http://host/directory` | `http://host/directory/src/assets/...` | 404 |
| 生产 `http://host/` | `http://host/src/assets/...` | 404——**生产产物里根本没有 `src/` 目录** |

核心误区：**`src/...` 是源码目录，不是 Web 可访问路径**。开发时偶尔"看起来能显示"，往往只是恰好在根路由、或浏览器缓存所致，换个路由 / 上生产立刻原形毕露。这些默认头像/图标其实一直是坏的。

## 三、Vite 里资源的正确打开方式：当成模块 import

Vite（以及 webpack）把静态资源视为**模块**。你 `import` 它，构建工具负责：

1. 把它**纳入依赖图**，参与打包；
2. 产出时给文件名加 **content hash**（如 `defaultAvatar-bJNzkOyf.jpg`），实现长效缓存 + 防缓存击穿；
3. `import` 的返回值是**构建后那个真实可访问的 URL**（开发是 `/src/...`，生产是 `/assets/xxx-hash.jpg`），由 Vite 保证两端都对。

修复——建一个集中导出模块，再在各处引用变量：

```ts
// src/assets/images/index.ts
import defaultAvatar from './defaultAvatar.jpg';
import searchUserIcon from './searchUser.png';
import newFriendIcon from './newFriend.png';
export { defaultAvatar, searchUserIcon, newFriendIcon };
```

```tsx
import { defaultAvatar } from '@/assets/images';
// ...
<img src={user.avatar ? buildServerUrl(user.avatar) : defaultAvatar} />
```

`defaultAvatar` 是个变量（运行时是带 hash 的 URL 字符串），不再是写死的相对路径。

> 类型支持：`vite-env.d.ts` 里的 `/// <reference types="vite/client" />` 已为 `*.jpg/*.png` 等提供模块声明，`import img from './x.png'` 才有类型。

## 四、一个值得注意的现象：小图被内联、大图被产出

构建后 `dist/assets/` 里只看到 `defaultAvatar-<hash>.jpg`，**没有** `searchUser.png` / `newFriend.png`。不是丢了，而是 Vite 的 **`build.assetsInlineLimit`（默认 4KB）** 机制：

- 小于阈值的资源 → 直接**内联成 base64 data URI** 塞进 JS，省一次 HTTP 请求；
- 大于阈值的 → 产出为带 hash 的独立文件，可被浏览器单独缓存。

两个 PNG 图标小于 4KB 被内联，`defaultAvatar.jpg` 较大被产出。**两种都是正确处理**，且是按"小文件省请求、大文件可缓存"自动权衡的结果——这正是字符串硬编码永远拿不到的好处。

## 五、取舍：集中导出 vs 各文件就地 import vs public 目录

| 方案 | 做法 | 取舍 |
|------|------|------|
| 集中模块导出（采用） | `assets/images/index.ts` 统一 import 再 re-export | 资源引用单点收口,改路径只改一处;`defaultAvatar` 复用多处天然适合 |
| 各文件就地 import | 每个组件各写 `import x from '.../x.png'` | 也对,但同一张默认头像被五个文件各 import 一遍,分散 |
| 放 `public/` 用绝对路径 `/xxx.jpg` | 不经构建,原样拷贝 | 适合 favicon、robots.txt 等"必须固定名、不参与打包"的文件;但**不带 hash → 不利缓存失效**,且不参与依赖优化。默认头像这类资源不该走 public |

选集中导出：默认头像在多个视图复用，收口到一处最省心；图标参与打包也能享受内联/hash 优化。

> 注意 `searchUser` 这个名字在 `directoryView` 里已被 `friendApi` 的同名函数占用，故图片导出取名 `searchUserIcon` / `newFriendIcon` 避免冲突。

## 六、业界做法

- **资源即模块**是 Vite/webpack 的标准范式；React 生态默认如此。
- `public/` 仅留给"需固定 URL、不参与构建"的少数文件。
- 大型项目会再配 CDN 域名（`build.assetsDir` + `base`），import 出来的 URL 自动带上 CDN 前缀——这套机制的前提正是"资源走 import 而非硬编码"。

## 七、验证

- `grep -rn "'src/assets" src` 图片路径全部清零（仅剩提示音，见报告 08）。
- `pnpm build` 通过；`dist/assets/` 产出 `defaultAvatar-<hash>.jpg`，小图标内联进 JS。
- 各路由下默认头像、搜索/新朋友图标正常显示，不再 404。
