# 12 · JWT 密钥改从环境变量读取（移除硬编码）

> 类别：安全（密钥管理）
> 涉及文件：`server/src/config/config.js`

## 一、漏洞本体：密钥硬编码进源码

改造前 JWT 签名密钥是写死在源码里的弱字符串（形如 `'114514'`）。JWT 的全部安全性都系于这个密钥：

> **谁知道密钥，谁就能签发任意合法 token，冒充任意用户、伪造任意权限——而且服务端验签会完全通过，无法察觉。**

硬编码的后果是多重的：

1. **密钥随源码进 Git 历史**。一旦仓库泄露（开源、离职拷走、CI 日志、第三方代码扫描），密钥即暴露，且**历史里永久存在**，改了也能从旧 commit 翻出来。
2. **弱值可被猜解/爆破**。`114514` 这种值连字典都不用查。
3. **所有环境共用一个密钥**。开发、测试、生产同密钥，任一环境泄露即全线沦陷。
4. **无法轮换**。改密钥要改代码、重新部署，运维无法独立操作。

## 二、概念扫盲

### 2.1 JWT 的密钥在签什么
本项目用 HMAC（`jwt.sign(payload, secret)`，默认 HS256）：用密钥对 `header.payload` 做 HMAC 得到 signature。验签是用**同一个密钥**重算比对。所以这是**对称密钥**——签发和验证同一把钥匙，密钥即"上帝权限"。（对比非对称的 RS256：私钥签、公钥验，公钥可公开；但本项目单体应用用 HMAC 足矣，无需引入密钥对管理。）

### 2.2 为什么用环境变量
**十二要素应用（12-Factor App）原则之三：配置应与代码分离，存于环境。** 密钥、数据库口令这类敏感配置不该进代码库，而应在运行环境注入（`.env` 文件 + 进程环境变量，`.env` 加进 `.gitignore`）。好处：同一份代码在不同环境用不同密钥；密钥不进版本库；运维可独立轮换。

## 三、实现

```js
function resolveJwtSecret() {
  const fromEnv = process.env.JWT_SECRET;
  if (fromEnv && fromEnv.trim()) return fromEnv;        // 正常路径：从环境读
  if (isProduction) {
    throw new Error('缺少环境变量 JWT_SECRET：生产环境必须显式配置 JWT 密钥');  // 生产 fail-fast
  }
  const generated = crypto.randomBytes(32).toString('hex');  // 开发兜底：随机生成
  console.warn('[config] 未设置 JWT_SECRET，已为开发环境生成临时随机密钥；进程重启后已签发的 token 将失效。');
  return generated;
}
export const config = {
  jwtSecret: resolveJwtSecret(),
  jwtExpiresIn: process.env.JWT_EXPIRES_IN || '7d',
};
```

三条分支体现了刻意的取舍：

- **有环境变量** → 直接用。这是部署的正常路径。
- **生产环境缺失** → **fail-fast 抛错，拒绝启动**。这是关键决策：宁可启动失败被立刻发现，也绝不用一个弱默认值偷偷上线（那等于把硬编码漏洞换个地方留着）。
- **开发环境缺失** → 用 `crypto.randomBytes(32)` 生成一次性强随机密钥并告警。开发者无需配 `.env` 也能跑，但进程重启后旧 token 失效（因为密钥变了）——对本地开发完全可接受，且这个提示本身在教育开发者"该配 JWT_SECRET 了"。

`randomBytes(32)`（256 bit）远超 HS256 所需的最小熵，杜绝弱密钥。

## 四、取舍

| 缺失密钥时的策略 | 后果 | 评价 |
|------|------|------|
| 硬编码弱默认值（旧） | 弱密钥静默上线，可伪造 token | ❌ 漏洞本体 |
| 生产 fail-fast（本次） | 没配就启动失败，强制显式配置 | ✅ 错误前置，不留隐患 |
| 生产也随机生成 | 每次重启所有用户掉线；多实例间密钥不一致，token 互不认 | ❌ 生产不可用 |
| 开发也强制配置 | 安全但开发体验差，新人 clone 即跑不起来 | △ 过严 |

本方案在"生产绝对安全"和"开发零配置可跑"间取了最优点。

## 五、踩坑与注意

1. **生产多实例必须共享同一密钥**。若水平扩展多个后端实例却各自随机生成，A 签的 token 到 B 验不过——所以生产绝不能走随机分支，必须统一注入。这正是"生产 fail-fast"而非"生产随机"的根本原因。
2. **`.env` 必须在 `.gitignore` 里**，否则等于换个文件继续硬编码。应提供 `.env.example`（不含真值）说明需要哪些变量。
3. **轮换密钥会使所有存量 token 失效**（全员重新登录）。需平滑轮换时，可支持"新密钥签发、新旧密钥都接受验签"的过渡期——本项目暂未到这一步，留作后续。
4. **历史泄露不可逆**：旧的硬编码密钥已进 Git 历史，本次改代码无法抹除历史。生产部署时**务必用一个全新的强密钥**，绝不能复用曾出现在源码里的那个旧值。

## 六、业界对比与 Web vs Native

- **业界**：密钥/凭据管理是 OWASP、12-Factor 的基本要求。成熟项目进一步用专门的密钥管理服务（AWS Secrets Manager、HashiCorp Vault、K8s Secrets）而非裸 `.env`，支持加密存储、访问审计、自动轮换。环境变量是这条路上的最低门槛、也是本项目体量下足够的一档。
- **Web vs Native**：本条是**纯后端服务端**问题，与前端 Web/Native 无关——密钥只存在于服务端，前端任何形态都拿不到也不该拿到。反过来说，这也提示一个原则：**任何需要保密的密钥都不能放进前端**（无论 Web bundle 还是 App 包，都能被逆向提取）。前端能安全持有的只有"公开即无害"的东西（如公钥、可读的 csrfToken）。
