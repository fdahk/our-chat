# 14 · 输入框草稿态隔离——避免每次按键重渲染整个聊天视图

> 类别：性能（渲染 / 重渲染范围控制）
> 涉及文件：`web/src/views/chatView/MessageInput.tsx`（新增）、`web/src/views/chatView/index.tsx`

## 一、问题本体

`ChatView` 是一个巨型组件，它同时持有：

- **高频变化的草稿态** `const [input, setInput] = useState('')`——用户**每敲一个字符**就 `setInput` 一次；
- **两个重列表**——左侧会话列表（`globalConversations.map(<DisplayItem/>)`）和右侧消息列表（antd `<List dataSource={messages}>`）。

React 的规则是：**一个组件的 state 变化，会触发该组件函数体重新执行，并默认重渲染它返回的整棵子树。** 于是这条链每次按键都跑一遍：

```
敲一个字 → setInput → ChatView 重新执行
        → 会话列表所有 DisplayItem 重渲染
        → 消息列表所有 List.Item + renderMessageContent 重渲染
        → renderMessageContent 内的 formatFileSize 等闭包全部重建
```

且**全代码库零 `React.memo`**（已确认），`DisplayItem` 没有任何记忆化，挡不住这次重渲染。结果是：**会话越多、聊天记录越长，打字越卡**——输入延迟随历史消息数量线性增长。这是 SPA 里最经典、也最隐蔽的性能问题之一：功能完全正确，只有在数据量上来后才显现卡顿。

## 二、概念扫盲：React 的重渲染模型

### 2.1 "重渲染"到底重渲染了什么
React 组件的"渲染"= 执行一次组件函数，得到新的虚拟 DOM（VDOM）。**渲染 ≠ 操作真实 DOM**。渲染后 React 做 **reconciliation（协调/diff）**：把新 VDOM 和上一次的比对，只把**真正变化的部分**写进真实 DOM。

所以"每次按键重渲染整个列表"的代价主要不在真实 DOM（diff 后大多无变化、不写 DOM），而在：

- **JS 执行成本**：成百上千个列表项组件函数被重新调用，重新创建 VDOM 对象、重新执行内部闭包（如 `formatFileSize`、`buildServerUrl`）；
- **diff 成本**：React 要逐项比对这成百上千个节点，即便结论是"没变"，比对本身也要花时间；
- 二者叠加，在主线程同步发生，**阻塞输入响应**，表现为打字掉帧。

### 2.2 触发重渲染的根因：state 的"位置"
一个组件重渲染的触发源只有几种：自身 `state` 变化、父组件重渲染、`context` 变化。**这里的根因是 state 放错了位置**——把"只影响输入框一小块"的草稿态，放在了"统辖两个大列表"的顶层组件里。state 的影响半径 = 它所在组件的整棵子树。草稿态本该只影响输入框，却被放到能波及全列表的高处。

### 2.3 解法：状态下沉 / 状态就近（state colocation）
React 官方性能指南的第一条优化原则不是 `memo`，而是 **"把 state 移到尽可能低的、真正需要它的地方"（colocate state）**。把 `input` 连同它的 `onChange`、键盘处理一起，搬进一个独立的 `MessageInput` 组件，让它**自己持有自己的草稿态**。这样：

```
敲一个字 → MessageInput 内部 setInput → 只有 MessageInput 重渲染
        → ChatView 不重渲染 → 两个列表纹丝不动
```

重渲染范围从"整个聊天视图"缩小到"一个输入框"，与消息条数彻底解耦。

## 三、实现

### 3.1 抽出 `MessageInput`
新组件自持草稿态，对外只暴露两个回调：

```tsx
interface MessageInputProps {
  onSend: (text: string) => void;          // 发送：把文本交给上层
  onHeaderIconClick: (method: string) => void; // 表情/文件/语音等图标点击
}
function MessageInput({ onSend, onHeaderIconClick }: MessageInputProps) {
  const [input, setInput] = useState('');
  const send = () => { if (!input.trim()) return; onSend(input); setInput(''); };
  // ...textarea / 图标 / 发送按钮
}
```
草稿态、`onChange`、`onKeyDown`（Enter 发送 / Shift+Enter 换行）、图标配置数组全部内聚到这里。

### 3.2 `ChatView` 瘦身
- 删除 `input`/`setInput` 状态、`handleKeyDown`、`inputAreaIcons` 配置、以及整段输入区 JSX；
- `sendMessage` 改为**接收文本参数**而非读取自身 state：`sendMessage(text)`——它内部 `socket.emit` 后**不再 `setState`**，所以发送也不会触发 ChatView 重渲染（清空草稿由子组件自己做）；
- 用一行 `<MessageInput onSend={sendMessage} onHeaderIconClick={handleClickHeaderIcon} />` 取代原本几十行输入区；
- 顺带移除不再使用的 antd `Input`、`Button` 引入。

### 3.3 为什么发送也不会回弹重渲染
关键细节：`onSend`（即 `sendMessage`）在 ChatView 里**不调用任何 `setState`**——它只 `socket.emit`。消息列表的更新是后端经 socket `receiveMessage` 回推、写入 redux 后才发生的（那是"真有新消息"的合理重渲染）。所以**正常打字与发送的整条交互链，都不会无谓地重渲染列表**；列表只在"真的来了新消息"时才更新。

## 四、各方案取舍

| 方案 | 思路 | 取舍 |
|------|------|------|
| **状态下沉（本次）** | 把草稿态搬进 MessageInput，缩小影响半径 | ✅ 根因修复、零额外 API、代码更清晰；首选 |
| `React.memo` 包列表项 | 让 DisplayItem/消息项在 props 不变时跳过重渲染 | 治标：ChatView 仍重渲染，靠 memo 在子层"刹住"；需保证 props 引用稳定（要配 `useCallback`/`useMemo`），心智负担大、易因一个不稳定 prop 失效 |
| `useMemo` 缓存列表 VDOM | 缓存 `messages.map(...)` 结果 | 同样治标，且依赖数组易写错 |
| 列表虚拟化（react-window / Virtuoso） | 只渲染视口内的消息 | 解决的是"**单次**渲染上千条很贵"，与本问题（**频繁**触发重渲染）正交；消息真的成千上万时该上，但不是本问题的解 |

**为什么不直接上 `React.memo`？** memo 是"在子组件门口设卡，props 没变就放行不重渲染"。但它要求传入的 props **引用稳定**——这里若给 memo 化的 MessageInput 传 `onSend`，而 `onSend` 每次 ChatView 渲染都是新函数，memo 就失效。memo 是一套需要 `useCallback` 配套、容易出错的"补救层"；而状态下沉是从源头让 ChatView 压根不因打字而重渲染，**根因 > 补救**。这也呼应 React 官方"先 colocate，再考虑 memo"的次序。

> 注：本次没有顺手给 `DisplayItem` 加 `memo`，那是另一个独立问题（"新消息到达时会话列表是否需要整列重渲染"），应单独评估、单独提交，不与本次混在一起。

## 五、踩坑记录

1. **受控组件的 state 必须和输入框同生共死地"贴近"它**。把受控 `value` 的 state 放在高层组件，是"输入卡顿"最常见的成因。经验法则：**谁的 `onChange` 最高频，谁的 state 就该放到最低层。**
2. **下沉后要切断对该 state 的外部依赖**。`sendMessage` 原本读 `input`，下沉后必须改成"接收参数"，否则父组件还要持有 input，等于没下沉。本次把"清空草稿"也交给子组件（`setInput('')`），父组件彻底不碰草稿态。
3. **别让发送回调里夹带 `setState`**。如果 `onSend` 在父组件里 `setState`，发送一次又会把父组件连同列表重渲染一遍，前功尽弃。本次 `sendMessage` 只 `emit`、不 setState。
4. **组件位置稳定，子组件 state 才会被保留**。`<MessageInput>` 始终挂在同一位置，即便 ChatView 因新消息重渲染，React 也按位置复用同一个 MessageInput 实例、保留其草稿——用户正在输入的内容不会因为来了条新消息而丢失。这正是 React"按位置保持 state"特性带来的免费红利。
5. **`autoSize`、样式类名要原样迁移**。输入区依赖 `chatViewStyle` 的若干类名，抽组件时连同 `style.module.scss` 的引用一起带走，避免样式丢失。

## 六、业界对比与 Web vs Native

- **业界**：React 官方文档《You Might Not Need an Effect》《Render and Commit》及性能指南都把"**state colocation / 状态下沉**"列为重渲染优化的第一手段，先于 `memo`/`useMemo`。"把受控输入拆成独立组件以隔离高频更新"是社区公认的标准做法（尤其表单库如 React Hook Form 的核心卖点之一，就是用非受控/订阅式更新把输入态从父树里摘出去）。
- **Web vs Native（重渲染模型差异）**：
  - **React DOM**：重渲染要经 VDOM diff 再落到浏览器 DOM；DOM 操作昂贵，所以 React 用 diff 尽量少碰 DOM——但"重新执行组件函数 + diff"本身的 JS 成本依旧存在，这正是本问题的来源。
  - **React Native**：同样的 VDOM/reconciler，但提交端不是 DOM 而是原生视图（UIView/Android View）经 bridge/JSI 更新；"频繁重渲染导致 JS 线程繁忙"的问题**一模一样**——状态下沉的优化在 RN 里同等有效。
  - **原生开发（SwiftUI / Jetpack Compose）**：它们也是声明式 + diff 模型（Compose 的 recomposition、SwiftUI 的 body 重算），同样讲究"缩小重算范围"（Compose 的 `@Stable`/`remember`、跳过未变化的 composable）。可见"**把高频变化的状态限制在最小作用域**"是所有声明式 UI 框架的共通性能法则，不止 Web。
  - 差异在于：Web 这边因为 DOM 昂贵，diff 的"省 DOM"收益更突出；但"省掉不必要的组件重算"这一层，各框架是相通的。
