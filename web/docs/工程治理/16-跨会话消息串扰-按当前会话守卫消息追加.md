# 16 · 跨会话消息串扰——按当前会话守卫消息追加


## 一、问题本体

`chatStore` 的 `globalMessages` 是一个**扁平消息数组**，语义是"当前打开会话的消息列表"：

```ts
globalMessages: Message[]            // 初值 []
initGlobalMessages(state, a) { state.globalMessages = a.payload; }  // 点开会话时整体替换为该会话历史
addGlobalMessage(state, a)  { state.globalMessages.push(a.payload); } // ← 问题在这
```

`ChatView` 右侧消息列表直接 `dataSource={messages}`（即 `globalMessages`）渲染。点开会话 A 时，`handleClickConversation` 调 `getConversationMessages('A')` 并 `initGlobalMessages(A 的消息)` —— 此刻数组里装的是 A 的历史，显示正确。

但 socket 收到新消息时，全局监听器无条件 `dispatch(addGlobalMessage(msg))`，而原 `addGlobalMessage` **不判断消息属于哪个会话，一律 `push`**：

```
正在看会话 A
  → 好友在会话 B 给你发消息
  → socket receiveMessage → addGlobalMessage(B 的消息)
  → B 的消息被 push 进当前数组（装着 A 的那个）
  → B 的消息错误地显示在 A 的聊天窗口里
```

**这是数据隔离缺失导致的串扰 bug**：消息会出现在错误的会话窗口。注意按会话 key 的正确写法其实就在源码里、但被注释掉了（`// state.globalMessages[conversationId].push(...)`）——这是一个**半成品重构**：作者本想把 `globalMessages` 设计成 `{ [conversationId]: Message[] }`，但最终落地成了扁平数组，`addGlobalMessage` 的按会话归类逻辑也随之失效。

## 二、概念扫盲：状态的"身份"与"作用域"

### 2.1 单一数据源下的"视图态"陷阱
Redux 单一 store 里，一份状态常常身兼两职：既是"全量数据仓库"，又是"当前视图要显示什么"。`globalMessages` 就掉进了这个陷阱——它名字像"全局所有消息"，实际职责是"当前会话的消息"。一旦写入方（`addGlobalMessage`）按"全局"理解、读取方（ChatView）按"当前会话"理解，**写读语义不一致**，串扰就发生了。

### 2.2 两种正解
1. **按会话 key 存**（作者原始意图）：`globalMessages: Record<conversationId, Message[]>`，读取时 `globalMessages[activeConversation]`。彻底隔离，但要改 store 形状 + ChatView 选择器 + 滚动 effect + `initGlobalMessages` 签名，改动面大，且本环境无后端难做端到端验证。
2. **写入时按当前会话守卫**（本次）：保持扁平数组语义不变，只在 `addGlobalMessage` 入口判断"这条消息属于当前打开的会话吗"，不属于就不进数组。改动最小、风险最低、用户可见行为立即正确。

`activeConversation` 恰好与 `globalMessages` 同在 `chat` slice（`state.activeConversation`），reducer 内可直接读，守卫天然可行。

## 三、实现

```ts
addGlobalMessage(state, action: PayloadAction<Message>) {
  // globalMessages 是「当前打开会话」的消息数组；只有属于当前会话的新消息才追加，
  // 其他会话的消息只更新 lastMessages 预览，不能串进当前窗口。
  if (action.payload.conversationId === state.activeConversation) {
    state.globalMessages.push(action.payload);
  }
}
```

配套不变量校验（均成立，无需改其他文件）：
- **自己发的消息**：`ChatView.sendMessage` 只 `socket.emit`，消息经后端 `receiveMessage` 回推 → `addGlobalMessage`。在 A 里发消息时 `conversationId === A === activeConversation` → 正常显示。✅
- **别的会话来消息**：`conversationId !== activeConversation` → 不进当前数组；`lastMessages` 由监听器单独 `dispatch(addLastMessage)` 更新 → 左侧会话列表的预览仍会刷新。✅
- **切到该会话**：`handleClickConversation` 走 `initGlobalMessages` 从后端拉全量历史整体替换 → 之前没追加也不丢消息。✅
- **没有打开任何会话**（`activeConversation === null`）：任何消息都不追加，符合"无窗口可显示"的预期。✅

## 四、各方案取舍

| 方案 | 改动面 | 取舍 |
|------|--------|------|
| **写入守卫（本次）** | 仅 1 个 reducer | ✅ 最小、零风险、行为立即正确；保持现有数据流，无需后端联调即可确信正确；首选 |
| 按 conversationId key 重构 store | store 形状 + ChatView + effect + 多个 reducer | 架构上更"正"（彻底隔离、切会话不必重拉历史可做缓存），但改动大、需端到端验证；作者原始意图，留作可选的独立重构 |
| 在 ChatView 渲染时 `messages.filter(m => m.conversationId === active)` | 仅渲染层 | 治标：脏数据仍进数组、只是显示时滤掉；每次渲染多一次 filter；不如在源头不写入干净 |

**为什么选写入守卫而非渲染过滤？** 同样能让用户看对，但"源头不写脏数据"比"显示时滤脏数据"更干净：数组始终只含当前会话消息，渲染层无需每次 filter，滚动到底等逻辑也不被异物干扰。**在数据入口拦截 > 在出口补救**——与报告 14/15"缩小影响半径、根因优先"一脉相承。

## 五、踩坑记录

1. **命名误导是 bug 温床**。`globalMessages` 这名字暗示"全局所有消息"，诱导 `addGlobalMessage` 写成"无差别 push"。状态命名应反映**真实作用域**（如 `activeConversationMessages` 更达意）。本次不改名以控制改动面，但用注释钉死语义。
2. **半成品重构要么做完要么回滚**。被注释掉的按会话 key 写法 + 扁平数组实现并存，是典型的"重构做了一半"——两套心智模型打架。发现此类代码应明确收口到一种。
3. **reducer 可以读同 slice 的其他字段**。很多人下意识以为 reducer 只能动自己那块 payload，其实 `state` 是整个 slice，`state.activeConversation` 直接可用——这正是本次能在源头守卫的关键。
4. **守卫条件别误伤自己发的消息**。一开始要确认"本端发出的消息回推时 conversationId 是否等于 activeConversation"，否则会把自己发的也挡掉。已验证：发消息时正处于该会话，相等，放行。

## 六、业界对比与 Web vs Native

- **业界**：聊天类应用普遍按会话/房间 key 存消息（`Map<roomId, Message[]>`），并只渲染 active room 的切片——这正是方案 2 的形态（Redux 里常配 `createEntityAdapter` 或 normalizr 做规范化）。本次受限于"半成品扁平数组 + 无后端验证"，采用最小守卫先消除可见 bug，把规范化重构留作独立项，符合"小步、可验证"的工程节奏。
- **消息去重/顺序**：成熟 IM 还会在写入处做幂等去重（按消息 id）与按时间戳排序，避免重连重发导致重复/乱序。本项目 `Message` 暂无稳定 id（见报告 15 §七），去重/稳定 key 是后续独立项。
- **Web vs Native**：
  - **状态隔离是跨端通用问题**。原生 IM（iOS/Android）同样按 `conversationId` 分桶存消息，当前会话只订阅自己那桶——SwiftUI 的 `@Published var messages[conversationId]`、Android 的 `StateFlow<Map<String, List<Message>>>` 都是这个套路。
  - **声明式 UI 的"写读一致性"**：无论 React 的 reducer、Compose 的 ViewModel 还是 SwiftUI 的 ObservableObject，只要"写入方按全局、读取方按当前"语义不一致，就会串扰。**根治都是"在状态层把数据按作用域分桶"**，本次的守卫是该思想的轻量版。

## 七、验证与局限

- `pnpm build`、`pnpm lint` 通过（0 error）。
- **未做运行期端到端验证**：理想验证是起后端 + 两个账号，在会话 A 打开时让会话 B 发消息，确认 B 的消息**不再**出现在 A 窗口、但 B 的列表预览**仍**更新。该联调需完整后端 + 数据库 + 双客户端，本环境不具备；结论基于对 reducer 数据流与 `activeConversation` 不变量的确定性推导。
- **后续独立项**：① 将 `globalMessages` 规范化为按 `conversationId` key（彻底隔离 + 切会话缓存，免重复拉历史）；② 为 `Message` 引入稳定 id 以支持消息去重与列表稳定 key（呼应报告 15 §七）。均为正交的独立改造，另行评估提交。
