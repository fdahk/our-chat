# 移动端(iOS)音视频深度项目规划

> 本文回答:**如何把 our-chat 的 iOS 客户端做成一个"既贴合现有项目(IM + 知识库助手 + 音视频通话)、又能展示客户端音视频深度技术"的项目。**
> 配套:KB `myReact/docs/音视频技术/`(原理)、`项目规划-自研播放器内核.md`(深核蓝本)、本目录 01–05(iOS 技术栈/工程结构/流程)。
> 现状结论基于对 `server/`、`gateway/`、`web/`、`mobile-swift/` 与 agent-server 的代码级勘察(2026-06-22)。

---

## 0. 结论先行(决断)

**做法:iOS 原生客户端,分两层——「基线层」打通现有全部功能(贴合),「深度层」钉自实现音视频深核(展示)。**

- **基线层(贴合现有,调库为主,深度有限)**:登录(Bearer+Keychain)、IM(Socket.io/WS)、知识库助手(消费 agent-server SSE)、**音视频通话(WebRTC iOS,补齐 web 已有的语音 + 升级视频)**。
- **深度层(展示技术,自实现深核,是项目的"讲得透"所在)**,按优先级:
  - **深核 A(首选)= 视频消息 + 自研播放器内核**:在 IM 里加"小视频/视频文件消息",**播放不用 AVPlayer 黑盒,而是自研内核**(自解封装 → VideoToolbox 驱动解码 → Metal 渲染 → A/V 同步 → seek)。直接落地 `项目规划-自研播放器内核.md` 的 M0–M5(iOS 端)。
  - **深核 B(次选)= 视频通话实时美颜/滤镜**:采集帧 → **自写 Metal GPU 管线**(CVPixelBuffer 零拷贝)→ 喂回 WebRTC 编码。落地 KB `20/22`。
  - **深核 C(可选)= 通话音频处理/变声**:实时音频线程 + lock-free ring buffer + 音效。落地 KB `21`。

**为什么这样分**:WebRTC 通话本身在客户端是**黑盒**(调 `libwebrtc`,`RTCPeerConnection` 内部的拥塞控制/JitterBuffer 你改不动),单做通话**讲不出自实现深度**(见 `项目规划-自研播放器内核.md` §0 的论证)。**深度必须落在"你能从字节级亲手写"的环节:解封装/解码驱动/渲染管线/音频实时线程/A/V 同步——而承载它们最自然的功能是"视频消息播放"和"通话美颜",这两个恰好都贴合 our-chat。**

**第一步(M0)**:在 mobile-swift 跑通"登录 + 进会话 + 收发文本消息"(打通 Bearer 鉴权 + Socket.io),再进音视频。详见 §7。

## 1. 现有项目现状盘点(代码级)

### 1.1 服务地图与技术栈

| 服务 | 栈 | 角色 | 移动端关系 |
|---|---|---|---|
| `server/` | Node + Express + Prisma(PostgreSQL) + **Socket.io**(+Redis adapter) | 业务 + **信令中转** + **IdP(签发 JWT)** | iOS 接它的 HTTP + Socket.io |
| `gateway/` | Go 1.22 + gorilla/websocket + JWT + Prometheus | 原生 WebSocket 长连接网关(P5) | iOS 长连接走它(`/ws`) |
| `web/` | React 19 + Redux + antd + socket.io-client | Web 端(音视频通话参照实现) | iOS 对照它移植 |
| `mobile-swift/` | **Swift 6 + SwiftUI + TCA + SPM + Tuist + GRDB** | iOS 端 | **仅工程骨架,无业务代码** |
| agent-server | NestJS + Prisma + **Milvus** + Redis + BullMQ | 知识库 RAG + 自建 ReAct agent | iOS 复用 our-chat token 直连 |

### 1.2 IM 现状(可直接复用)

- 传输:Socket.io(双向 + 应答),消息用 Zod 校验(`server/src/contracts/message.ts`)。
- 上行 `message.send`(带 `clientMsgId` 幂等);下行 `receiveMessage`(带单调 `seq`)。
- 单聊/群聊统一 `conversationId`(群 `group_` 前缀);已读 `read.report`/`read.sync`;离线读扩散 + `GET /user/sync?conv=&since=&limit=` 补拉。
- Presence:Redis + TTL,多设备 `deviceId` 区分。
- **iOS 复用**:协议已定型,iOS 照 `web` 的 socket 客户端实现即可。

### 1.3 音视频通话现状(关键:浅,且有缺口)

- **WebRTC P2P**,**仅语音**(无视频/屏幕共享/多人),**无媒体服务器**。
- **STUN only**(Google 公开 STUN),**未自建 TURN** → 生产环境很多 NAT 下会连不通(**真缺口**)。
- **信令复用 Socket.io**(不独立服务),事件在 `server/src/utils/socket.ts` L270–358:

| 事件 | 含义 |
|---|---|
| `call:start` | 主叫 → 被叫,带 Offer SDP |
| `call:accept` | 被叫 → 主叫,带 Answer SDP |
| `call:ice` | 双向 ICE candidate |
| `call:reject` / `call:end` | 拒接 / 挂断 |

- Web 实现:`web/src/hooks/useVoiceCall.ts`(状态机)+ `web/src/utils/webrtc.ts`(RTCPeerConnection 生命周期、pendingIceCandidates 暂存)。
- **iOS 现状**:**未实现**。要引入 iOS WebRTC SDK(Google `libwebrtc` 的 Swift 绑定),按同一套 Socket.io 信令做。

### 1.4 鉴权现状(移动端有一处必须改后端)

- our-chat 自身 = IdP:HS256 内部签发 + **JWKS(RS256)** 供下游验签(`/.well-known/...` + `/oauth/keys`)。
- **当前是 cookie-only,主动拒收 `Authorization: Bearer`**(为 web 同源 + CSRF 设计)。
- **移动端缺口**:需把鉴权中间件改成 **dual-accept**(先认 `Bearer`、无则回退 cookie+CSRF)——`docs/技术方案/Web与多端鉴权方案.md` §6.3 已写演进路径:iOS 把 AT/RT 存 **Keychain**,按 Bearer 调用,AT 过期用 RT 静默刷新。
- **agent-server 端无需改**:它已支持 `Bearer`(JWKS 验签),iOS 拿 our-chat 的 token 可**直连**,无需二次登录。

### 1.5 知识库助手现状(iOS 直接消费)

- agent-server `/api` 前缀,关键流式接口:
  - `POST /api/conversations/:id/messages` → **SSE**(`token`/`done`+citations/`error`),RAG 问答。
  - `POST /api/agent/tasks` → `runId`;`GET /api/runs/:runId/stream` → **SSE**(`step`/`tool_called`/`tool_result`/`final_answer`,支持 `Last-Event-ID` 断线重放)。
  - 文档 CRUD:`/api/documents`(上传 multipart、列表、删)。
- **iOS 消费要点**:SSE 用 POST body → 不能用原生 `EventSource`,要用 `URLSession` 的 `bytes(for:)` 流式读;或 SSE 用 `?access_token=` 兜底带 token。OpenAPI 在 `/api/docs-json`,可用 **swift-openapi-generator** 生成客户端类型。

### 1.6 一句话盘点

**后端信令/IM/鉴权/知识库基础齐全且可复用;iOS 是干净骨架(现代栈就绪);最大缺口=iOS 业务全未写 + 后端 dual-accept Bearer + 通话缺 TURN。基线层是"接现成",深度层才是项目价值。**

## 2. 设计思路(为什么这么规划)

### 2.1 "基线 + 深度"分层的理由

- **基线层贴合但不够深**:登录/IM/知识库/通话基本都是"调库 + 接现成后端"。它必要(让项目是个完整可用的产品、贴合现有),但**单靠它讲不透技术**(根因:无自实现硬核)。
- **深度层才是"讲得透"**:把客户端音视频里**你能亲手从字节级写的环节**做出来。WebRTC 通话内核是黑盒(`项目规划-自研播放器内核.md` §0 已论证不适合自实现);而**播放器内核(解封装/解码驱动/渲染/同步/seek)和采集渲染管线(Metal 美颜)在客户端、能你自己写、Web↔iOS 对称**——这正是"提升客户端音视频技术"的最优落点。

### 2.2 为什么深核 A/B/C 能落地到 our-chat(不牵强)

- **深核 A(视频消息→播放器内核)**:任何 IM 都有"发小视频/视频文件",这是 our-chat 自然要补的功能;而"播放"正是承载自研播放器内核的最佳容器——**功能真实需要 + 深度真实硬核**,不是为做而做。
- **深核 B(通话美颜)**:视频通话天然要美颜/滤镜/虚拟背景,把它做成**自写 Metal 管线**而非调三方美颜 SDK,就把"调库"变成"自实现渲染管线深核"。
- **深核 C(通话音频/变声)**:语音/视频通话的趣味音效、降噪是真实诉求,载体即现有通话。

### 2.3 取舍:不做什么(防范围爆炸)

- **不自建 SFU/多人通话**:重后端基建,客户端深度反而被分散(`项目规划-自研播放器内核.md` §9)。先把 P2P 通话 + 播放器内核做透。
- **不自己写编解码器**:解码驱动 VideoToolbox/硬件,绝不手写 H.264 解码器。
- **不一上来做直播**:M6 直播是 stretch,要先有点播播放器内核。

## 3. 项目总架构(iOS 如何接现有后端)

```
┌─────────────────────── iOS App(mobile-swift, TCA)───────────────────────┐
│  Features:  Auth · ConversationList · Chat · KnowledgeAssistant · Call    │
│  Services:                                                                 │
│   ├ AuthService      Keychain 存 AT/RT,Bearer 调用,RT 静默刷新           │
│   ├ SocketService    Socket.io/WS 客户端(IM + 通话信令复用)             │
│   ├ AgentService     消费 agent-server SSE(URLSession bytes 流式)        │
│   ├ CallService      WebRTC iOS(信令复用 call:* 事件)  ← 基线           │
│   ├ VideoMessage     录制/上传 + 【自研播放器内核】     ← 深核 A         │
│   ├ MediaPipeline    采集 → Metal 美颜 → 喂 WebRTC      ← 深核 B         │
│   └ AudioEngine      实时音频线程 + 变声               ← 深核 C(可选)  │
└───────────────┬───────────────────────────┬──────────────────────────────┘
        Bearer  │ HTTP/Socket.io/WS          │ Bearer/SSE
                ▼                            ▼
        our-chat(server + gateway)     agent-server(知识库/agent)
        IM·信令·IdP(JWKS)·文件S3      RAG·ReAct·Milvus
```

**鉴权链**:iOS 登录拿 our-chat 签发的 JWT → 存 Keychain → 调 server/gateway(需后端 dual-accept)与 agent-server(已支持 Bearer)。

## 4. 基线层规划(贴合现有,先让项目"完整可用")

| 模块 | 做什么 | 复用什么 | 深度 |
|---|---|---|---|
| **Auth** | 登录、Keychain 存 AT/RT、Bearer 拦截器、RT 静默刷新 | our-chat OAuth(需后端开 dual-accept) | 低 |
| **IM** | 会话列表、单/群聊、收发、已读、离线 sync、Presence | Socket.io 协议 + `message.ts` 契约 | 低-中 |
| **知识库助手** | 文档列表/上传、RAG 问答(SSE 流式渲染 token + 引用)、agent 任务流(step/tool 事件) | agent-server SSE 接口 + OpenAPI | 中(SSE 流式 + 断线重放是亮点) |
| **音视频通话(基线)** | iOS WebRTC:语音补齐 + **升级视频通话**(加 video track),信令复用 `call:*` | `web/useVoiceCall.ts` 逻辑 + 同后端信令 | 中(调 libwebrtc,黑盒) |

> 基线层用 swift-openapi-generator 从 agent-server 的 OpenAPI 生成类型,IM/信令用手写 Socket.io 客户端。**基线层完成 = 一个能登录、聊天、问知识库、打视频电话的 iOS App。**

## 5. 深度层规划(项目的技术含金量所在)

### 5.1 深核 A:视频消息 + 自研播放器内核(首选,最硬)

**功能**:IM 里发"小视频消息 / 视频文件";接收方点击播放——**播放走自研内核,而非 `AVPlayer`**。

**自实现链路(对应 `项目规划-自研播放器内核.md` iOS 端 M0–M5)**:
```
录制:AVCaptureSession → AVAssetWriter 编码 H.264/AAC → 上传(复用现有 S3/MinIO 文件链路)
播放(自研):
  下载/边下边播 → 【M1 自解封装】手写 MP4 box+stbl(给时间→字节偏移+是否关键帧)
  → 【M2 解码】VTDecompressionSession 驱动,处理回调乱序(DPB 重排)
  → 【M2 渲染】Metal + CVMetalTextureCache 零拷贝 NV12;或 AVSampleBufferDisplayLayer 直显
  → 【M3 音频】AURenderCallback/AVAudioSourceNode + lock-free ring buffer
  → 【M4 A/V 同步】音频主时钟 + compute_target_delay(ffplay 三阈值)
  → 【M5 seek/起播/缓冲】视频文件消息带进度条:找 IDR + flush + serial 丢旧帧;倍速
```

**为什么是最强深核**:它把客户端音视频**全链路**(解封装→解码驱动→渲染管线→音频实时线程→A/V 同步→随机访问)全部亲手实现,是"懂原理 + 有自实现深核"的标志;且 iOS 端有 VideoToolbox/Metal/AVSampleBufferDisplayLayer 这套最贴近底层的 API。**深度细节、踩坑、验收标准、LOC、双端对照全部已在 `项目规划-自研播放器内核.md` 写好,本项目直接执行那份。**

**与现有功能的接点**:录制/上传复用 our-chat 已有文件存储(`docs/技术方案/文件存储改造-迁移到S3兼容对象存储.md` 的分片/秒传/断点续传);消息类型在 `message.ts` 的 `type` 字段扩展一个 `video`。

**进阶(可选,stretch)**:知识库里若有视频资料 → 同一自研播放器复用为"视频知识库播放器";M6 直播则需新增直播功能(范围较大,默认不做)。

### 5.2 深核 B:视频通话实时美颜/滤镜(次选)

**功能**:视频通话时磨皮/滤镜/虚拟背景——**自写 Metal 渲染管线**,不调三方美颜 SDK。

**自实现链路(对应 KB `20` 渲染管线 / `22` 采集美颜)**:
```
AVCaptureSession 出 CMSampleBuffer(NV12)
  → CVMetalTextureCache 零拷贝转 Metal 纹理
  → 自写 Metal compute/fragment shader:YUV→RGB、磨皮(高斯/双边)、LUT 滤镜、(进阶)分割虚拟背景
  → 输出 CVPixelBuffer → 包成 RTCVideoFrame,经自定义 RTCVideoCapturer/RTCVideoSource 喂给 WebRTC 编码
```

**硬核点**:stride≠width 绿边、BT.601/709 与 range 色彩矩阵、CVMetalTextureCache 零拷贝、实时帧率下 GPU 管线不掉帧、与 WebRTC 编码线程的 buffer 生命周期(及时释放 surface)。

**与现有功能的接点**:插在基线层"视频通话"的采集与编码之间,后端零改动。

### 5.3 深核 C:通话音频处理 / 变声(可选)

**功能**:通话趣味变声 / 音效 / 简单降噪。

**自实现链路(对应 KB `21` 音频实时线程)**:
```
AURenderCallback 实时线程取 PCM → lock-free SPSC ring buffer(原子读写指针 + 正确内存序)
  → 音效处理(变调/混响,实时禁 malloc/锁)→ 送 WebRTC 音频轨/声卡
```
**硬核点**:实时线程纪律(不分配/不加锁)、内存序 acquire/release(偶发爆音最难查)、欠载补静音。

## 6. 后端配套(最小改动,遵守"不砍功能")

| 改动 | 必要性 | 量级 | 依据 |
|---|---|---|---|
| **server/gateway 鉴权 dual-accept Bearer** | **必须**(iOS 无 cookie jar) | 小 | `docs/技术方案/Web与多端鉴权方案.md` §6.3 |
| **自建 TURN(coturn)** | **视频通话生产必须**(当前仅 STUN,NAT 下连不通) | 小-中(部署 + 下发 ICE 配置) | 现状 §1.3 缺口 |
| 信令扩展支持视频 | 升级视频通话(加 video transceiver,SDP 自带) | 小(主要在客户端) | `call:*` 复用 |
| 视频消息类型 | `message.ts` `type` 加 `video` + 缩略图/时长元数据 | 小 | 复用现有文件链路 |

> 改后端遵守 our-chat CLAUDE.md:鉴权先读 `跨服务鉴权方案/方案D`;长连接统一走 gateway;不砍既有功能。

## 7. 分阶段路线(每阶段可跑可验证)

| 阶段 | 内容 | 验收 |
|---|---|---|
| **M0 打通地基** | Auth(Keychain+Bearer)+ Socket.io 连上 + 收发文本 | iOS 能登录、进会话、收发消息(需后端先开 dual-accept) |
| **M1 基线 IM + 知识库** | 会话列表/群聊/已读/离线;知识库 SSE 问答 + agent 任务流 | 完整聊天 + 流式问知识库 + 看到 agent 工具调用 |
| **M2 基线通话** | iOS WebRTC 语音 + 升级视频;部署 TURN | iOS↔Web 互打音/视频电话,跨 NAT 通 |
| **M3 深核 A·点播内核**(主深核) | 视频消息录制/上传 + 自研播放器内核 M1–M4(自解封装→VT 解码→Metal 渲染→A/V 同步) | 发的小视频用**自研内核**播放,音画对齐、色彩正确、无爆音 |
| **M4 深核 A·随机访问** | seek/起播/缓冲/倍速(视频文件消息进度条) | 拖进度条跟手不花屏、起播 <1s |
| **M5 深核 B·Metal 美颜** | 自写 Metal 采集管线接进视频通话 | 通话实时美颜/滤镜,不掉帧、无绿边色偏 |
| **M6 深核 C(可选)/ 打磨** | 变声音效;性能 benchmark;弱网 | 通话变声无爆音;弱网注入不崩 |

> **M3/M4(播放器内核)是项目的硬核大山**,做透即立住深度。M0–M2 是"接现成",尽快过。

## 9. 技术难点 / 踩坑预判

| 阶段 | 易卡点 | 预案 |
|---|---|---|
| M0 鉴权 | 后端 dual-accept 未开 / RT 刷新竞态 | 先改后端;AT/RT 刷新加单飞(single-flight) |
| M2 通话 | 仅 STUN 跨 NAT 连不通 | 部署 coturn,ICE 配置下发;iOS WebRTC SDK 版本/编译坑 |
| M3 解码 | VTDecompressionSession 回调乱序要自己 DPB 重排 | 按 KB `08`;先 AVAssetReader 对照验证自写解封装 |
| M3 渲染 | stride≠width 绿边、601/709 色偏、CVPixelBuffer 不释放 | 按 KB `20`;先渲一张静态 YUV 帧 |
| M3 音频 | 实时回调 malloc/锁、ring 内存序错 → 偶发爆音 | 严格 lock-free;先压测纯环形缓冲 |
| M4 同步/seek | 没减硬件延迟漂移、serial 丢旧帧不彻底花屏 | 音频主时钟减 outputLatency;queue_serial++ |
| M5 美颜 | Metal 管线掉帧、与编码线程 buffer 生命周期 | CVMetalTextureCache 零拷贝;及时释放 surface |

## 11. 范围护栏

1. **深度只钉在能自实现的环节**(解封装/解码驱动/渲染/音频/同步/美颜管线),WebRTC 通话内核当黑盒,不假装"自研 RTC 内核"。
2. **先基线后深核**,基线尽快过(接现成,别在调库上耗)。
3. **深核 A(播放器内核)优先级最高**,B 次之,C 可选;别三个并行。
4. **不自建 SFU/不一上来做直播/不手写编解码器**。
5. **后端改动最小化**(dual-accept + TURN + 消息类型),遵守 our-chat"不砍功能、长连接走 gateway"。
6. 先 iOS 做透;Web 已有通话可作对照(KB 双端对照价值)。
