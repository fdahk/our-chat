# 02 · 工程结构

## 1. 目录全景

```
mobile-swift/
├── Project.swift                       # Tuist 工程主清单(等价 vite.config.ts + package.json)
├── Tuist/
│   └── Package.swift                   # SPM 依赖声明(等价 dependencies 字段)
├── Sources/                            # 框架初始化时为空,按下方约定添加
│   ├── OurChatApp.swift                # @main 入口(SwiftUI App 协议,首次创建)
│   ├── Features/                       # 按业务域划分,每个 Feature 一个子目录
│   │   ├── AppFeature.swift            # 根 Reducer,组合所有子 Feature
│   │   └── <YourFeature>/
│   │       ├── <YourFeature>Feature.swift  # @Reducer(state/action/business logic)
│   │       └── <YourFeature>View.swift     # SwiftUI View(纯渲染,无业务逻辑)
│   ├── Models/                         # 数据模型(Codable + Sendable + Equatable)
│   └── Services/                       # @DependencyClient 服务层(替代手写 protocol)
├── Resources/
│   └── Assets.xcassets/                # 图片、颜色、AppIcon
│       ├── AppIcon.appiconset/         # 1024x1024 主图标(初始为占位)
│       └── AccentColor.colorset/       # 主色调
├── Tests/                              # 框架初始化时为空
│   ├── AppFeatureTests.swift           # Swift Testing 根 Feature 测试
│   └── <YourFeature>Tests.swift        # Swift Testing + TCA TestStore
├── docs/                               # 选型与流程文档
├── .swiftformat                        # SwiftFormat 配置
├── .swiftlint.yml                      # SwiftLint 配置
├── .mise.toml                          # 工具版本锁(tuist / swiftlint / swiftformat)
├── .gitignore                          # 排除 Derived/、.build/、*.xcodeproj/
└── README.md
```

> 框架 `init` 提交里 `Sources/` 和 `Tests/` 是空的。上面列出的文件名是**约定模板**,不是必备文件。你按约定添加自己的 Feature 即可。

## 2. 关键文件说明

### `Project.swift` — 工程主清单

声明 Target、Bundle ID、Info.plist、依赖、Build Settings。是 Tuist 工程的"package.json + vite.config.ts"二合一。

每次改动后跑 `tuist generate` 重新生成 `.xcodeproj`。**`.xcodeproj` 不入库**,被 `.gitignore` 排除。

关键 Build Settings(已在 `Project.swift` 里写死):
```swift
"SWIFT_VERSION": "6.0",
"SWIFT_STRICT_CONCURRENCY": "complete",   // 严格并发检查,等价 Rust 的 -W warnings as errors
"IPHONEOS_DEPLOYMENT_TARGET": "18.0",     // 最低部署 iOS 18
"DEAD_CODE_STRIPPING": "YES",             // 减小包体
```

### `Tuist/Package.swift` — 依赖声明

纯 SPM 语法,Tuist 解析后调度 `swift package` 解析 + 编译。需要新加包时:

```swift
.package(url: "https://github.com/xxx/yyy", from: "1.0.0"),
```

然后跑:
```bash
tuist install      # 拉取新依赖
tuist generate     # 重新生成工程
```

注意:**只在 `Tuist/Package.swift` 加依赖,不要在 Xcode UI 里 Add Package Dependencies**。否则会与 Tuist 状态分歧。

### `Sources/Features/` — TCA Feature 组织约定

每个 Feature 一个目录,内含:
- `XxxFeature.swift` — `@Reducer` 定义 state + action + body
- `XxxView.swift` — 纯渲染 SwiftUI View,通过 `StoreOf<XxxFeature>` 接收状态

**规则**:
- View 文件**不**写业务逻辑,所有副作用回到 Reducer
- Feature 文件**不** import SwiftUI,只 import ComposableArchitecture + Foundation
- 跨 Feature 通信用 `Scope` 组合,**不**用 NotificationCenter / 全局 singleton

### `Sources/Services/` — 服务层

所有外部 IO(网络 / 数据库 / 文件 / 系统 API)封装为 `@DependencyClient` struct。

模板:

```swift
@DependencyClient
struct XxxClient: Sendable {
    var doSomething: @Sendable (_ input: T) async throws -> R
}

extension XxxClient: DependencyKey {
    static let liveValue = XxxClient(...)  // 真实实现
    static let previewValue = XxxClient(...)  // SwiftUI Preview 用
    // testValue 由 @DependencyClient 宏自动生成(全部字段 unimplemented(),逼测试 stub)
}

extension DependencyValues {
    var xxxClient: XxxClient {
        get { self[XxxClient.self] }
        set { self[XxxClient.self] = newValue }
    }
}
```

Reducer 里用 `@Dependency(\.xxxClient) var xxxClient` 注入。

### `Tests/` — Swift Testing

```swift
@MainActor   // TCA TestStore 要求主线程
struct XxxFeatureTests {
    @Test
    func someBehavior() async {
        let store = TestStore(initialState: ...) {
            XxxFeature()
        } withDependencies: {
            $0.xxxClient.doSomething = { _ in .stub }
        }

        await store.send(.someAction) {
            $0.someState = newValue
        }

        await store.receive(\.someResponse) {
            $0.otherState = otherValue
        }
    }
}
```

## 3. 何时模块化拆分

单 Target 适合的项目规模(凭经验):
- 5-20 个 Feature
- 编译时间 < 30 秒
- 团队 < 5 人

超出后拆模块:在 `Project.swift` 加 `.target` 项,例如:

```swift
.target(
    name: "FeatureCore",          // 纯业务逻辑,无 UI
    destinations: .iOS,
    product: .framework,
    bundleId: "com.ourchat.feature-core",
    sources: ["Modules/FeatureCore/Sources/**"],
    dependencies: [.external(name: "ComposableArchitecture")]
),
.target(
    name: "FeatureUI",
    ...
    dependencies: [.target(name: "FeatureCore")]
),
.target(
    name: "OurChat",
    ...
    dependencies: [.target(name: "FeatureUI"), ...]
),
```

拆分前先读 Tuist 官方的 [Modular architecture](https://docs.tuist.io/guides/develop/projects/dependencies) 文档。

## 4. 与 mobile-flutter 目录的关系

| 维度 | mobile-swift | mobile-flutter |
|------|--------------|----------------|
| 渲染层 | SwiftUI 原生 | Flutter Skia/Impeller |
| 状态管理 | TCA + @Observable | Riverpod / Bloc |
| 包管理 | SPM(via Tuist) | pub |
| 测试 | Swift Testing + TestStore | flutter_test |
| 目标平台 | iOS only | iOS + Android + Web |

两边**不共享代码**,通过后端契约(`our-chat/record.md` 描述的服务端)对齐。如果未来需要共享文案/常量,放后端下发,不要在两个目录之间符号链接。

## 5. 文件不该出现的地方

下面这些是常见反模式,本项目里**绝不**出现:

- ❌ `Sources/Constants.swift`(全局常量类)— 用 Dependency 注入或 enum namespace
- ❌ `Sources/Utils.swift`(杂项工具)— 按业务域归类,放对应 Feature 或新建 `Sources/Foundation/`
- ❌ `Sources/Singletons/`(单例)— TCA 时代不需要,所有共享状态走 `@Dependency`
- ❌ `*ViewModel.swift`(SwiftUI MVVM)— TCA 已经是更彻底的 MV pattern,不要混
- ❌ `*Manager.swift`(Foo Manager / Bar Manager)— 改用 `*Client.swift`(@DependencyClient)
