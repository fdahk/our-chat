# 01 · 技术栈选型

> 本文记录本项目每个技术选型的**为什么**。如果未来要替换某个组件,先读这里,确认替换的成本与收益再动手。

## 0. 一句话定位

**2026 业界最佳实践栈 · SwiftUI 优先 · TCA 架构 · 严格并发**。不是 UIKit 历史项目的兼容方案,也不是 Apple "MV pattern" 的最小架构,而是面向有前端工程经验、追求可测试性与类型安全的工程师的现代默认栈。

## 2. 各项选型的理由

### 2.1 Swift 6 + 严格并发

`SWIFT_VERSION = 6.0` + `SWIFT_STRICT_CONCURRENCY = complete`。

**为什么**:
- Swift 6 (2024-09 发布) 把 data race 检测从警告升级为编译错误。所有 `@MainActor`/`Sendable`/Actor 隔离边界编译期可证明,等价 Rust 的 Send/Sync。
- 与 React 19 的并发渲染心智模型对齐:UI 永远在主线程,业务逻辑可并行,但跨边界必须显式声明。

**代价**:
- 第三方库还在适配中,部分库需要 `@preconcurrency` 临时绕过。本项目 6 个核心依赖都已支持 Swift 6,无此问题。

### 2.2 SwiftUI 单栈,不要 UIKit 主

**为什么**:
- 新项目 2026 起,Apple 官方导向、社区资源、招聘市场都已经压在 SwiftUI 一侧。
- UIKit 适合的场景(超复杂列表、自定义手势协调)在我们的 chat 业务里很少触发。需要时通过 `UIViewRepresentable` 局部桥接即可,不需要主框架是 UIKit。
- 入门曲线和迭代速度比 UIKit 快 2-3 倍。

**例外场景**:
- 富文本编辑器(SwiftUI TextEditor 性能不足)→ 用 UITextView 桥接
- 复杂手势协调(多指 + 长按 + 旋转组合)→ 用 UIGestureRecognizer

### 2.3 TCA(The Composable Architecture)

**为什么用,不用 Apple 推的 "MV Pattern"**:
- Apple 官方 MV(直接用 @Observable + Environment)适合 50 行内的小项目。一旦逻辑变复杂,会演变成 service singleton + 散落 closure 回调,丢失可测试性。
- TCA 提供:
  - **完整状态机**:每个 Action 是一个明确的事件,State 转移可追踪、可时间旅行
  - **可测试性**:`TestStore` 强制断言每一步状态变化,接近 100% 覆盖
  - **可组合**:`Scope` + `Reducer` 嵌套,模块化天然
  - **副作用纯化**:`.run { send in ... }` 把异步关进 Effect,不污染 State

**前端类比**:Redux Toolkit + RTK Query + Immer 一体化。如果你写过 RTK 的 `createSlice` + `createAsyncThunk`,TCA 的 `@Reducer` + `.run` 五分钟就能上手。

**代价**:
- 学习曲线陡(必须先理解 Reducer 不可变 state 转移)
- 样板代码比 MV 多 30%

### 2.4 swift-dependencies + @DependencyClient

**为什么**:
- 替代手写 protocol + struct mock 的传统方式。
- `@DependencyClient` 宏自动生成 mock-friendly struct,字段是 `@Sendable () async throws -> T` 闭包,**默认实现是 fatalError**,逼你在测试里显式 stub。
- 与 TCA 的 `@Dependency(\.chatClient)` 无缝集成,测试用 `withDependencies` 一句话替换。

**前端类比**:接近 `inversify` 但比它优雅。或者像 MSW 的 handlers,但是是函数级而不是 HTTP 级。

### 2.5 Swift Testing,不用 XCTest

**为什么**:
- WWDC 2024 发布,iOS 18 原生支持。`@Test` 注解 + `#expect` 替代 `func test_xxx() { XCTAssertEqual(...) }`。
- 支持参数化测试、tag 分组、并行执行,语义上是 Rust 的 `#[test]` + Vitest 的 `test()` 合体。
- TCA 的 `TestStore` 也已完整支持 Swift Testing。

**XCTest 何时还需要**:
- UI 测试目前仍然必须用 XCTest(`XCUITest`)
- iOS 17 及以下的最低部署需要兜底

### 2.6 Tuist,不用裸 Xcode 工程

**为什么**:
- `.xcodeproj` 是二进制 plist + 大量自动生成内容,Git 冲突灾难现场。Tuist 让你只把 `Project.swift`(纯 Swift 代码)入库,工程通过 `tuist generate` 重新生成。
- 模块化拆分零成本:`targets: [.target(...), .target(...)]` 加一项即可。
- 远程缓存(`tuist cache`)可以让 CI 编译时间砍半,大型项目刚需。

**前端类比**:`vite.config.ts` 之于 webpack 的 `output.path = path.resolve(__dirname, ...)`。

**代价**:
- 多一个工具版本要管理(本项目用 `.mise.toml` 锁版本)
- `tuist generate` 比 Xcode 直接打开慢 5-15 秒

### 2.7 SPM,不用 CocoaPods/Carthage

CocoaPods 2024 官方宣布 wind-down(进入维护模式),Carthage 早已边缘化。新项目 100% SPM。本项目所有依赖通过 `Tuist/Package.swift` 声明,Tuist 翻译成 SPM resolved。

## 3. 一定要避开的过时栈

| 别用 | 因为 |
|------|------|
| **Combine** 写新业务代码 | 2024 起 Apple 全面推 Observation + AsyncSequence,Combine 进入维护模式 |
| **CocoaPods** | 2024 官方宣布 wind-down,新项目 100% SPM |
| **RxSwift** | 同 Combine,但比 Combine 更尴尬 |
| **VIPER / Clean Architecture** 写新项目 | TCA 时代的过度设计,样板代码爆炸 |
| **Realm** | Atlas Device Sync 2024 EOL,MongoDB 不再投入 |
| **SDWebImage** 写新项目 | Kingfisher 是 Swift 时代标准答案 |
| **Storyboard / XIB** 写新页面 | SwiftUI 时代毫无理由再用 |

## 4. 升级窗口

iOS 主线版本一年一升(每年 9 月)。本项目最低部署 iOS 18.0,**计划在 2026-09 iOS 26 / 27 GA 后评估**:
- 是否提到 iOS 19+ 最低部署,以使用更新的 Observation/Swift Concurrency 特性
- 是否拥抱 Liquid Glass 设计语言(Apple WWDC 2025 推出)
- 是否引入 Apple Intelligence 系统级 AI 能力(端侧 Foundation Model)
